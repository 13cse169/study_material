What Is SQL?

SQL (Structured Query Language) is the standard language for communicating with relational databases. It is declarative — you describe what you want, the database engine figures out how to get it. Every major database (PostgreSQL, MySQL, SQLite, Oracle, SQL Server, BigQuery) speaks SQL.

🗄️
Database
Container holding related tables, views, and procedures
📋
Table
Grid of rows (records) and typed columns (fields)
🔑
Primary Key
Unique ID for each row — never NULL, never duplicated
🔗
Foreign Key
Column referencing a PK in another table
📐
Schema
Blueprint defining table structure & constraints
⚡
Index
Structure that speeds up column lookups dramatically

SQL Statement Categories

CategoryFull NameCommandsPurpose
DDLData Definition LanguageCREATE, ALTER, DROP, TRUNCATEDefine / modify structure
DMLData Manipulation LanguageINSERT, UPDATE, DELETE, MERGEAdd / change / remove records
DQLData Query LanguageSELECTRetrieve data
DCLData Control LanguageGRANT, REVOKEManage permissions
TCLTransaction Control LanguageBEGIN, COMMIT, ROLLBACKManage transactions
🤖
Why SQL Matters for AI/MLReal ML training data lives in databases. Writing efficient SQL lets you extract, filter, join, and aggregate at the database level — far faster than doing it in Python for large datasets. It's a day-one skill for any data scientist.

SELECT Basics

SELECT retrieves data from tables. It's the most-used SQL statement — master it first.

SQL
-- Select all columns
SELECT * FROM employees;

-- Select specific columns with aliases
SELECT
    employee_id,
    first_name  AS name,
    salary      AS annual_pay,
    department
FROM employees;

-- Computed columns
SELECT
    name,
    salary,
    salary * 1.10   AS after_raise,
    salary / 12    AS monthly
FROM employees;

-- Remove duplicate rows
SELECT DISTINCT department FROM employees;
04Beginner

WHERE & Filtering

WHERE filters rows before they are returned. You can combine multiple conditions using AND, OR, and NOT.

SQL
-- Single condition
SELECT * FROM employees WHERE salary > 50000;

-- Multiple conditions
SELECT name, salary, department
FROM   employees
WHERE  salary     BETWEEN 40000 AND 80000
  AND  department = 'Engineering'
  AND  is_active  = 1;

-- IN: match a list of values
SELECT * FROM products
WHERE category IN ('Electronics', 'Books', 'Software');

-- NULL checks — must use IS NULL, not = NULL
SELECT * FROM employees WHERE manager_id IS NULL;
SELECT * FROM employees WHERE phone      IS NOT NULL;
⚠️
SQL Execution OrderClauses run in this order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. You cannot use a SELECT alias in WHERE because WHERE runs before SELECT evaluates aliases.

ORDER BY & LIMIT

Sort results and restrict how many rows are returned. Essential for leaderboards, pagination, and top-N analyses.

SQL
-- Sort descending
SELECT name, salary
FROM   employees
ORDER BY salary DESC;

-- Multi-column sort
SELECT department, name, salary
FROM   employees
ORDER BY department ASC, salary DESC;

-- Top 10 highest earners
SELECT name, salary
FROM   employees
ORDER BY salary DESC
LIMIT  10;

-- Pagination: rows 21–30 (page 3, page_size=10)
SELECT * FROM products
ORDER BY product_id
LIMIT  10 OFFSET 20;

08Beginner

GROUP BY & HAVING

GROUP BY splits rows into groups, applying aggregates to each group independently. HAVING filters groups after aggregation — it's the WHERE for groups.

SQL — GROUP BY
-- Stats per department
SELECT
    department,
    COUNT(*)     AS headcount,
    AVG(salary) AS avg_salary,
    SUM(salary) AS dept_payroll
FROM   employees
GROUP BY department
ORDER BY avg_salary DESC;

-- HAVING: departments with 5+ people AND avg salary > 60k
SELECT
    department,
    COUNT(*)     AS headcount,
    AVG(salary) AS avg_salary
FROM   employees
GROUP BY department
HAVING   COUNT(*) > 5
     AND AVG(salary) > 60000
ORDER BY headcount DESC;

-- Multi-column grouping
SELECT department, job_title, COUNT(*) AS cnt, AVG(salary) AS avg
FROM   employees
GROUP BY department, job_title
ORDER BY department, avg DESC;

Data Definition Language commands define and modify the structure of your database objects.

SQL — DDL
-- CREATE TABLE with full constraints
CREATE TABLE employees (
    id          SERIAL          PRIMARY KEY,
    name        VARCHAR(100)    NOT NULL,
    email       VARCHAR(200)    UNIQUE,

    hired_at    TIMESTAMP       DEFAULT NOW(),

ALTER TABLE employees ADD COLUMN    phone   VARCHAR(20);
ALTER TABLE employees RENAME COLUMN phone  TO mobile;
ALTER TABLE employees ALTER COLUMN  salary  TYPE BIGINT;
ALTER TABLE employees DROP COLUMN   mobile;

-- Remove objects
DROP TABLE IF EXISTS temp_data;     -- removes table + data
TRUNCATE TABLE logs;                -- removes rows, keeps structure

DML — INSERT, UPDATE, DELETE

DML commands add, modify, and remove records from tables. Always test DELETE/UPDATE with a SELECT first.

SQL — DML
-- INSERT single row
INSERT INTO employees (name, email, salary, department)
VALUES ('Arjun Sharma', 'arjun@co.com', 75000, 'Engineering');

-- INSERT multiple rows
INSERT INTO products (name, price, category)
VALUES
    ('Laptop Pro',    89999, 'Electronics'),
    ('SQL Handbook',  499,   'Books'),
    ('Keyboard',      5499,  'Accessories');

-- INSERT from another table
INSERT INTO archive_orders
SELECT * FROM orders
WHERE order_date < '2023-01-01';

-- UPDATE with condition
UPDATE employees
SET    salary     = salary * 1.10,
       updated_at = NOW()
WHERE  department = 'Engineering'
  AND  years_exp  >= 3;

-- DELETE with safety condition
DELETE FROM sessions
WHERE last_active < NOW() - INTERVAL '30 days';

JOINs — Combining Tables

JOINs combine rows from two or more tables based on a related column. They are the cornerstone of relational database design.

◉◯
INNER JOIN
Only rows with a match in both tables. Non-matching rows excluded.
→ Intersection only
◉◉
LEFT JOIN
All left rows + matching right rows. Non-matching right = NULL.
→ All left + matched right
⬤⬤
RIGHT JOIN
All right rows + matching left rows. Non-matching left = NULL.
→ All right + matched left
⬡⬡
FULL OUTER
All rows from both sides. Non-matches get NULLs.
→ Complete union
⬛⬛
CROSS JOIN
Every combination of rows. Use carefully — result is rows_a × rows_b.
→ Cartesian product
🔄
SELF JOIN
Table joined to itself. Classic for hierarchies (employee–manager).
→ Same table aliased
SQL — JOINs
-- INNER JOIN: orders with customer info
SELECT c.name, o.order_date, o.total_amount
FROM   customers c
INNER JOIN orders o ON c.id = o.customer_id
ORDER BY o.order_date DESC;

-- LEFT JOIN: all customers even those with zero orders
SELECT
    c.name,
    COUNT(o.id)                          AS order_count,
    COALESCE(SUM(o.total_amount), 0)    AS lifetime_value
FROM   customers c
LEFT JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name
ORDER BY lifetime_value DESC;

-- Three-table JOIN
SELECT c.name, o.order_date, p.product_name, oi.quantity
FROM   customers   c
JOIN   orders      o  ON c.id          = o.customer_id
JOIN   order_items oi ON o.id          = oi.order_id
JOIN   products    p  ON oi.product_id = p.id;

-- SELF JOIN: employee and their manager
SELECT
    e.name AS employee,
    m.name AS manager
FROM   employees e
LEFT JOIN employees m ON e.manager_id = m.id;
12Intermediate

Subqueries

A subquery is a SELECT nested inside another query. They appear in WHERE, FROM, SELECT, or HAVING clauses for multi-step logic.

SQL — Subqueries
-- In WHERE: employees earning above company average
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees);

-- In FROM: derived table (virtual table)
SELECT department, avg_salary
FROM (
    SELECT department, AVG(salary) AS avg_salary
    FROM   employees
    GROUP BY department
) dept_summary
WHERE avg_salary > 70000;

-- EXISTS: customers with at least one order
SELECT * FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.id
);

-- Correlated subquery: compare each row to its dept avg
SELECT name, department, salary,
       (SELECT AVG(salary) FROM employees e2
        WHERE e2.department = e1.department) AS dept_avg
FROM employees e1;
13Intermediate

CASE WHEN

CASE is SQL's if/else. Use it to create conditional labels, buckets, and pivot-style aggregations.

SQL — CASE WHEN
-- Salary band labeling
SELECT name, salary,
    CASE
        WHEN salary <  40000 THEN 'Junior'
        WHEN salary <  80000 THEN 'Mid-Level'
        WHEN salary < 150000 THEN 'Senior'
        ELSE                       'Executive'
    END AS band
FROM employees;

-- Conditional aggregation (pivot pattern)
SELECT
    department,
    COUNT(CASE WHEN gender = 'M' THEN 1 END) AS male_count,
    COUNT(CASE WHEN gender = 'F' THEN 1 END) AS female_count,
    SUM(CASE WHEN is_active THEN 1 ELSE 0 END) AS active
FROM employees
GROUP BY department;

Date & Time Functions

Date manipulation is critical for time-series analysis, cohort studies, and ML feature engineering with temporal data.

SQL — Date Functions (PostgreSQL)
-- Current date/time
SELECT NOW(), CURRENT_DATE, CURRENT_TIME;

-- Extract parts
EXTRACT(YEAR FROM  hired_at) -- 2024
EXTRACT(MONTH FROM hired_at) -- 1–12
EXTRACT(DOW FROM   hired_at) -- 0=Sun, 6=Sat

-- Truncate to period (great for monthly grouping)
SELECT
    DATE_TRUNC('month', order_date) AS month,
    COUNT(*)                         AS orders,
    SUM(total_amount)               AS revenue
FROM   orders
GROUP BY DATE_TRUNC('month', order_date)
ORDER BY month;

-- Date arithmetic
SELECT
    name,
    AGE(NOW(), hired_at)              AS tenure,
    EXTRACT(DAY FROM NOW()-hired_at) AS days_employed,
    hired_at + INTERVAL '90 days'    AS probation_end
FROM employees;

Set Operations

Set operations combine results from multiple SELECT statements. Both queries must have the same number of columns with compatible types.

OperatorReturnsKeeps Duplicates?
UNIONRows in either result setNo (removes)
UNION ALLRows in either result setYes (faster)
INTERSECTRows in BOTH result setsNo
EXCEPTRows in first but NOT secondNo
SQL — Set Operations
-- UNION: combined emails from two tables (deduped)
SELECT email FROM customers_india
UNION
SELECT email FROM customers_usa;

-- UNION ALL: combine with labels, keep dupes (faster)
SELECT id, name, 'active'   AS status FROM active_users
UNION ALL
SELECT id, name, 'inactive'            FROM inactive_users;

-- EXCEPT: customers who never ordered
SELECT id FROM customers
EXCEPT
SELECT DISTINCT customer_id FROM orders;

Indexes & Constraints

Indexes speed up SELECT queries at the cost of slightly slower writes and extra storage. Understanding them is essential for production performance.

SQL — Indexes
-- Standard B-tree index
CREATE INDEX idx_emp_dept ON employees(department);

-- Composite index (multi-column queries)
CREATE INDEX idx_dept_sal ON employees(department, salary);

-- Unique index: enforces uniqueness + speed
CREATE UNIQUE INDEX idx_email ON users(email);

-- Partial index: only index active rows
CREATE INDEX idx_active ON employees(salary)
WHERE is_active = TRUE;

-- List indexes on a table (PostgreSQL)
SELECT indexname, indexdef
FROM   pg_indexes
WHERE  tablename = 'employees';

Window Functions

Window functions compute results across a set of rows related to the current row — without collapsing them like GROUP BY. They are the most powerful analytical tool in SQL.

FunctionCategoryDescription
ROW_NUMBER()RankingUnique sequential number — no ties
RANK()RankingTies share same rank; next rank skips
DENSE_RANK()RankingTies share same rank; next rank does NOT skip
NTILE(n)RankingDivide into n equal-sized buckets
LAG(col, n)OffsetValue from n rows before current row
LEAD(col, n)OffsetValue from n rows after current row
FIRST_VALUE(col)OffsetFirst value in the window frame
SUM() OVERAggregateRunning total or partition total
AVG() OVERAggregateMoving/rolling average
PERCENT_RANK()RankingRelative rank as 0.0–1.0
SQL — Window Functions
SELECT
    name, department, salary,

    -- Rankings within each department
    RANK()        OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
    DENSE_RANK()  OVER (PARTITION BY department ORDER BY salary DESC) AS dense,
    ROW_NUMBER()  OVER (PARTITION BY department ORDER BY salary DESC) AS rownum,
    NTILE(4)      OVER (ORDER BY salary)                              AS quartile,

    -- Previous/next salary in same department
    LAG(salary,  1, 0) OVER (PARTITION BY department ORDER BY salary) AS prev_sal,
    LEAD(salary, 1, 0) OVER (PARTITION BY department ORDER BY salary) AS next_sal,

    -- Running total and dept percentage
    SUM(salary) OVER (PARTITION BY department) AS dept_total,
    SUM(salary) OVER (ORDER BY hire_date)     AS running_payroll,
    ROUND(salary*100.0 /
        SUM(salary) OVER(PARTITION BY department), 1) AS pct_of_dept
FROM employees
ORDER BY department, rnk;

-- Classic: top earner per department
SELECT * FROM (
    SELECT *,
        RANK() OVER(PARTITION BY department ORDER BY salary DESC) AS rnk
    FROM employees
) t WHERE rnk = 1;

-- 3-day moving average of daily revenue
SELECT dt, revenue,
    AVG(revenue) OVER (
        ORDER BY dt
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS moving_avg_3d
FROM daily_revenue;
19Advanced

CTEs & Recursive Queries

Common Table Expressions (WITH clause) are named temporary result sets that exist only for the current query. They make complex queries readable and enable recursion for hierarchies.

SQL — CTEs & Recursive
-- Simple CTE
WITH active_emp AS (
    SELECT * FROM employees WHERE is_active = TRUE
)
SELECT department, AVG(salary) FROM active_emp GROUP BY department;

-- Chained CTEs: multi-step ML feature pipeline
WITH
raw_sessions AS (
    SELECT user_id, COUNT(*) AS sessions, AVG(duration) AS avg_dur
    FROM   user_sessions
    GROUP BY user_id
),
enriched AS (
    SELECT r.*, u.plan_type, u.signup_date
    FROM   raw_sessions r
    JOIN   users u ON r.user_id = u.id
),
features AS (
    SELECT *,
        DATEDIFF(NOW(), signup_date) AS days_old,
        CASE WHEN plan_type='premium' THEN 1 ELSE 0 END AS is_premium
    FROM enriched
)
SELECT * FROM features;

-- Recursive CTE: org hierarchy traversal
WITH RECURSIVE org AS (
    -- Base: top-level (no manager)
    SELECT id, name, manager_id, 0 AS depth
    FROM   employees
    WHERE  manager_id IS NULL
    UNION ALL
    -- Recursive: direct reports of each level
    SELECT e.id, e.name, e.manager_id, o.depth+1
    FROM   employees e
    JOIN   org o ON e.manager_id = o.id
    WHERE  o.depth < 10  -- safety depth limit
)
SELECT REPEAT('  ', depth) || name AS tree, depth
FROM   org
ORDER BY depth, name;
20Advanced

Views & Stored Procedures

Views are saved named queries you can query like tables. Materialized views physically cache the results for speed.

SQL — Views
-- Create a regular view
CREATE VIEW active_dept_summary AS
    SELECT
        department,
        COUNT(*)     AS headcount,
        AVG(salary) AS avg_salary,
        MAX(salary) AS max_salary
    FROM   employees
    WHERE  is_active = TRUE
    GROUP BY department;

-- Use it just like a table
SELECT * FROM active_dept_summary
WHERE headcount > 10;

-- Materialized view (stores data physically)
CREATE MATERIALIZED VIEW mv_monthly_rev AS
    SELECT DATE_TRUNC('month', order_date) AS month,
           SUM(total) AS revenue
    FROM   orders GROUP BY 1;

-- Refresh after data changes
REFRESH MATERIALIZED VIEW mv_monthly_rev;

Transactions & ACID

Transactions group multiple SQL statements into an all-or-nothing unit. ACID properties guarantee data integrity even under failures.

PropertyMeaning
AtomicityAll statements succeed or all are rolled back. No partial updates.
ConsistencyDatabase always moves from one valid state to another.
IsolationConcurrent transactions do not interfere with each other.
DurabilityCommitted changes persist even after a crash.
SQL — Transactions
-- Bank transfer: atomic debit + credit
BEGIN;
    UPDATE accounts SET balance = balance - 5000 WHERE id = 101;
    UPDATE accounts SET balance = balance + 5000 WHERE id = 202;
COMMIT;    -- both succeed together

-- Rollback on error
BEGIN;
    UPDATE inventory SET stock = stock-1 WHERE id=5;
    INSERT INTO orders (product_id, qty) VALUES (5, 1);
ROLLBACK;  -- cancels everything on error

-- Savepoints: partial rollback
BEGIN;
    INSERT INTO logs(msg) VALUES('start');
    SAVEPOINT sp1;
    UPDATE users SET status='locked' WHERE id=42;
    ROLLBACK TO sp1;  -- undo only the UPDATE
COMMIT;             -- INSERT still commits
22Advanced

Performance & EXPLAIN

Understanding query execution plans lets you diagnose and fix slow queries in production systems.

📊 Use EXPLAIN ANALYZE
Prefix any slow query with EXPLAIN ANALYZE to see how the database plans it. Look for "Seq Scan" on large tables — that means a missing index.
🗂️ Index What You Filter / Join On
Index columns used in WHERE, JOIN ON, and ORDER BY. Don't over-index — each index slows down writes slightly.
🎯 SELECT Only What You Need
Never SELECT * in production. Fewer columns = less I/O, less memory, faster transfers, especially on wide tables.
⚡ UNION ALL vs UNION
UNION deduplicates rows (expensive sort). If you know results won't overlap, use UNION ALL — no deduplication overhead.
🔗 Filter Before Joining
Reduce dataset size with WHERE before a JOIN. Use CTEs to pre-filter so the engine processes fewer rows during the join.
🚫 Avoid Functions on Indexed Columns
WHERE YEAR(date) = 2024 prevents index use. Use WHERE date >= '2024-01-01' AND date < '2025-01-01' instead.
SQL — EXPLAIN ANALYZE
-- Check query plan (PostgreSQL)
EXPLAIN ANALYZE
SELECT c.name, COUNT(o.id)
FROM   customers c
JOIN   orders    o ON c.id = o.customer_id
WHERE  c.country = 'India'
GROUP BY c.name;
-- "Seq Scan" = no index being used → add one
-- "Index Scan" = good, index is working

30 Real Interview Questions

Curated across easy, medium, and hard levels. Click any question to reveal the full answer. These cover conceptual understanding, coding scenarios, and system-design thinking.

🟢 Easy — Conceptual
EasyWhat is the difference between WHERE and HAVING?+
WHERE filters individual rows before any grouping. HAVING filters groups after GROUP BY is applied.

Key rule: if your condition involves an aggregate function (COUNT, SUM, AVG…), you must use HAVING — because WHERE runs before aggregation. 
SELECT department, AVG(salary)
FROM   employees
WHERE  is_active = 1             -- runs BEFORE GROUP BY
GROUP BY department
HAVING   AVG(salary) > 60000;  -- runs AFTER GROUP BY
EasyWhat is the difference between DELETE, TRUNCATE, and DROP?+
DELETE: Removes rows matching a WHERE clause. Can be rolled back. Fires triggers. Slower on large tables.

TRUNCATE: Removes ALL rows instantly. Cannot be rolled back in most databases. Resets auto-increment. No triggers. Much faster than DELETE for wiping a table.

DROP: Removes the entire table — structure, data, indexes. Cannot easily be undone.
EasyWhat is a PRIMARY KEY vs a UNIQUE KEY?+
PRIMARY KEY: Uniquely identifies each row. Cannot be NULL. Only ONE per table. Creates a clustered index automatically.

UNIQUE KEY: Enforces uniqueness, but CAN have NULL values (most databases allow multiple NULLs). A table can have multiple unique constraints.

Example: id is the PK; email has a UNIQUE constraint — a user might not have an email yet (NULL), but no two users can share one.
EasyExplain all JOIN types with a real-world example.+
INNER JOIN: Orders that have a matching customer. Non-matched rows excluded.
LEFT JOIN: All customers, even those who never placed an order (order columns are NULL).
RIGHT JOIN: Mirror of LEFT JOIN — all from the right table.
FULL OUTER JOIN: All rows from both sides; non-matches become NULL.
CROSS JOIN: Every customer paired with every product — Cartesian product. Use carefully.
SELF JOIN: Each employee paired with their manager, where both are rows in the same employees table.
EasyWhat is the difference between COUNT(*) and COUNT(column_name)?+
COUNT(*) counts ALL rows, including rows with NULL values in any column.

COUNT(column_name) counts only rows where that specific column is NOT NULL. 
SELECT
    COUNT(*)     AS total_users,        -- all rows
    COUNT(email) AS users_with_email    -- non-NULL emails only
FROM users;
EasyWhat is normalization? Explain 1NF, 2NF, and 3NF.+
Normalization organizes a database to reduce redundancy and improve integrity.

1NF: Each cell holds atomic (single) values. No repeating groups. Each row is unique.
2NF: Meets 1NF + all non-key columns depend on the ENTIRE primary key (eliminates partial dependencies).
3NF: Meets 2NF + no transitive dependencies — non-key columns don't depend on other non-key columns.

OLTP systems target 3NF for integrity. Data warehouses often use denormalized star/snowflake schemas for query speed.
EasyWhat is the difference between CHAR and VARCHAR?+
CHAR(n): Fixed length — always stores exactly n characters, padding shorter values with spaces. Fast for fixed-width data like country codes ('IN', 'US').

VARCHAR(n): Variable length — stores only the actual characters used, up to n. More space-efficient for names, emails, and descriptions of varying lengths.

Rule of thumb: CHAR for codes of fixed width; VARCHAR for everything else.
EasyWhat are SQL constraints? Name the main types.+
Constraints enforce rules on column data at the database level — they prevent invalid data from being inserted or updated.

NOT NULL: Column cannot be NULL.
UNIQUE: All values must be distinct (NULLs allowed).
PRIMARY KEY: Uniquely identifies each row — NOT NULL + UNIQUE combined.
FOREIGN KEY: References a primary key in another table — enforces referential integrity.
CHECK: Values must satisfy a custom condition: CHECK (salary >= 0).
DEFAULT: Provides a value when none is specified: DEFAULT NOW().
EasyWhat is a VIEW? What are its advantages?+
A VIEW is a saved SELECT query that you can query as if it were a table. The data is not physically stored (unless it's a materialized view).

Advantages:
• Simplicity: Complex multi-table queries become a simple SELECT from the view
• Security: Grant users access to a view without exposing the underlying tables
• Maintainability: Change the underlying query in one place instead of everywhere
• Abstraction: Hide schema changes from applications by updating the view definition
EasyWhat is a subquery? When would you use EXISTS vs IN?+
A subquery is a SELECT statement nested inside another query.

IN: Retrieves a list of values and checks membership. Works well with short lists from small tables.
EXISTS: Checks whether the subquery returns any rows at all. Short-circuits as soon as one match is found — more efficient with large subqueries.

Prefer EXISTS when the subquery result set is large; prefer IN for small known lists or when the values are from a small lookup table.

MediumWhat is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()?+
All three assign a number to each row, but handle ties differently: 
-- Salaries: 100, 90, 90, 80
ROW_NUMBER()  → 1, 2, 3, 4   -- always unique, arbitrary tie-breaking
RANK()        → 1, 2, 2, 4   -- ties share rank; NEXT rank skips
DENSE_RANK()  → 1, 2, 2, 3   -- ties share rank; next rank does NOT skip
Use ROW_NUMBER when you need exactly one row per result (deduplication).
Use RANK for sports-style rankings with gaps after ties.
Use DENSE_RANK when you don't want gaps in ranking numbers.

MediumWrite a query to calculate a 3-day moving average of daily revenue.+
SELECT
    order_date,
    daily_revenue,
    ROUND(
        AVG(daily_revenue) OVER (
            ORDER BY order_date
            ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
        ), 2

MediumWhat is a CTE? When would you use it over a subquery?+
A CTE (Common Table Expression) is a named temporary result set defined with the WITH clause. It exists only for the current query.

Use CTE when:
• Same subquery is referenced more than once
• You want to break a complex query into readable named steps
• You need recursion (hierarchies, graphs)
• Readability and maintainability matter

Use subquery when:
• Simple, one-off logic
• Legacy databases without CTE support

Performance is often identical — the query planner usually optimizes them equivalently. CTEs are primarily a readability tool.

    FROM   users
    GROUP BY email

HardFind users with 3 or more consecutive days of activity (gaps-and-islands).+
WITH daily AS (
    SELECT DISTINCT
        user_id,
        DATE(login_time) AS dt
    FROM user_sessions
),
grp AS (
    SELECT
        user_id, dt,
        dt - (ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY dt)
              )::INTEGER      AS island
    FROM daily
),
streaks AS (
    SELECT user_id, island, COUNT(*) AS streak_len
    FROM   grp
    GROUP BY user_id, island
)
SELECT DISTINCT user_id
FROM   streaks
WHERE  streak_len >= 3;
The trick: subtracting ROW_NUMBER from the date makes consecutive dates produce the same constant value — creating an "island" group.

Isolation LevelDirty ReadNon-Repeatable ReadPhantom Read

Dirty Read: Reading data from an uncommitted transaction that may be rolled back.
Non-Repeatable Read: Re-reading the same row gives a different result because another transaction updated it.
Phantom Read: Re-running the same query returns additional rows inserted by another transaction.

Most databases default to READ COMMITTED. PostgreSQL uses a snapshot-based approach that avoids most anomalies at REPEATABLE READ.
HardCalculate month-over-month user retention rate.+
WITH mau AS (
    SELECT DISTINCT
        user_id,
        DATE_TRUNC('month', activity_date) AS month
    FROM user_activity
),
retention AS (
    SELECT
        curr.month,
        COUNT(DISTINCT curr.user_id)  AS active_users,
        COUNT(DISTINCT prev.user_id)  AS retained
    FROM      mau curr
    LEFT JOIN mau prev
        ON curr.user_id = prev.user_id
       AND prev.month   = curr.month - INTERVAL '1 month'
    GROUP BY curr.month
)
SELECT
    month, active_users, retained,
    ROUND(retained*100.0 /
          NULLIF(LAG(active_users) OVER(ORDER BY month), 0), 1)
        AS retention_pct
FROM retention
ORDER BY month;
HardWhy does column order matter in composite indexes?+
A B-tree composite index on (department, salary) is sorted first by department, then by salary within each department.

The index is useful only when your query filters on the leftmost prefix: 
CREATE INDEX idx ON employees(department, salary);

-- Uses index (leftmost column present)
WHERE department = 'Eng'
WHERE department = 'Eng' AND salary > 80000

-- Cannot use index efficiently
WHERE salary > 80000  -- missing leftmost column
Rule: put the most-selective or most-frequently-queried column first. Put range-condition columns last.
HardWhat is a deadlock? How do you detect and prevent it?+
A deadlock occurs when two transactions each hold a lock the other needs, creating a circular wait where neither can proceed.

Detection: Most databases auto-detect deadlocks via a cycle-detection algorithm and automatically abort one transaction (the deadlock victim), returning an error to the application.

Prevention:
1. Always acquire locks in the same consistent order across all transactions
2. Keep transactions short — acquire and release locks quickly
3. Use lower isolation levels where correctness permits
4. Implement retry logic in application code on deadlock errors
5. Use SELECT ... FOR UPDATE SKIP LOCKED for queue-style patterns

HardRecursive CTE: find all direct and indirect reports of manager ID 5.+
WITH RECURSIVE reports AS (
    -- Base: direct reports of manager 5
    SELECT id, name, manager_id, 1 AS level
    FROM   employees
    WHERE  manager_id = 5

    UNION ALL

    -- Recursive: each report's direct reports
    SELECT e.id, e.name, e.manager_id, r.level+1
    FROM   employees e
    JOIN   reports   r ON e.manager_id = r.id
    WHERE  r.level < 10  -- safety limit prevents infinite loops
)
SELECT id, name, manager_id, level
FROM   reports
ORDER BY level, name;
Always include a depth limit to prevent infinite loops on cyclic data.
HardHow would you optimize a query doing a full scan on a 500M row table?+
A structured approach:

1. Diagnose: Run EXPLAIN ANALYZE. Identify Seq Scans on large tables, missing indexes, row estimate errors.

2. Index: Add a composite index matching your WHERE + ORDER BY columns. Use partial indexes if you always filter on a fixed condition (e.g., WHERE is_active = TRUE).

3. Partition: Range-partition the table by date. Queries with date filters will scan only relevant partitions.

4. Materialize: Create a materialized view for repeated aggregation queries. Refresh on a schedule.

5. Rewrite the query: Select only needed columns, filter early, avoid functions on indexed columns, replace correlated subqueries with window functions.

6. Archive: Move old/historical rows to a separate archive table or cold storage.

7. Read replicas: Offload heavy analytics queries to a read replica so production OLTP is unaffected.
HardWhat is optimistic vs pessimistic locking?+
Pessimistic Locking: Assumes conflicts are likely. Locks the row when you read it so no one else can modify it until you commit. 
BEGIN;
SELECT * FROM inventory
WHERE  product_id=5
FOR UPDATE;   -- row locked until COMMIT
UPDATE inventory SET stock=stock-1 WHERE product_id=5;
COMMIT;
Optimistic Locking: Assumes conflicts are rare. No lock on read. Before writing, checks if the data changed since it was read (via a version column). If changed, retry. 
-- Update only if version hasn't changed
UPDATE inventory
SET    stock   = stock-1,
       version = version+1
WHERE  product_id=5 AND version=7;
-- 0 rows updated → conflict → retry
Pessimistic is safer for high-contention scenarios; optimistic is better for high-read, low-conflict workloads.
HardHow would you design a leaderboard system in SQL that updates in near real-time?+
Schema design: 
CREATE TABLE scores (
    user_id    BIGINT    REFERENCES users(id),
    game_id    BIGINT,
    score      INT       NOT NULL,
    scored_at  TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_scores_user ON scores(user_id, scored_at);
Leaderboard query with ranking: 
SELECT
    u.username,
    SUM(s.score)                          AS total_score,
    RANK() OVER(ORDER BY SUM(s.score) DESC) AS rank
FROM   scores s
JOIN   users  u ON s.user_id = u.id
WHERE  s.scored_at >= NOW() - INTERVAL '7 days'
GROUP BY u.user_id, u.username
ORDER BY total_score DESC
LIMIT  100;
For real-time at scale: use a materialized view refreshed every minute, or maintain a separate leaderboard summary table updated via triggers or a background job. Redis sorted sets are often used alongside SQL for millisecond-latency reads.
