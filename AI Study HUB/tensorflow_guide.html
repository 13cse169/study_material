1. What is TensorFlow?

TensorFlow is Google's open-source machine learning framework for building, training, and deploying ML models at scale. It uses a dataflow graph to represent computations — nodes are operations, edges are tensors (multidimensional arrays) flowing between them.

🔢

Tensors

The fundamental data structure — N-dimensional arrays. Scalars (0D), vectors (1D), matrices (2D), and beyond.

⚙️

Operations

Math ops (add, matmul, relu) that consume and produce tensors. Automatically differentiable via GradientTape.

📦

Keras API

High-level API built into TF2. Lets you build, train, and evaluate models with minimal boilerplate.

🚀

Eager Execution

TF2 default: operations run immediately, like NumPy. Makes debugging and prototyping much easier.

🏭

tf.function

Compile Python functions into TF graphs for production-speed inference via tracing and XLA compilation.

🌐

TF Ecosystem

TF Serving, TF Lite, TF.js, TFX — one training pipeline deployed everywhere from edge to cloud.

2. Working with Tensors

Tensors are immutable, typed multidimensional arrays. Unlike NumPy arrays, they can live on GPU/TPU memory and have automatic gradient support.

Python · Tensor Basics 
import tensorflow as tf
import numpy as np

# ── Creating Tensors ──────────────────────────────────────
scalar   = tf.constant(42)                          # rank-0

zeros    = tf.zeros([3, 4])                          # 3×4 zero tensor

print(matrix.dtype)    # tf.int32

a = tf.constant([[1.,2.],[3.,4.]])

3. Variables & Automatic Differentiation

tf.Variable holds mutable state (model weights). tf.GradientTape records operations so you can compute gradients — the backbone of all training loops.

Python · GradientTape 
import tensorflow as tf

# tf.Variable = trainable weight
w = tf.Variable(3.0)
b = tf.Variable(0.0)

# ── Simple gradient computation ───────────────────────────
x = tf.constant(2.0)
with tf.GradientTape() as tape:
    y = w * x + b          # y = 3*2 + 0 = 6
    loss = (y - 5.0) ** 2  # loss = (6-5)² = 1

# ∂loss/∂w and ∂loss/∂b
dw, db = tape.gradient(loss, [w, b])
print(f"∂loss/∂w = {dw.numpy()}")   # 4.0  (2*(y-5)*x = 2*1*2)
print(f"∂loss/∂b = {db.numpy()}")   # 2.0  (2*(y-5)*1)

# ── Manual gradient descent step ──────────────────────────
lr = 0.1
w.assign_sub(lr * dw)    # w = w - lr * grad_w
b.assign_sub(lr * db)    # b = b - lr * grad_b
print(f"Updated w: {w.numpy()}")    # 2.6
print(f"Updated b: {b.numpy()}")    # -0.2

# ── Persistent tape for multiple gradient calls ───────────
with tf.GradientTape(persistent=True) as tape:
    z = w ** 2
grad_z_w = tape.gradient(z, w)   # dz/dw = 2w
del tape                           # Free resources when done
💡 Key insight: GradientTape tracks all operations on Variables inside the with block. When you call tape.gradient(loss, vars), TensorFlow uses the chain rule to compute all gradients in one backward pass.

from tensorflow import keras

# ── 1. Load & preprocess data ─────────────────────────────

6. Efficient Data Pipelines with tf.data

tf.data builds fast, scalable input pipelines with prefetching and parallel processing — essential for GPU utilization above 95%.

Python · tf.data Pipeline 
import tensorflow as tf
AUTOTUNE = tf.data.AUTOTUNE

# ── From NumPy arrays ─────────────────────────────────────
train_ds = tf.data.Dataset.from_tensor_slices((x_train, y_train))
train_ds = (train_ds
    .shuffle(10000)                  # shuffle buffer of 10k
    .batch(32)                        # mini-batch of 32
    .map(augment, num_parallel_calls=AUTOTUNE)  # parallel augment
    .prefetch(AUTOTUNE)               # overlap CPU & GPU work
)

# ── From image directory ──────────────────────────────────
train_ds = tf.keras.utils.image_dataset_from_directory(
    'data/train',
    image_size=(224, 224),
    batch_size=32
)

# ── Data Augmentation layer (runs on GPU) ─────────────────
augment = tf.keras.Sequential([
    tf.keras.layers.RandomFlip("horizontal"),
    tf.keras.layers.RandomRotation(0.1),
    tf.keras.layers.RandomZoom(0.1),
    tf.keras.layers.RandomContrast(0.1),
])

# ── Interleave for large datasets of TFRecord files ───────
files = tf.data.Dataset.list_files("data/*.tfrecord")
ds = files.interleave(
    tf.data.TFRecordDataset,
    cycle_length=8,
    num_parallel_calls=AUTOTUNE
)
⚡ Performance tip: Always end your pipeline with .prefetch(AUTOTUNE). Without it, the GPU waits idle for the CPU to prepare the next batch. Prefetch overlaps data preprocessing with model execution.

7. Custom Training Loops

When you need full control — gradient clipping, mixed precision, custom metrics — skip model.fit() and write your own training loop with GradientTape.

Python · Custom Training Loop 
import tensorflow as tf

optimizer  = tf.keras.optimizers.Adam(1e-3)
loss_fn    = tf.keras.losses.SparseCategoricalCrossentropy()
train_acc  = tf.keras.metrics.SparseCategoricalAccuracy()
val_acc    = tf.keras.metrics.SparseCategoricalAccuracy()

# ── Decorate with tf.function for graph compilation ───────
@tf.function
def train_step(x, y):
    with tf.GradientTape() as tape:
        logits = model(x, training=True)
        loss = loss_fn(y, logits)
        loss += sum(model.losses)   # regularization losses
    
    grads = tape.gradient(loss, model.trainable_variables)
    
    # Gradient clipping (prevents exploding gradients)
    grads, _ = tf.clip_by_global_norm(grads, clip_norm=1.0)
    
    optimizer.apply_gradients(zip(grads, model.trainable_variables))
    train_acc.update_state(y, logits)
    return loss

@tf.function
def val_step(x, y):
    logits = model(x, training=False)
    val_acc.update_state(y, logits)

# ── Training loop ─────────────────────────────────────────
for epoch in range(20):
    for x_batch, y_batch in train_ds:
        loss = train_step(x_batch, y_batch)
    
    for x_val, y_val in val_ds:
        val_step(x_val, y_val)
    
    print(f"Epoch {epoch+1}: loss={loss:.4f} | "
          f"train_acc={train_acc.result():.4f} | "
          f"val_acc={val_acc.result():.4f}")
    
    train_acc.reset_state()
    val_acc.reset_state()

10. Custom Layers & Models

Build any architecture by subclassing tf.keras.Layer and tf.keras.Model. This gives you full control over forward pass, trainable weights, and sub-layer composition.

Python · Custom Layer & Residual Block 
import tensorflow as tf

# ── Custom Layer ──────────────────────────────────────────
class LinearWithBias(tf.keras.Layer):
    def __init__(self, units):
        super().__init__()
        self.units = units
    
    def build(self, input_shape):
        # Weights created lazily on first call
        self.w = self.add_weight(shape=(input_shape[-1], self.units),
                                  initializer='glorot_uniform',
                                  trainable=True)
        self.b = self.add_weight(shape=(self.units,),
                                  initializer='zeros',
                                  trainable=True)
    
    def call(self, x):
        return tf.matmul(x, self.w) + self.b

# ── Residual Block (skip connection) ─────────────────────
class ResidualBlock(tf.keras.Layer):
    def __init__(self, filters):
        super().__init__()
        self.conv1 = tf.keras.layers.Conv2D(filters, 3, padding='same')
        self.conv2 = tf.keras.layers.Conv2D(filters, 3, padding='same')
        self.bn1   = tf.keras.layers.BatchNormalization()
        self.bn2   = tf.keras.layers.BatchNormalization()
        self.proj  = tf.keras.layers.Conv2D(filters, 1)  # dim match
    
    def call(self, x, training=False):
        residual = self.proj(x)
        x = tf.nn.relu(self.bn1(self.conv1(x), training=training))
        x = self.bn2(self.conv2(x), training=training)
        return tf.nn.relu(x + residual)   # skip connection!

# ── Custom Model subclass ─────────────────────────────────
class MiniResNet(tf.keras.Model):
    def __init__(self, num_classes):
        super().__init__()
        self.stem   = tf.keras.layers.Conv2D(64, 7, strides=2, padding='same')
        self.block1 = ResidualBlock(64)
        self.block2 = ResidualBlock(128)
        self.pool   = tf.keras.layers.GlobalAveragePooling2D()
        self.fc     = tf.keras.layers.Dense(num_classes, activation='softmax')
    
    def call(self, x, training=False):
        x = tf.nn.relu(self.stem(x))
        x = self.block1(x, training=training)
        x = self.block2(x, training=training)
        x = self.pool(x)
        return self.fc(x)

11. Building a Transformer from Scratch

The Transformer architecture powers all modern NLP. Here's multi-head attention and a Transformer encoder block implemented in TensorFlow.

Python · Multi-Head Attention 
import tensorflow as tf

class MultiHeadAttention(tf.keras.Layer):
    def __init__(self, d_model, num_heads):
        super().__init__()
        self.num_heads = num_heads
        self.d_model   = d_model
        assert d_model % num_heads == 0
        self.depth = d_model // num_heads

        self.wq = tf.keras.layers.Dense(d_model)
        self.wk = tf.keras.layers.Dense(d_model)
        self.wv = tf.keras.layers.Dense(d_model)
        self.wo = tf.keras.layers.Dense(d_model)
    
    def split_heads(self, x, batch_size):
        x = tf.reshape(x, (batch_size, -1, self.num_heads, self.depth))
        return tf.transpose(x, perm=[0,2,1,3])   # (B, H, T, D)

    def call(self, q, k, v, mask=None):
        B = tf.shape(q)[0]
        Q = self.split_heads(self.wq(q), B)
        K = self.split_heads(self.wk(k), B)
        V = self.split_heads(self.wv(v), B)

        # Scaled dot-product attention
        scores = tf.matmul(Q, K, transpose_b=True) / tf.math.sqrt(
                     tf.cast(self.depth, tf.float32))
        if mask is not None:
            scores += (mask * -1e9)
        weights = tf.nn.softmax(scores, axis=-1)
        ctx = tf.matmul(weights, V)                 # (B,H,T,D)
        ctx = tf.transpose(ctx, perm=[0,2,1,3])    # (B,T,H,D)
        ctx = tf.reshape(ctx, (B, -1, self.d_model)) # (B,T,d_model)
        return self.wo(ctx)

class TransformerBlock(tf.keras.Layer):
    def __init__(self, d_model, num_heads, dff, dropout_rate=0.1):
        super().__init__()
        self.attn = MultiHeadAttention(d_model, num_heads)
        self.ffn  = tf.keras.Sequential([
            tf.keras.layers.Dense(dff, activation='relu'),
            tf.keras.layers.Dense(d_model)
        ])
        self.norm1   = tf.keras.layers.LayerNormalization(epsilon=1e-6)
        self.norm2   = tf.keras.layers.LayerNormalization(epsilon=1e-6)
        self.dropout = tf.keras.layers.Dropout(dropout_rate)
    
    def call(self, x, training=False, mask=None):
        attn_out = self.attn(x, x, x, mask)
        x = self.norm1(x + self.dropout(attn_out, training=training))
        ffn_out  = self.ffn(x)
        x = self.norm2(x + self.dropout(ffn_out,  training=training))
        return x

12. Distributed Training & tf.function

Scale training across multiple GPUs or TPUs with tf.distribute. Switching strategies requires minimal code changes — most of the complexity is handled by TF.

Python · Multi-GPU & tf.function 
import tensorflow as tf

# ── Multi-GPU strategy ────────────────────────────────────
strategy = tf.distribute.MirroredStrategy()
# Copies model to each GPU; gradients averaged across all
print(f"GPUs: {strategy.num_replicas_in_sync}")

with strategy.scope():
    model = build_model()   # model created inside scope
    model.compile(
        optimizer='adam',
        loss='sparse_categorical_crossentropy',
        metrics=['accuracy']
    )

# Scale batch size with number of GPUs
BATCH_PER_REPLICA = 64
GLOBAL_BATCH = BATCH_PER_REPLICA * strategy.num_replicas_in_sync

# Same model.fit() API — strategy handles distribution
model.fit(train_ds, epochs=20)

# ── TPU strategy ──────────────────────────────────────────
try:
    resolver = tf.distribute.cluster_resolver.TPUClusterResolver()
    tf.config.experimental_connect_to_cluster(resolver)
    tf.tpu.experimental.initialize_tpu_system(resolver)
    strategy = tf.distribute.TPUStrategy(resolver)
except ValueError:
    strategy = tf.distribute.MirroredStrategy()  # fallback to GPU

# ── tf.function: compile Python → TF graph ───────────────
@tf.function(input_signature=[
    tf.TensorSpec(shape=[None, 28, 28], dtype=tf.float32)
])
def fast_predict(x):
    return model(x, training=False)

# First call: traces (slow). Subsequent calls: graph (fast)
result = fast_predict(x_test[:100])   # ~5× faster than eager

13. Model Saving, Export & TF Serving

SavedModel is TensorFlow's universal serialization format — captures weights, computation graph, and signatures. Deploy the same artifact to TF Serving, TF Lite, or TF.js.

Python · Save, Load, Export 
import tensorflow as tf

# ── SavedModel format (recommended) ──────────────────────
model.save('my_model')           # saves whole model
model = tf.keras.models.load_model('my_model')

# ── Keras H5 format ───────────────────────────────────────
model.save('model.h5')
model = tf.keras.models.load_model('model.h5')

# ── Save only weights (requires matching architecture) ────
model.save_weights('weights.ckpt')
model.load_weights('weights.ckpt')

# ── Export with custom serving signature ──────────────────
@tf.function(input_signature=[
    tf.TensorSpec(shape=[None, 224, 224, 3], dtype=tf.float32)
])
def serving_fn(image):
    probabilities = model(image, training=False)
    class_id = tf.argmax(probabilities, axis=-1)
    return {'class_id': class_id, 'probabilities': probabilities}

tf.saved_model.save(model, 'export/',
    signatures={'serving_default': serving_fn})

# ── Convert to TF Lite (mobile/edge deployment) ───────────
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]  # quantize
tflite_model = converter.convert()
with open('model.tflite', 'wb') as f:
    f.write(tflite_model)

14. Mixed Precision & Performance Optimization

Python · Mixed Precision + XLA 
from tensorflow.keras import mixed_precision

# ── Enable FP16 compute (2-3× speedup on modern GPUs) ─────
mixed_precision.set_global_policy('mixed_float16')

model = tf.keras.Sequential([
    tf.keras.layers.Dense(1024, activation='relu'),
    # Final layer must be float32 for numerical stability
    tf.keras.layers.Dense(10, activation='softmax',
                           dtype='float32')
])

# Use LossScaleOptimizer to prevent underflow
optimizer = tf.keras.optimizers.Adam(1e-3)
optimizer = mixed_precision.LossScaleOptimizer(optimizer)

# ── XLA JIT compilation ───────────────────────────────────
@tf.function(jit_compile=True)   # enable XLA for this function
def train_step(x, y):
    with tf.GradientTape() as tape:
        logits = model(x, training=True)
        loss = loss_fn(y, logits)
        scaled = optimizer.get_scaled_loss(loss)
    grads = tape.gradient(scaled, model.trainable_variables)
    grads = optimizer.get_unscaled_gradients(grads)
    optimizer.apply_gradients(zip(grads, model.trainable_variables))
    return loss

Q01 What is the difference between tf.constant and tf.Variable? ›

tf.constant creates an immutable tensor. Its value cannot change after creation and it has no gradient tracking by default. Used for input data, hyperparameters, and fixed values.

tf.Variable creates a mutable tensor that persists across multiple calls to tf.function. Variables automatically tracked by GradientTape, making them the correct container for model weights and biases. Use var.assign(new_value) or var.assign_add/sub(delta) to update them.

Key distinction: TF optimizers update Variables via gradients. Attempting to create model weights as constants and update them manually breaks the TF graph execution model.

Tensors Core
Q02 Explain the vanishing gradient problem and how to solve it in TensorFlow. ›

In deep networks, gradients are multiplied layer-by-layer during backpropagation. When activation derivatives are <1 (like sigmoid's max derivative of 0.25), this repeated multiplication causes gradients to shrink exponentially toward the input layers — making early layers learn extremely slowly or not at all.

Solutions in TF/Keras:

• ReLU activations: gradient = 1 for positive inputs (no squashing)
• Batch Normalization (layers.BatchNormalization()): normalizes activations per batch, preventing extreme values
• Residual connections: skip connections let gradients flow directly to early layers (ResNet)
• Glorot/He initialization: kernel_initializer='he_uniform' keeps activations in good range
• Gradient clipping: tf.clip_by_global_norm(grads, 1.0)
• LSTM/GRU: designed with gates to preserve gradients over long sequences

Training Architecture
Q03 What does tf.function do and when should you use it? ›

@tf.function traces a Python function and compiles it into a TensorFlow computation graph. On the first call, TF traces the function (executing Python to build a graph). Subsequent calls execute the compiled graph, bypassing Python overhead.

Benefits: 2–10× speedup over eager execution for training steps, supports XLA compilation with jit_compile=True, enables deployment as SavedModel signatures.

When to use: Wrap your train_step and val_step functions. Don't decorate everything — Python-side logic (logging, early stopping checks) should stay outside.

Gotchas: Each unique Python argument value triggers a retrace. Don't pass Python scalars as arguments — use tf.Tensors or tf.Variables instead to avoid excessive tracing.

Performance
Q04 What is Batch Normalization and why does it help training? ›

Batch Normalization normalizes each feature to have zero mean and unit variance across the current mini-batch, then applies learnable scale (γ) and shift (β) parameters: y = γ·(x − μ)/σ + β.

Benefits:

1. Reduces internal covariate shift — each layer gets inputs in a stable distribution
2. Allows much higher learning rates without divergence
3. Acts as a regularizer, often reducing need for Dropout
4. Reduces sensitivity to weight initialization
5. Significantly faster convergence

Critical detail: BatchNorm behaves differently during training (training=True) vs inference (training=False). During training it uses batch statistics; during inference it uses running mean/variance accumulated during training. Always pass the training flag correctly.

Q06 What is the difference between model.fit() and a custom training loop? ›

model.fit(): Handles the entire training loop automatically — batching, callbacks, metric tracking, multi-GPU distribution, mixed precision, progress bars. Ideal for standard training scenarios, fast prototyping, and when built-in callbacks cover your needs.

Custom loop: Write your own for epoch in range(...): loop with GradientTape. Necessary when you need: custom loss computation (e.g., multiple interleaved models like GANs), gradient clipping with custom logic, curriculum learning, meta-learning (MAML), non-standard backward passes, or research experiments requiring step-level control.

A good middle ground: override model.train_step() to customize a single training step while keeping model.fit()'s callback/logging infrastructure.

Training
Q07 How does tf.data improve training performance? ›

tf.data pipelines overlap data preprocessing with model execution, eliminating GPU starvation. Key performance techniques:

1. Prefetch: While the GPU trains on batch N, CPU prepares batch N+1 in parallel. Always end with .prefetch(AUTOTUNE)
2. Parallel map: .map(fn, num_parallel_calls=AUTOTUNE) preprocesses multiple examples simultaneously
3. Cache: .cache() stores processed data in memory after first epoch
4. TFRecords: Binary format with better I/O than JPEGs for large datasets
5. Vectorized map: Process entire batches at once instead of per-element

Without these optimizations, a powerful GPU can sit 40–70% idle waiting for data.

Data Performance
Q08 What is transfer learning and when does fine-tuning help vs hurt? ›

Transfer learning reuses a model pretrained on a large dataset (e.g., ImageNet) as a starting point for a related task, leveraging learned feature hierarchies instead of random initialization.

Fine-tuning helps when:

• Your dataset has >1000 images per class
• Your domain is moderately different from the pretrain domain
• You unfreeze only the top layers with a small learning rate (1e-5 to 1e-4)

Fine-tuning hurts when:

• Your dataset is tiny (<500 samples): overfitting risk. Use frozen features + new head only
• Learning rate is too high: destroys pretrained features ("catastrophic forgetting")
• You unfreeze too many layers with a small dataset

Transfer Learning
Q09 How do you handle class imbalance in TensorFlow? ›

Class imbalance (e.g., 99% negative, 1% positive) causes models to predict the majority class always. TF approaches:

1. Class weights: model.fit(class_weight={0: 1.0, 1: 99.0}) — penalizes misclassifying the minority class more
2. Sample weights: Pass sample_weight array with per-sample importance to model.fit()
3. Oversampling minority: Use tf.data to sample minority class more frequently
4. Undersampling majority: Take random subset of majority class
5. Focal loss: Custom loss that down-weights easy examples
6. Metrics: Use AUC-ROC, precision-recall, F1 instead of accuracy — accuracy is meaningless on imbalanced data

Architecture
Q11 What are the different optimizers and when should you use Adam vs SGD? ›

SGD with momentum: Slow to converge but often finds flatter minima → better generalization. Required for competitive image classification results (ResNet on ImageNet). Works well with learning rate schedules (cosine annealing, warm restarts).

Adam: Adapts learning rate per parameter using first and second moment estimates. Fast to converge, robust to hyperparameter choices, great for NLP, transformers, and default for most tasks. Can overfit on small datasets.

AdamW: Adam + proper weight decay (decoupled from gradient scaling). Standard for transformer training (BERT, GPT fine-tuning).

Practical rule: Start with Adam(1e-3). If generalization is poor, switch to SGD(0.01, momentum=0.9) + cosine LR schedule.

Optimizers
Q12 How does MirroredStrategy work for multi-GPU training? ›

MirroredStrategy implements synchronous data parallelism:

1. The model is copied (mirrored) to each GPU
2. Each GPU receives a different slice of the global batch (batch_size / num_gpus)
3. Each GPU computes forward + backward pass on its slice independently
4. Gradients from all GPUs are aggregated using all-reduce (default: NCCL for GPU-GPU communication)
5. All GPU replicas receive the averaged gradient and update their weights identically
6. Result: all replicas stay in sync; computation scales with number of GPUs

Important: Scale global batch size with num_gpus (e.g., 32 per GPU × 8 GPUs = 256 global batch). You may need to scale the learning rate accordingly (linear scaling rule: lr × num_gpus).

Distributed
Q13 What is the SavedModel format and how is it different from H5? ›

SavedModel (TensorFlow's native format): Saves the full TF program including weights, computation graph (as tf.functions), optimizer state, and custom serving signatures. Language/framework agnostic — load in C++, Java, TF Serving, TF.js without Python. Supports custom objects without special registration.

HDF5 (.h5): Legacy Keras-specific format. Stores architecture (as JSON) + weights (as numpy arrays). Requires Python and Keras to load. Doesn't save tf.function graphs. Can have issues with custom objects.

Recommendation: Always use SavedModel (model.save('path/to/dir')). Use H5 only for interoperability with older pipelines or when you specifically need the JSON architecture representation.

Deployment
Q14 Explain mixed precision training — what is it and why use it? ›

Mixed precision training uses float16 (half precision) for most computations and float32 (single precision) only where numerical stability requires it (output layer, loss computation, weight updates).

Benefits:

• Speed: 2–3× faster on Tensor Core GPUs (V100, A100, RTX series) due to specialized FP16 matrix multiplication units
• Memory: FP16 uses half the GPU memory → larger batch sizes or larger models
• Bandwidth: Smaller tensors transfer faster between memory and compute

The catch — underflow: FP16 has limited dynamic range. Small gradients underflow to zero. LossScaleOptimizer multiplies the loss by a large scale factor before backward pass, then unscales gradients before the weight update, preserving small gradient values.

Performance Training
Q15 How do you debug a TensorFlow model that isn't training (loss not decreasing)? ›

Systematic debugging checklist:

1. Sanity check with tiny dataset: Can the model overfit 1 batch? If not, architecture or loss is wrong
2. Check loss function: Using softmax output with categorical_crossentropy? Or logits with from_logits=True? Mismatch silently gives wrong gradients
3. Learning rate: Too high → loss explodes or oscillates. Too low → no movement. Try 1e-3 with Adam first
4. Gradient flow: Use tf.GradientTape manually and print gradient norms per layer. NaN or zero gradients = problem
5. Data preprocessing: Are inputs normalized? Wrong dtype? Check x_train.min(), max(), mean()
6. training=True/False: BatchNorm and Dropout must receive correct flag. Wrong flag = train/eval discrepancy
7. Callbacks: Add TensorBoard callback — visualize loss curves, weight histograms, gradient distributions
