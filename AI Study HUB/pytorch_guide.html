STEP 01 Why PyTorch? Core Philosophy

PyTorch uses a define-by-run (dynamic) computation graph — the graph is built on-the-fly as operations execute. This makes debugging trivial: you can use standard Python debugging tools, print tensors mid-forward-pass, and change the graph shape based on data. Compare this to TensorFlow 1.x's static graph approach.

⚡

Dynamic Graphs

Graph rebuilt every forward pass. Enables variable-length inputs, conditional logic, and easy debugging with Python's native tools.

🐍

Pythonic API

Feels like NumPy with GPU support and autograd. No sessions, no placeholders — just Python objects and functions.

🔬

Research-First

Dominant in research (used by ~80% of ML papers). New ideas are prototyped in PyTorch first. Transformers, diffusion models, NeRFs all born here.

🏭

TorchScript

Compile models to a static graph via tracing or scripting for production deployment without Python overhead.

🌐

Ecosystem

torchvision, torchaudio, torchtext, HuggingFace Transformers, PyTorch Lightning, ONNX export — one framework for everything.

⚖️

vs TensorFlow

PyTorch: research, flexibility, pythonic. TF: production pipelines, TFX, mobile-first. Both excellent — know PyTorch first.

STEP 03 Autograd — Automatic Differentiation

PyTorch's autograd engine records all operations on tensors with requires_grad=True into a computation graph. Calling .backward() traverses this graph in reverse, computing gradients via the chain rule.

x (input)
→
y = x²+3x
→
loss
→
loss.backward()
→
x.grad = ∂loss/∂x
Python · Autograd Deep Dive 
import torch

# ── Simple gradient computation ───────────────────────────
x = torch.tensor(3.0, requires_grad=True)
y = x ** 2 + 3 * x + 2   # y = x²+3x+2
y.backward()              # compute dy/dx
print(x.grad)              # dy/dx = 2x+3 = 2(3)+3 = 9

# ── Multi-variable gradients ──────────────────────────────
a = torch.tensor([2.0, 3.0], requires_grad=True)
b = torch.tensor([4.0, 5.0], requires_grad=True)
loss = (a * b).sum()       # loss = 2×4 + 3×5 = 23
loss.backward()
print(a.grad)              # ∂loss/∂a = b = [4, 5]
print(b.grad)              # ∂loss/∂b = a = [2, 3]

# ── Gradient accumulation (common bug!) ───────────────────
w = torch.tensor(1.0, requires_grad=True)
for _ in range(3):
    loss = w ** 2
    loss.backward()
    print(w.grad)    # 2, 4, 6 — gradients ACCUMULATE!
    w.grad.zero_()  # must zero before next iteration

# ── Detach & no_grad ──────────────────────────────────────
with torch.no_grad():
    inference = model(x)   # no gradient tracking → faster

detached = t.detach()    # new tensor, no gradient history

# ── Inspecting the computation graph ─────────────────────
x = torch.tensor(2.0, requires_grad=True)
y = x * 3
z = y + 1
print(z.grad_fn)           # AddBackward0
print(z.grad_fn.next_functions)  # [(MulBackward0,0),(None,0)]
⚠️ Most common beginner bug: Gradients accumulate by default. Always call optimizer.zero_grad() (or param.grad.zero_()) before computing gradients for the next batch.

STEP 04 Your First Neural Network with nn.Module

Every PyTorch model inherits from nn.Module. You define the layers in __init__ and the forward computation in forward(). PyTorch automatically handles backward propagation.

Python · nn.Module + Training Loop 
import torch
import torch.nn as nn
import torch.optim as optim
from torchvision import datasets, transforms
from torch.utils.data import DataLoader

# ── 1. Define Model ───────────────────────────────────────
class MLP(nn.Module):
    def __init__(self, input_dim=784, hidden=256, num_classes=10):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden),   # Wx + b
            nn.ReLU(),
            nn.Dropout(0.2),
            nn.Linear(hidden, 128),
            nn.ReLU(),
            nn.Linear(128, num_classes)    # logits (no softmax)
        )
    
    def forward(self, x):
        x = x.flatten(start_dim=1)  # [B,1,28,28] → [B,784]
        return self.net(x)

# ── 2. Data Loading ───────────────────────────────────────
transform = transforms.Compose([
    transforms.ToTensor(),
    transforms.Normalize((0.1307,), (0.3081,))
])
train_loader = DataLoader(
    datasets.MNIST('data', train=True,  download=True, transform=transform),
    batch_size=64, shuffle=True
)

# ── 3. Setup ──────────────────────────────────────────────
device    = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
model     = MLP().to(device)
optimizer = optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

# ── 4. Training Loop ──────────────────────────────────────
for epoch in range(10):
    model.train()           # enable Dropout, BatchNorm
    total_loss, correct = 0, 0

    for images, labels in train_loader:
        images, labels = images.to(device), labels.to(device)
        
        optimizer.zero_grad()         # ① zero gradients
        logits = model(images)          # ② forward pass
        loss   = criterion(logits, labels)  # ③ compute loss
        loss.backward()               # ④ backprop
        optimizer.step()              # ⑤ update weights
        
        total_loss += loss.item()
        correct += (logits.argmax(dim=1) == labels).sum().item()
    
    acc = correct / len(train_loader.dataset)
    print(f"Epoch {epoch+1}: loss={total_loss/len(train_loader):.4f} acc={acc:.4f}")

import torch

class TabularDataset(Dataset):

STEP 06 Convolutional Neural Networks

CNNs exploit spatial locality and parameter sharing. A Conv2d layer applies learnable filters across an input image, detecting edges → textures → shapes as depth increases.

Python · CNN for Image Classification 
import torch.nn as nn
import torch.nn.functional as F

class ConvNet(nn.Module):
    def __init__(self, num_classes=10):
        super().__init__()
        
        # ── Feature Extractor ────────────────────────────────
        self.features = nn.Sequential(
            # Block 1: [B,3,32,32] → [B,32,32,32]
            nn.Conv2d(3, 32, kernel_size=3, padding=1),
            nn.BatchNorm2d(32),
            nn.ReLU(inplace=True),
            nn.Conv2d(32, 32, kernel_size=3, padding=1),
            nn.ReLU(inplace=True),
            nn.MaxPool2d(2, 2),           # → [B,32,16,16]
            nn.Dropout2d(0.25),

            # Block 2: [B,32,16,16] → [B,64,8,8]
            nn.Conv2d(32, 64, kernel_size=3, padding=1),
            nn.BatchNorm2d(64),
            nn.ReLU(inplace=True),
            nn.Conv2d(64, 64, kernel_size=3, padding=1),
            nn.ReLU(inplace=True),
            nn.MaxPool2d(2, 2),           # → [B,64,8,8]
        )
        
        # ── Classifier ───────────────────────────────────────
        self.classifier = nn.Sequential(
            nn.Dropout(0.5),
            nn.Linear(64 * 8 * 8, 512),
            nn.ReLU(inplace=True),
            nn.Linear(512, num_classes)
        )
    
    def forward(self, x):
        x = self.features(x)
        x = x.flatten(start_dim=1)    # or x.view(x.size(0),-1)
        return self.classifier(x)

# Conv2d signature:
# nn.Conv2d(in_channels, out_channels, kernel_size,
#           stride=1, padding=0, dilation=1, groups=1, bias=True)
# Output size: floor((W - K + 2P) / S) + 1

STEP 10 Custom Autograd Functions

When you need non-standard gradients — sparse operations, custom CUDA kernels, straight-through estimators — implement torch.autograd.Function with explicit forward and backward methods.

Python · Custom Backward Pass 
import torch
from torch.autograd import Function

# ── Straight-Through Estimator (used in quantization/BNNs) ──
class StraightThrough(Function):
    @staticmethod
    def forward(ctx, x):
        # Forward: sign function (non-differentiable)
        ctx.save_for_backward(x)
        return x.sign()
    
    @staticmethod
    def backward(ctx, grad_output):
        # Backward: pass gradient through as-is (STE trick)
        x, = ctx.saved_tensors
        grad_input = grad_output.clone()
        grad_input[x.abs() > 1] = 0   # zero out if |x|>1
        return grad_input

# Use as a regular function
sign = StraightThrough.apply
x = torch.randn(4, requires_grad=True)
y = sign(x)
y.sum().backward()
print(x.grad)     # valid gradients despite non-differentiable sign

# ── Custom gradient for numerical stability ───────────────
class StableLogSigmoid(Function):
    @staticmethod
    def forward(ctx, x):
        sig = torch.sigmoid(x)
        ctx.save_for_backward(sig)
        return sig.log()

    @staticmethod
    def backward(ctx, grad):
        sig, = ctx.saved_tensors
        return grad * (1 - sig)  # analytically stable gradient

STEP 11 Transformer from Scratch in PyTorch

Python · Multi-Head Attention + Transformer Block 
import torch
import torch.nn as nn
import math

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model=512, num_heads=8, dropout=0.1):
        super().__init__()
        assert d_model % num_heads == 0
        self.d_k = d_model // num_heads
        self.h   = num_heads
        self.qkv = nn.Linear(d_model, 3 * d_model, bias=False)
        self.out = nn.Linear(d_model, d_model)
        self.drop = nn.Dropout(dropout)

    def forward(self, x, mask=None):
        B, T, C = x.shape
        # Project and split into Q,K,V
        qkv = self.qkv(x).reshape(B, T, 3, self.h, self.d_k)
        qkv = qkv.permute(2, 0, 3, 1, 4)   # (3, B, H, T, d_k)
        q, k, v = qkv.unbind(0)

        # Scaled dot-product attention
        scale  = math.sqrt(self.d_k)
        scores = (q @ k.transpose(-2, -1)) / scale   # (B,H,T,T)
        if mask is not None:
            scores = scores.masked_fill(mask == 0, -1e9)
        attn   = self.drop(torch.nn.functional.softmax(scores, dim=-1))
        out    = (attn @ v).transpose(1,2).reshape(B, T, C)
        return self.out(out)

class TransformerBlock(nn.Module):
    def __init__(self, d_model, num_heads, mlp_ratio=4, dropout=0.1):
        super().__init__()
        self.attn  = MultiHeadAttention(d_model, num_heads, dropout)
        self.ffn   = nn.Sequential(
            nn.Linear(d_model, mlp_ratio * d_model),
            nn.GELU(),
            nn.Dropout(dropout),
            nn.Linear(mlp_ratio * d_model, d_model),
        )
        self.norm1 = nn.LayerNorm(d_model)
        self.norm2 = nn.LayerNorm(d_model)
        self.drop  = nn.Dropout(dropout)

    def forward(self, x, mask=None):
        # Pre-LayerNorm variant (more stable training)
        x = x + self.drop(self.attn(self.norm1(x), mask))
        x = x + self.drop(self.ffn(self.norm2(x)))
        return x

# ── Quick test: batch of 4, sequence of 16, d_model=256 ──
blk = TransformerBlock(d_model=256, num_heads=8)
x   = torch.randn(4, 16, 256)
print(blk(x).shape)    # torch.Size([4, 16, 256])

STEP 12 Distributed Data Parallel (DDP)

DDP is PyTorch's recommended multi-GPU approach. Unlike DataParallel (Python GIL bottleneck), DDP launches one process per GPU, communicates via NCCL, and achieves near-linear scaling.

Python · DDP Training · torchrun 
# Run: torchrun --nproc_per_node=4 train.py
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data.distributed import DistributedSampler
import os

def main():
    # ── Initialize process group ─────────────────────────
    dist.init_process_group(backend='nccl')  # 'gloo' for CPU
    local_rank = int(os.environ['LOCAL_RANK'])
    torch.cuda.set_device(local_rank)
    device = torch.device(f'cuda:{local_rank}')

    # ── Wrap model ───────────────────────────────────────
    model = MyModel().to(device)
    model = DDP(model, device_ids=[local_rank])

    # ── Distributed sampler (no overlap between GPUs) ────
    sampler = DistributedSampler(dataset)
    loader  = DataLoader(dataset, batch_size=32, sampler=sampler)

    for epoch in range(100):
        sampler.set_epoch(epoch)   # critical! different shuffle each epoch
        for x, y in loader:
            x, y = x.to(device), y.to(device)
            optimizer.zero_grad()
            loss = criterion(model(x), y)
            loss.backward()            # DDP all-reduces gradients here
            optimizer.step()

    # ── Save only from rank 0 to avoid duplicate writes ──
    if dist.get_rank() == 0:
        torch.save(model.module.state_dict(), 'model.pth')
    
    dist.destroy_process_group()

if __name__ == '__main__':
    main()
💡 DataParallel vs DDP: DataParallel is single-process, multi-thread — limited by Python's GIL. DDP is multi-process — one process per GPU, no GIL contention. Always use DDP for serious multi-GPU training. It's ~3× faster than DP on 4 GPUs.

STEP 13 Mixed Precision & torch.compile

Python · AMP + torch.compile 
import torch
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()   # scales loss to prevent FP16 underflow

for x, y in loader:
    x, y = x.to(device), y.to(device)
    optimizer.zero_grad()

    # Forward pass in FP16 (where safe)
    with autocast(device_type='cuda', dtype=torch.float16):
        logits = model(x)
        loss   = criterion(logits, y)

    # Scale loss → backward → unscale → clip → step
    scaler.scale(loss).backward()
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(optimizer)
    scaler.update()

# ── torch.compile (PyTorch 2.0+) ──────────────────────────
# Compiles model into optimized kernel using TorchInductor
# ~30-100% speedup with zero code changes
model = torch.compile(model)                    # default mode
model = torch.compile(model, mode='reduce-overhead')  # minimize overhead
model = torch.compile(model, mode='max-autotune')     # max speed, slow compile

# ── BF16: better than FP16 on A100/H100/TPU ──────────────
with autocast(device_type='cuda', dtype=torch.bfloat16):
    output = model(x)    # BF16: wider range, no GradScaler needed

15 critical questions for ML Engineer, DL Research Engineer, and AI roles

15 Questions
Q01 What is the difference between a static and dynamic computation graph? Why does PyTorch use dynamic? ›

Static graph (TF1, Theano): The computation graph is defined first as a blueprint, then compiled and executed. Changing the graph requires recompilation. Enables heavy optimization (fusion, XLA) but makes debugging extremely hard — you can't step through with a debugger or use Python's print mid-computation.

Dynamic graph (PyTorch's "define-by-run"): The graph is constructed on-the-fly as Python code executes. Every forward pass can have a different graph. This means standard Python debugging works, variable-length sequences are trivial, and research experimentation is fast.

PyTorch's answer to production speed: torch.compile and TorchScript capture and optimize dynamic graphs for deployment, giving you the best of both worlds.

AutogradArchitecture
Q02 Explain how PyTorch's autograd engine works under the hood. ›

When you perform operations on tensors with requires_grad=True, PyTorch builds a directed acyclic graph (DAG) of Function objects. Each tensor has a grad_fn pointing to the function that created it (e.g., MulBackward0).

Calling .backward() does a reverse-mode traversal of this DAG: starting from the output, it calls each function's backward() method to compute the gradient of the loss with respect to that operation's inputs (chain rule). Gradients accumulate in the .grad attribute of leaf tensors.

Key details: The graph is freed after .backward() by default (use retain_graph=True to keep it). Non-leaf tensors don't keep their gradients unless you call .retain_grad(). This is why activations from the forward pass must be cached by custom Function objects via ctx.save_for_backward().

AutogradCore
Q03 Why do gradients accumulate in PyTorch, and when is this actually useful? ›

Gradients accumulate (add to .grad) because PyTorch doesn't automatically zero them. This is a deliberate design choice — it enables gradient accumulation, a crucial technique.

Gradient accumulation use case: If you can't fit a large batch on GPU, run N smaller forward+backward passes without calling optimizer.step(), then step once. This simulates a batch N× larger:

accum_steps = 8  # simulate 8× batch size
for i, (x, y) in enumerate(loader):
    loss = criterion(model(x), y) / accum_steps
    loss.backward()   # gradients accumulate
    if (i+1) % accum_steps == 0:
        optimizer.step()
        optimizer.zero_grad()

This allows training GPT-scale models with limited VRAM. Without this behavior, the workaround would require explicit gradient summing.

TrainingMemory
Q04 What is the difference between .view() and .reshape()? When does .view() fail? ›

.view(): Returns a tensor with a new shape that shares memory with the original (zero-copy). Fails with RuntimeError if the tensor is not contiguous in memory (e.g., after transpose or permute).

.reshape(): Returns a new shape, using a view if possible, otherwise copying memory to create a contiguous tensor. Never fails on shape-compatible tensors.

Performance tip: Transposing makes tensors non-contiguous. If you need to reshape after transpose, call .contiguous().view() explicitly to signal the copy is intentional, or just use .reshape(). Hidden copies in tight loops can cause significant slowdowns.

t = torch.randn(4, 6).transpose(0, 1)
# t.view(-1)  → RuntimeError! not contiguous
t.contiguous().view(-1)  # ✓
t.reshape(-1)           # ✓ (same result)
Tensors
Q05 What does model.train() and model.eval() actually do? ›

These methods control the behavior of layers that differ between training and inference:

Layers affected by model.train():

• Dropout: In train mode, randomly zeroes neurons with probability p. In eval mode, passes all activations through (scaled by 1/(1-p) during training to maintain expected activation magnitude)
• BatchNormalization: In train mode, normalizes using the current batch's statistics and updates running mean/variance. In eval mode, uses the accumulated running statistics — making inference deterministic and independent of batch size
• Other layers (Linear, Conv, etc.) are unaffected

Critical bug pattern: Forgetting model.eval() during validation/inference causes batch statistics to contaminate results and Dropout to randomly zero outputs — leading to lower and noisy metrics. Always pair model.eval() with torch.no_grad().

TrainingAPI
Q06 Explain DataParallel vs DistributedDataParallel — why prefer DDP? ›

DataParallel (DP): Single Python process. Splits batches across GPUs, runs forward pass in parallel, gathers outputs on GPU 0, computes loss on GPU 0, scatters gradients back. Limited by Python GIL on backward pass. One GPU (rank 0) is heavily overloaded. Not suitable for NLP (variable-length sequences hard to split).

DistributedDataParallel (DDP): One Python process per GPU. All processes are symmetric — no "master" GPU. Gradients are reduced across all processes via NCCL all-reduce after backward. No GIL contention. Scales near-linearly. Works across multiple machines. Is the industry standard.

Speed difference: On 4× V100 GPUs, DDP is typically 2–4× faster than DP due to eliminated GIL bottleneck and better load balancing.

Distributed
Q07 What is mixed precision training and how does GradScaler prevent underflow? ›

Mixed precision uses float16 (FP16) for computations where numerical precision is sufficient (most of the forward and backward pass) and float32 for numerically sensitive parts (weight updates, loss accumulation). Benefits: 2–3× throughput on Tensor Core GPUs, 2× memory savings.

The underflow problem: FP16 has a minimum representable non-zero value of ~6×10⁻⁸. Many gradients in deep networks are smaller than this, causing them to round to zero — model stops learning.

GradScaler solution: Before the backward pass, multiply the loss by a large scale factor S (e.g., 65536). Gradients are also scaled by S, pushing small gradients above FP16's minimum threshold. After the backward pass, unscale gradients before the optimizer step. GradScaler dynamically adjusts S: increases it if no inf/nan is detected, decreases it otherwise.

PerformanceTraining
Q08 When would you use TorchScript vs torch.compile vs ONNX for deployment? ›

TorchScript (trace/script): Serializes model to a portable format that runs without Python. Use when: deploying to C++ applications, mobile (TorchMobile), or environments without Python. Tracing works for fixed-topology models; scripting supports control flow. Mature and stable.

torch.compile (PyTorch 2.0+): JIT compiles model for speed — stays in Python, no export needed. Use when: maximizing training/inference throughput in Python environments. Easiest to use (one line), 30–100% speedup. Not portable outside Python.

ONNX: Open format for cross-framework deployment. Use when: deploying to TensorRT, OpenVINO, ONNX Runtime, or non-PyTorch serving infrastructure. Enables aggressive hardware-specific optimization. More complex to maintain (opset versions, custom ops).

Deployment
Q09 What is the purpose of pin_memory=True and num_workers in DataLoader? ›

num_workers: Number of subprocess workers for parallel data loading. With num_workers=0, data loading happens in the main process, blocking GPU training. With num_workers=4, 4 parallel processes prefetch the next batch while the GPU trains on the current one. Rule of thumb: 4× the number of GPUs.

pin_memory=True: Allocates loaded tensors in pinned (page-locked) CPU memory rather than pageable memory. Pinned memory enables asynchronous, DMA-based transfers to GPU, which are significantly faster than pageable memory transfers. Essential for high-throughput training. Only use with CUDA — has no effect on CPU-only training and wastes memory.

Optimal config: DataLoader(ds, num_workers=8, pin_memory=True, persistent_workers=True). persistent_workers=True keeps workers alive between epochs to avoid the startup overhead.

Q10 How do you implement a custom loss function in PyTorch? ›

Three approaches, in order of complexity:

# Method 1: Plain function (simplest — autograd handles it)
def focal_loss(logits, targets, gamma=2.0, alpha=0.25):
    probs = torch.sigmoid(logits)
    bce   = F.binary_cross_entropy_with_logits(logits, targets, reduction='none')
    pt    = torch.where(targets == 1, probs, 1 - probs)
    return (alpha * (1 - pt) ** gamma * bce).mean()

# Method 2: nn.Module (if you need learnable parameters)
class ContrastiveLoss(nn.Module):
    def __init__(self, margin=1.0):
        super().__init__()
        self.margin = margin
    def forward(self, d, y):   # d=distance, y=is_same
        pos = y * d.pow(2)
        neg = (1 - y) * F.relu(self.margin - d).pow(2)
        return (pos + neg).mean()

Q11 What is weight initialization and why does it matter? ›

Weight initialization sets the starting values of model parameters before training. Initialization affects whether gradients flow well in the first few steps, and can determine whether training converges at all.

Problems with wrong initialization:

• Too large: Activations explode → saturated activations → zero gradients (dying sigmoid)
• Too small: Activations shrink layer by layer → vanishing gradients
• All zeros: Every neuron computes the same gradient → no differentiation between neurons (symmetry breaking problem)

Standard initializations in PyTorch:

• Glorot/Xavier: nn.init.xavier_uniform_(w) — scales by √(2/(fan_in+fan_out)). Best for sigmoid/tanh
• He/Kaiming: nn.init.kaiming_normal_(w, nonlinearity='relu') — scales by √(2/fan_in). Best for ReLU
• Normal(0, 0.02): Common for GANs

Training
Q12 How does BatchNorm differ from LayerNorm, and when do you use each? ›

BatchNorm (nn.BatchNorm2d/1d): Normalizes across the batch dimension. Statistics computed over all samples in the batch for each feature/channel. Depends on batch size — fails with small batches (B=1 is undefined), behaves differently at train/eval time. Excellent for CNNs with large batches.

LayerNorm (nn.LayerNorm): Normalizes across the feature dimension for each individual sample — batch-independent. Same behavior at train and eval time. Works with any batch size (including B=1). Standard for transformers, RNNs, and any sequence model.

GroupNorm: Divides channels into groups, normalizes within each group. Best for object detection/segmentation where batch sizes are small (B=1-2).

Rule: CNN with large batch → BatchNorm. Transformer/RNN/small batch → LayerNorm. Detection/segmentation → GroupNorm.

Q13 What is torch.no_grad() and when would you NOT use it even during inference? ›

torch.no_grad() disables gradient tracking for all operations within its scope. The computation graph is not built, saving memory and computation (~20–40% faster inference).

Always use during: validation loops, inference/prediction, evaluation metrics, feature extraction when you only need the forward pass output.

Don't use when:

• Adversarial attacks (FGSM, PGD): You need gradients with respect to the input, not parameters
• Neural style transfer / DeepDream: Optimizing the input image
• Meta-learning (MAML): Computing gradients of gradients
• Saliency maps / GradCAM: Need input gradients for interpretability
• Test-time adaptation: Updating certain statistics during inference

Q14 How do you debug a NaN loss in PyTorch? ›

Systematic NaN debugging checklist:

1. Check inputs first: assert not torch.isnan(x).any() — NaN often comes from the data (missing values, log(0), division by zero)

2. Check loss function: log(0) = -inf → NaN. Use BCEWithLogitsLoss instead of BCELoss(sigmoid(x)) for numerical stability. Use CrossEntropyLoss with raw logits, not softmax outputs

3. Check learning rate: Too high → loss explodes → NaN. Reduce by 10×

4. Add gradient hooks to locate first NaN gradient layer (see STEP 15)

5. Use anomaly detection: torch.autograd.set_detect_anomaly(True) — raises an error with a full traceback at the exact operation that produces NaN

6. Check weight initialization: Overly large initial weights saturate activations

7. Add gradient clipping: torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)

Q15 What is the difference between contiguous and non-contiguous tensors in memory? ›

A contiguous tensor stores elements sequentially in memory in row-major (C) order — the last dimension varies fastest. PyTorch can directly pass contiguous tensors to kernels using a simple pointer and stride calculation.

Operations like transpose, permute, and slicing with non-unit steps create non-contiguous tensors — they return a view with modified strides, but the underlying memory layout is unchanged. For example, t.T doesn't move data; it just swaps the strides.

Why it matters: Many low-level kernels (CUBLAS, cuDNN) require contiguous inputs. PyTorch automatically calls .contiguous() when needed, but this triggers a hidden memory copy — in a tight training loop this wastes time and memory.

Check and fix: t.is_contiguous() returns True/False. Fix with t.contiguous(). In models, place x = x.contiguous() after permute/transpose calls that feed into view().
