Beginner · Lesson 01

Why RAG?

// the problem RAG solves

🟢 BEGINNER — No prior ML knowledge needed to start here
📖 THEORY

Large Language Models (LLMs) have two critical limitations: a knowledge cutoff date and hallucination — generating plausible-sounding but incorrect facts. RAG (Retrieval-Augmented Generation) solves both by grounding the LLM's responses in retrieved, factual documents at query time.

  • Without RAG: LLM answers from parametric memory → stale, hallucinated
  • With RAG: LLM answers from retrieved context → current, grounded, citable
  • First proposed by Lewis et al., 2020 (Facebook AI Research)

❌ Without RAG

LLM generates answers from weights baked in during training. Cannot access new information, company-specific data, or real-time knowledge. May confabulate facts confidently.

✅ With RAG

LLM receives relevant document chunks at runtime. Answers are grounded in retrieved evidence. You can update the knowledge base without retraining the model.

The Basic RAG Pipeline

📖 THEORY

basic_rag.py 
# Minimal RAG pipeline using LangChain + OpenAI
from langchain.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.llms import OpenAI
from langchain.chains import RetrievalQA

# ① INDEXING: Load → Chunk → Embed → Store
loader = TextLoader("company_docs.txt")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(docs)

vectorstore = Chroma.from_documents(
    chunks,
    OpenAIEmbeddings()
)

# ② QUERYING: Retrieve → Augment → Generate
qa_chain = RetrievalQA.from_chain_type(
    llm=OpenAI(),
    retriever=vectorstore.as_retriever(
        search_kwargs={"k": 4}
    )
)

answer = qa_chain.run("What is our refund policy?")
print(answer)

Document Chunking

// splitting documents into retrievable pieces

📖 THEORY

Chunking splits large documents into smaller pieces that fit within the LLM's context window and can be individually retrieved. The chunk strategy dramatically affects RAG quality:

  • Too small chunks: lose context, fragments sentences mid-thought
  • Too large chunks: exceed context limits, reduce retrieval precision
  • Overlap: a sliding window overlap (e.g. 50-100 tokens) prevents losing context at chunk boundaries
  • Common strategies: Fixed-size, Recursive, Semantic, Sentence, Markdown-aware

Embeddings

// converting text into dense vectors

📖 THEORY

An embedding is a dense numerical vector representing the semantic meaning of text. Semantically similar texts have vectors that are close together in high-dimensional space. Similarity is measured using cosine similarity or dot product.

  • Popular models: text-embedding-3-small (OpenAI), all-MiniLM-L6-v2 (SBERT), BGE-M3
  • Dimensions: typically 384–3072 floating point numbers per chunk
  • Critical: use the SAME embedding model for indexing AND querying

embeddings.py 
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("all-MiniLM-L6-v2")

# Embed chunks during indexing
chunks = ["RAG retrieves relevant docs...",
          "Embeddings encode semantic meaning..."]
chunk_embeddings = model.encode(chunks)
# shape: (2, 384) — 384-dim vectors

# Embed query at inference time
query = "How does RAG work?"
query_embedding = model.encode([query])

# Cosine similarity
def cosine_sim(a, b):
    return np.dot(a, b) / (
        np.linalg.norm(a) * np.linalg.norm(b)
    )

scores = [cosine_sim(query_embedding[0], c)
          for c in chunk_embeddings]
# → [0.87, 0.72]
Beginner · Lesson 05

Vector Databases

// storing and searching embeddings at scale

📖 THEORY

A vector database stores embeddings and enables fast Approximate Nearest Neighbour (ANN) search. Brute-force cosine similarity over millions of vectors is too slow — ANN algorithms (HNSW, IVF) trade a tiny accuracy hit for massive speed gains.

  • HNSW (Hierarchical Navigable Small World) — graph-based, very fast
  • IVF (Inverted File Index) — cluster-based, memory efficient
  • Popular DBs: Pinecone, Weaviate, Qdrant, Chroma, pgvector, FAISS

# Example: Qdrant vector database

    Distance, VectorParams, PointStruct
)

client = QdrantClient(":memory:")  # or host=

# Create collection

    collection_name="knowledge_base",
    vectors_config=VectorParams(
        size=384,          # embedding dimensions
        distance=Distance.COSINE
    )

# Upsert vectors with metadata
client.upsert(
    collection_name="knowledge_base",
    points=[
        PointStruct(
            id=1,
            vector=chunk_embeddings[0].tolist(),

                "text": chunks[0],
                "source": "docs/intro.md",
                "page": 1

        )
    ]
)

# Search top-4 similar chunks
results = client.search(
    collection_name="knowledge_base",
    query_vector=query_embedding[0].tolist(),
    limit=4
)
for r in results:
    print(r.score, r.payload["text"])

Basic Retrieval

// finding the most relevant chunks for a query

📖 THEORY

Retrieval finds the top-K chunks most semantically similar to the user's query. Key parameter: k — how many chunks to retrieve. Too few = missing information. Too many = exceeds context window, adds noise. Typical range: k = 3–10.

generation.py 
def generate_answer(query, retrieved_chunks, llm_client):
    # Build context from retrieved chunks
    context = "\n\n".join([
        f"[Source: {c['source']}]\n{c['text']}"
        for c in retrieved_chunks
    ])

    # Prompt template
    prompt = f"""You are a helpful assistant.
Answer ONLY using the context below.
If not found, say "I don't have that information."

Context:
{context}

Question: {query}
Answer:"""

    response = llm_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0.1  # low temp for factual answers
    )

    return response.choices[0].message.content
Intermediate · Lesson 08

Reranking

// two-stage retrieval for higher precision

🔵 INTERMEDIATE — Building better pipelines with improved retrieval quality
📖 THEORY

Embedding-based retrieval is fast but not always accurate — it operates on compressed representations. A reranker (cross-encoder) takes each query-chunk pair and computes a precise relevance score. The pipeline becomes: retrieve large pool (e.g. top-50) → rerank → keep top-K (e.g. top-5).

  • Bi-encoder (fast): embed query and chunks independently, compare vectors
  • Cross-encoder (accurate): process query+chunk together, no information loss
  • Popular rerankers: bge-reranker, Cohere Rerank, Jina Reranker

reranking.py 
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query, candidates, top_k=5):
    # Pair query with each candidate chunk
    pairs = [(query, c["text"]) for c in candidates]
    
    # Cross-encoder scores each pair jointly
    scores = reranker.predict(pairs)
    
    # Sort by reranker score and return top_k
    ranked = sorted(
        zip(candidates, scores),
        key=lambda x: x[1],
        reverse=True
    )
    
    return [c for c, _ in ranked[:top_k]]

# Two-stage pipeline
candidates = vectorstore.similarity_search(query, k=30)
top_chunks = rerank(query, candidates, top_k=5)

Hybrid Search

// combining dense + sparse retrieval

📖 THEORY

Semantic (dense) search excels at understanding intent but misses exact keyword matches. BM25 (sparse) excels at exact term matching but misses semantics. Hybrid search combines both using Reciprocal Rank Fusion (RRF):

  • Dense (vector): finds semantically related chunks — "car" matches "automobile"
  • Sparse (BM25/TF-IDF): finds exact keyword matches — "GPT-4 API" matches precisely
  • RRF formula: score = Σ 1/(k + rank_i) where k=60 is a smoothing constant
hybrid_search.py 
from rank_bm25 import BM25Okapi

def reciprocal_rank_fusion(rankings, k=60):
    scores = {}
    for ranking in rankings:
        for rank, doc_id in enumerate(ranking):
            if doc_id not in scores:
                scores[doc_id] = 0
            scores[doc_id] += 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

# Dense retrieval (semantic)
dense_results = vectorstore.similarity_search(query, k=20)
dense_ids = [r.metadata["id"] for r in dense_results]

# Sparse retrieval (BM25 keyword)
bm25 = BM25Okapi([c.split() for c in corpus])
bm25_scores = bm25.get_scores(query.split())
sparse_ids = list(np.argsort(bm25_scores)[::-1][:20])

# Fuse rankings
fused = reciprocal_rank_fusion([dense_ids, sparse_ids])

Metadata Filtering

// filtering retrieval by structured attributes

📖 THEORY

Storing metadata alongside chunks (source file, date, category, author) enables pre-filter or post-filter retrieval. This dramatically improves precision for multi-tenant or multi-domain knowledge bases:

  • pre-filter: filter vector DB before ANN search (faster, supported by most DBs)
  • post-filter: retrieve top-K then filter by metadata (simpler, may leave few results)
  • Use case: "What are Q4 2023 sales numbers?" → filter year=2023, quarter=Q4

Intermediate · Lesson 11

Prompt Engineering for RAG

// crafting prompts that reduce hallucination

📖 THEORY

Prompt design is critical for RAG quality. Key patterns:

  • System grounding: instruct the LLM to answer ONLY from context
  • Citation instruction: ask it to cite source documents inline as [1], [2]
  • Graceful fallback: "Say 'I don't know' if not in context" reduces hallucination
  • Compression: for long contexts, compress each chunk with a summarizer before injection
  • Chain-of-thought: ask LLM to reason step by step before answering
rag_prompt_template.py 
SYSTEM_PROMPT = """You are a precise assistant.
Rules:
1. ONLY answer from the provided context.
2. Cite sources inline: [Doc 1], [Doc 2], etc.
3. If the answer is not in the context, respond:
   "I don't have information about that."
4. Be concise and factual. No speculation."""

def build_prompt(query, chunks):
    context_parts = []
    for i, chunk in enumerate(chunks, 1):
        context_parts.append(
            f"[Doc {i}] (Source: {chunk['source']})\n"
            f"{chunk['text']}"
        )
    context = "\n\n".join(context_parts)

    return [
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content":
            f"Context:\n{context}\n\nQuestion: {query}"}
    ]

HyDE — Hypothetical Document Embeddings

// query expansion for better recall

🟣 ADVANCED — Cutting-edge techniques for production RAG systems
📖 THEORY

HyDE (Gao et al., 2022) improves retrieval for short or ambiguous queries. Instead of embedding the raw question, it asks the LLM to generate a hypothetical answer, then embeds that answer for retrieval. Hypothetical answers exist in the same semantic space as real document chunks.

  • Step 1: Generate a hypothetical answer to the query (may contain errors — that's OK)
  • Step 2: Embed the hypothetical answer (not the query)
  • Step 3: Use the hypothetical embedding to search the vector DB
  • Useful when queries are too short to carry sufficient semantic signal

def hyde_retrieve(query, vectorstore, llm, k=5):
    # Step 1: Generate hypothetical answer
    hypo_prompt = f"""Write a short passage that would 
    answer this question if it appeared in a document:
    
    Question: {query}
    
    Write only the passage, no preamble:"""
    
    hypothetical_doc = llm.complete(hypo_prompt)
    
    # Step 2: Embed the hypothetical answer
    hypo_embedding = embedding_model.encode(
        [hypothetical_doc]
    )[0]
    
    # Step 3: Search using hypothetical embedding
    results = vectorstore.similarity_search_by_vector(
        hypo_embedding, k=k
    )
    
    return results

GraphRAG

// knowledge graphs + vector retrieval

📖 THEORY

GraphRAG (Microsoft, 2024) builds a knowledge graph from documents — extracting entities and relationships — then uses graph traversal alongside vector search for retrieval. Answers multi-hop questions that require combining information from multiple disconnected documents.

  • Entity extraction: LLM identifies entities (people, orgs, concepts) and relations
  • Community detection: clusters related entities into communities
  • Global search: queries spanning the entire corpus use graph summaries
  • Local search: specific queries use entity-centred graph traversal

│ ↓ requires
LLM ─generates→ Answer
graphrag_extract.py 
# Entity + relationship extraction for GraphRAG
EXTRACT_PROMPT = """Extract entities and relationships.
Format: ENTITY|type|description
        RELATION|entity1|relation|entity2

Text: {text}"""

def extract_graph(chunk_text, llm):
    result = llm.complete(EXTRACT_PROMPT.format(
        text=chunk_text
    ))
    
    entities, relations = [], []
    for line in result.split("\n"):
        parts = line.split("|")
        if parts[0] == "ENTITY":
            entities.append({
                "name": parts[1],
                "type": parts[2]
            })
        elif parts[0] == "RELATION":
            relations.append({
                "source": parts[1],
                "relation": parts[2],
                "target": parts[3]
            })
    return entities, relations

Agentic RAG

// LLM decides what to retrieve and when

📖 THEORY

In Agentic RAG, the LLM acts as an agent that decides when and what to retrieve, rather than always retrieving on every query. It can issue multiple retrieval calls, decompose complex questions, reflect on retrieved results, and iteratively refine its answer.

  • Corrective RAG (CRAG): evaluates retrieved docs — refetches if quality is low
  • Self-RAG: LLM generates special tokens to decide if retrieval is needed
  • ReAct: Reason + Act loop — agent reasons about next retrieval step
  • Multi-hop: answer complex questions requiring chained retrieval steps

RAG Evaluation

// measuring retrieval and generation quality

📖 THEORY

RAG systems need separate evaluation of both components. The RAGAS framework (Es et al., 2023) is the standard benchmark. Key metrics:

  • Faithfulness: is the answer factually grounded in the retrieved context? (0–1)
  • Answer Relevancy: does the answer actually answer the question? (0–1)
  • Context Recall: did retrieval find all necessary information? (0–1)
  • Context Precision: are the retrieved chunks all relevant? (0–1)
  • Context Relevancy: fraction of retrieved context that is relevant

Interview Prep · 25 Questions

Interview Questions

// beginner to advanced · click to expand answers

🔶 INTERVIEW — Common RAG questions asked in ML / AI Engineer roles
1
What is RAG and what problem does it solve? easy
▼

RAG (Retrieval-Augmented Generation) is a framework that enhances LLM responses by retrieving relevant external documents at query time and injecting them into the prompt context.

It solves two core LLM limitations:

  • Knowledge cutoff: LLMs are trained on static data and don't know recent events
  • Hallucination: LLMs generate plausible-sounding but incorrect facts from parametric memory

With RAG, the LLM answers are grounded in retrieved documents, making them verifiable and updatable without retraining.

2
Explain the two phases of a RAG pipeline. easy
▼

Phase 1 — Indexing (offline): Documents are loaded, chunked into smaller pieces, converted to embeddings via an encoder model, and stored in a vector database.

Phase 2 — Querying (runtime): The user's query is embedded, the vector DB performs ANN search to find top-K similar chunks, those chunks are injected into an LLM prompt alongside the query, and the LLM generates a grounded answer.

3
What is an embedding and how is similarity computed? easy
▼

An embedding is a dense numerical vector (e.g. 384 or 1536 floats) that encodes the semantic meaning of text. Semantically similar texts have vectors that are close in high-dimensional space.

Cosine similarity is the standard metric: sim(a,b) = (a·b) / (|a| × |b|). Range is -1 to 1 where 1 = identical meaning. Dot product is also used when vectors are normalized (equivalent to cosine similarity).

4
What chunking strategy would you use for a legal document? medium
▼

For legal documents I would use hierarchical semantic chunking:

  • First split on structure: sections, clauses, sub-clauses (using heading/numbering patterns)
  • For oversized sections, apply RecursiveCharacterTextSplitter with a conservative chunk size (~300-400 tokens) and overlap (~50 tokens)
  • Store the parent section/clause as metadata with each chunk for hierarchical retrieval
  • Consider parent-child chunking: small chunks for retrieval precision, return larger parent chunk to the LLM for context
5
What is the difference between a bi-encoder and a cross-encoder? medium
▼
  • Bi-encoder: encodes query and document independently, compares via cosine similarity. Fast — can pre-compute all document embeddings. Used for initial retrieval (ANN search over millions of docs).
  • Cross-encoder: processes query + document together in a single forward pass. Much more accurate — full attention between query and document tokens. Slow — cannot pre-compute. Used for reranking top-K candidates.

Typical pipeline: bi-encoder retrieves top-50 → cross-encoder reranks → LLM uses top-5.

6
What is HNSW and why is it used in vector databases? medium
▼

HNSW (Hierarchical Navigable Small World) is a graph-based algorithm for Approximate Nearest Neighbour search. It builds a multi-layer graph where higher layers have fewer, long-range connections and lower layers have more, fine-grained connections.

During search, HNSW starts at the top layer (wide jumps) and greedily navigates toward the query, descending through layers for increasing precision. This achieves near-logarithmic search time, making it practical for millions of vectors.

7
How would you handle a RAG system for a multi-tenant SaaS product? medium
▼
  • Namespace isolation: Pinecone namespaces or Qdrant collection-per-tenant for hard isolation
  • Metadata filtering: store tenant_id as metadata, apply mandatory filter on every query
  • Shared index with filters is more cost-efficient for many small tenants; separate collections better for large tenants with privacy requirements
  • Ensure no cross-tenant data leakage — validate filters server-side, never trust client-sent filters
8
What is HyDE and when would you use it? medium
▼

HyDE (Hypothetical Document Embeddings): generate a hypothetical answer to the query with the LLM, then embed that answer (not the original question) for retrieval. The hypothesis exists in the same embedding space as real document chunks.

Use it when:

  • Queries are very short and lack semantic richness (e.g. "HNSW?")
  • Query phrasing differs significantly from how documents are written
  • Initial retrieval recall is low despite good document coverage

Downside: adds one extra LLM call, which increases latency and cost.

9
Explain Reciprocal Rank Fusion (RRF). medium
▼

RRF merges multiple ranked lists (e.g. from dense + sparse retrieval) without needing score normalization. Formula:

RRF_score(d) = Σ 1 / (k + rank_i(d))

where k=60 is a smoothing constant and rank_i is the document's rank in list i. A document ranked #1 in dense and #3 in sparse will have a high combined score. k=60 prevents very high-ranked documents from dominating too much.

10
How do you prevent hallucination in RAG? medium
▼
  • Grounding instruction: explicitly tell the LLM to answer ONLY from context
  • Graceful abstention: instruct "say 'I don't know' if not in context"
  • Low temperature: use 0.0–0.1 for factual RAG tasks
  • Citation requirement: ask LLM to cite [Doc N] for each claim — forces it to attribute information
  • Faithfulness eval: use RAGAS faithfulness metric to detect hallucinated claims post-hoc
  • NLI verification: run a Natural Language Inference model to verify each claim is entailed by the retrieved context
11
What are the RAGAS evaluation metrics and how are they computed? hard
▼
  • Faithfulness: LLM decomposes answer into claims, checks each claim is supported by context. Score = claims_supported / total_claims
  • Answer Relevancy: LLM generates N questions from the answer, embeds them, measures cosine sim to original question. High sim = answer is on-topic
  • Context Recall: compares ground-truth answer sentences to retrieved context using LLM-based entailment. Fraction of GT sentences attributable to context
  • Context Precision: for each retrieved chunk, LLM judges if it's relevant to the question. Precision@k averaged across positions

How does GraphRAG differ from naive RAG? hard
▼

Naive RAG retrieves flat, disconnected chunks. GraphRAG builds a knowledge graph of entities and their relationships across the corpus.

Key advantages:

  • Answers multi-hop questions that require connecting information from separate documents
  • Global queries (e.g. "What are the main themes in all these documents?") — uses graph community summaries, impossible for naive RAG
  • Entity-level understanding — knows that "OpenAI" and "the company behind GPT-4" refer to the same entity

Tradeoffs: significantly higher indexing cost (many LLM calls), complex to build and maintain.

What is the "lost in the middle" problem in RAG? hard
▼

Research (Liu et al., 2023) shows that LLMs perform best when relevant information appears at the beginning or end of the context window, and worst when it's in the middle.

Mitigations:

  • Place most relevant chunks at the start and end of context (not the middle)
  • Reduce the number of chunks (k) so less critical content sits in middle positions
  • Use reranking to ensure the highest-relevance chunk is always placed first
  • Use models with better long-context handling (e.g. Gemini 1.5 with 1M tokens)

How would you implement Self-RAG? hard
▼

Self-RAG (Asai et al., 2023) fine-tunes the LLM to generate special reflection tokens:

  • [Retrieve]: LLM decides retrieval is needed for this segment
  • [IsRel]: rates if retrieved docs are relevant
  • [IsSup]: rates if generated text is supported by retrieved docs
  • [IsUse]: overall usefulness score

The model only triggers retrieval when needed (not always), and grades its own outputs. This reduces unnecessary retrieval overhead for simple questions while improving quality for complex ones. Requires fine-tuning — not achievable with prompt engineering alone on base models.

What is parent-child chunking? medium
▼

Parent-child chunking uses two granularities:

  • Child chunks (small, ~100 tokens): stored in vector DB, used for precise retrieval
  • Parent chunks (large, ~500+ tokens): stored separately, returned to LLM

Retrieval logic: match on child chunks → return their parent chunks to the LLM. This gives retrieval precision of small chunks with the rich context of large chunks. Implemented in LangChain as ParentDocumentRetriever.

How do you handle PDFs and tables in RAG? medium
▼
  • PDFs: use pypdf for text, pdfplumber for tables, unstructured for complex layouts. OCR with tesseract for scanned PDFs
  • Tables: convert to markdown format before chunking — LLMs understand markdown tables well
  • Multimodal: for PDFs with charts/images, use GPT-4V or Claude to generate text descriptions of visuals
  • Always preserve table structure — never split a table across chunks

What is query decomposition in RAG? medium
▼

Query decomposition breaks a complex, multi-part question into simpler sub-queries that can each be independently retrieved and answered.

Example: "Compare the revenue growth of Apple and Microsoft in 2023" → sub-queries: (1) "Apple revenue 2023", (2) "Microsoft revenue 2023", (3) "Apple revenue 2022", (4) "Microsoft revenue 2022". Results are retrieved separately then synthesized.

Implementations: LangChain's MultiQueryRetriever, LlamaIndex's SubQuestionQueryEngine.

It depends on the use case:

  • OpenAI text-embedding-3-small: great balance of cost/quality, 1536 dims, easy to use with OpenAI ecosystem
  • BGE-M3: open-source, supports 100+ languages, multi-granularity (dense + sparse + multi-vector), best for multilingual or on-premise

How would you reduce RAG latency in production? hard
▼
  • Semantic caching: cache (query_embedding → answer) pairs. On new query, check if a similar embedding exists in cache before hitting vector DB + LLM

What is the difference between RAG and fine-tuning? When to use which? medium
▼

Fine-tuning bakes knowledge/style into model weights. RAG provides knowledge at inference time via context injection.

  • Use RAG when: knowledge changes frequently, sources need to be cited, domain data is large, you need traceability
  • Use Fine-tuning when: you need a specific response style/format, task behaviour improvement, or edge-case handling
  • Best practice: use both — fine-tune for style and format, RAG for up-to-date knowledge

What is Corrective RAG (CRAG)? hard
▼

CRAG (Yan et al., 2024) adds a relevance evaluator to the RAG pipeline that grades retrieved documents:

  • Correct (high relevance): use chunks directly
  • Incorrect (low relevance): discard and trigger web search (e.g. Tavily) as fallback
  • Ambiguous (medium): use a combination of retrieved chunks + web search

The evaluator uses a lightweight LLM or fine-tuned classifier. This prevents bad retrievals from polluting the generation step — a key failure mode in naive RAG.

How do you handle conversational/multi-turn RAG? medium
▼

Challenge: follow-up questions like "What about its pricing?" are ambiguous without prior context.

  • Query contextualization: use LLM to rewrite the follow-up question using chat history into a standalone query before retrieval
  • Chat history injection: include last N turns in the prompt alongside retrieved context
  • Memory compression: summarize older history to avoid exceeding context limits
  • LangChain provides ConversationalRetrievalChain with built-in history handling

What is vector quantization and why does it matter for RAG at scale? hard
▼

Storing full float32 vectors is expensive at scale. Vector quantization compresses vectors:

  • Scalar quantization (SQ): maps each float32 → int8, 4× memory reduction with ~1% accuracy loss
  • Product quantization (PQ): splits vector into sub-vectors, each compressed via codebook lookup. 8–64× compression
  • Binary quantization: each component → 1 bit (sign). 32× compression, fast with Hamming distance

For 10M chunks with 1536-dim vectors: full float32 = ~60GB. SQ int8 = ~15GB. Critical for cost and memory in production.

What are common RAG failure modes and how do you debug them? medium
▼
  • Retrieval failure: wrong chunks retrieved → check embeddings, try hybrid search, improve chunking
  • Hallucination despite good retrieval: LLM ignores context → stronger system prompt, lower temperature, add citation requirement
  • Context overflow: too many chunks → reduce k, use summarization, parent-child chunking
  • Chunk fragmentation: answers span multiple chunks not retrieved together → increase overlap, use semantic chunking
  • Debug tool: log retrieved chunks for every query — review if relevant documents are being found
