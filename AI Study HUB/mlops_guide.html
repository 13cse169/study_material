MODULE 01 // FOUNDATIONS
What is MLOps?
Machine Learning + DevOps + Data Engineering = MLOps
Definition

MLOps (Machine Learning Operations) is a set of practices that combines ML, DevOps, and Data Engineering to reliably and efficiently build, deploy, monitor, and maintain ML systems in production. It aims to shorten the development cycle, reduce failure, and automate the end-to-end ML lifecycle — from data ingestion to model retirement.

The core problem MLOps solves: only 22% of ML models ever make it to production, and of those, most degrade quickly without proper monitoring. MLOps builds the engineering discipline to close this gap.

🌍 Real-World Example

Netflix uses MLOps to manage 1,000+ ML models powering recommendations, search ranking, and content quality scoring. Their MLOps platform (Metaflow) enables data scientists to run experiments with a few lines of Python while the infrastructure handles distributed training, versioning, and deployment automatically — reducing time-to-production from months to days.

The Three Pillars

🔬

ML Engineering

Feature engineering, model training, evaluation, hyperparameter optimization. The experimental science side of the lifecycle.

DevOps Practices

CI/CD pipelines, version control, testing, containerization, infrastructure-as-code. Automation and reliability engineering applied to ML.

Engineering
📊

Data pipelines, feature stores, data quality monitoring, lineage tracking. The foundation every model is built on.

Data & Model Versioning
Reproducibility as a first-class requirement

In software, git solves reproducibility. In ML, you also need to version data (which can be terabytes), models (binary artifacts), pipelines (DAG definitions), and environments (conda/pip specs). Without all four, experiments are not reproducible.

Data Versioning with DVC

# DVC: Git for data — tracks large files via pointers
dvc init
dvc add data/train.csv          # creates data/train.csv.dvc (tracked by git)
git add data/train.csv.dvc .gitignore
git commit -m "Add training dataset v1"
dvc push                        # pushes actual data to S3/GCS/Azure

# Reproduce exact experiment
git checkout experiment-v3
dvc checkout                    # restores the exact data for that commit
dvc repro                       # reruns the full pipeline with that data

MODULE 09 // MONITORING & DRIFT
Model Monitoring & Drift Detection
Models decay — here's how to catch it before users do
Model Drift

Model drift occurs when model performance degrades in production because the real-world data distribution has shifted from the training distribution. Unlike software bugs, drift is gradual, silent, and dangerous. A fraud model trained in 2023 may fail on 2025 fraud patterns.

Types of Drift

📉

Data Drift (Covariate Shift)

Input feature distribution changes. P(X_prod) ≠ P(X_train). Example: a model trained on summer data deployed in winter — temperature features now have completely different distributions.

Most Common
🔄

Concept Drift

The relationship between X and Y changes. P(Y|X) changes. Example: sentiment model trained before COVID on restaurant reviews — "outdoor seating" was positive. Post-COVID meaning shifted.

Hardest to Detect
🎯

Label Drift

Distribution of targets changes. P(Y) changes. Example: fraud detection — fraud rate spikes from 1% to 3% due to new attack pattern. Model never sees these new patterns.

Upstream Cause
📊

Prediction Drift

Distribution of model outputs changes. Often the first observable signal — can be detected without ground truth labels, unlike accuracy-based metrics.

Early Warning

Drift Detection Methods

What to Monitor

📊

Feature Statistics

Mean, std, min/max, null rate, value distributions for each feature. Alert when any statistic deviates >3σ from training baseline.

🎯

Prediction Distribution

Score distribution, class probabilities. A fraud model's average score shifting from 0.12 to 0.28 is a red flag even before you see accuracy degradation.

⚡

System Performance

Latency p50/p95/p99, throughput (RPS), error rate (5xx), memory/GPU utilization. Standard SRE monitoring applies to ML serving.

✅

Business Metrics

Conversion rate, click-through, fraud loss rate, customer satisfaction. The ultimate ground truth — when business metrics degrade, the model is failing.

          python train.py --config config.yaml
          # Outputs: model.pkl, metrics.json

      - name: Quality Gate
        run: |
          ACCURACY=$(cat metrics.json | jq '.accuracy')
          if (( $(echo "$ACCURACY < 0.92" | bc -l) )); then
            echo "Accuracy $ACCURACY below threshold 0.92 — blocking deploy"

      - name: Register Model
        if: github.ref == 'refs/heads/main'
        run: python register_model.py --stage Staging

      - name: Deploy Canary
        run: kubectl set image deployment/fraud-api model=registry/fraud:$GITHUB_SHA

MODULE 12 // FEATURE STORES
Feature Stores
Eliminating train/serve skew — the silent killer
Training/Serving Skew

Train/serve skew is when the features used during training are computed differently than during serving. Example: training uses 30-day average transaction amount; serving accidentally uses 7-day average. The model is calibrated on one distribution but receives another — causing silent degradation. Feature stores solve this by computing features once and serving them to both.

🏪 Feature Store Architecture

  • Offline Store: Historical features in data warehouse (S3 + Parquet). Used for model training.
  • Online Store: Latest features in low-latency store (Redis, DynamoDB). Used for serving, <10ms lookup.
  • Feature Registry: Catalog of all features — definitions, owners, lineage, statistics.
  • Feature Pipeline: Compute features once; materialize to both stores.

🛠️ Popular Feature Stores

  • Feast — Open source, framework-agnostic, popular baseline
  • Tecton — Enterprise, managed, real-time + batch
  • Vertex AI Feature Store — GCP-native, managed
  • SageMaker Feature Store — AWS-native, offline+online
  • Hopsworks — Open source, Spark-native, Python SDK
🌍 Real-World Example — Uber

Uber's Michelangelo feature store powers 1000+ ML features across 200+ models. A feature like driver_avg_rating_last_30_trips is computed once — materialized to the offline store for training and to Redis for real-time serving. When a new driver-matching model needs this feature, it's ready in minutes, not weeks of engineering.

MODULE 16 // LLMOPS
LLMOps — MLOps for Large Language Models
The new operational challenges introduced by foundation models

LLMOps extends traditional MLOps with new concerns specific to foundation models: prompt management, evaluation without ground truth, cost at token-level, hallucination monitoring, and the complexity of fine-tuning vs. prompting.

🔵 Traditional MLOps

  • Train from scratch on labeled data
  • Single model per use case
  • Versioning: weights + code
  • Latency: 5–50ms typical
  • Cost: GPU hours for training
  • Evaluation: accuracy on test set

🟢 LLMOps (New Challenges)

  • Prompt engineering + RAG as primary dev loop
  • Foundation model + adapter/prompt per use case
  • Versioning: prompt templates + RAG indexes
  • Latency: 500ms–5s, token streaming critical
  • Cost: $0.01–$0.10 per request at scale
  • Evaluation: LLM-as-judge, human feedback, RAGAS

LLMOps Stack

📝

Prompt Management

Version and test prompt templates like code. LangSmith, PromptLayer, Humanloop track prompt versions, A/B test variants, and measure performance over time.

MLOps Interview Q&A
55+ questions from fundamentals to system design
🟢 Fundamentals
QWhat is MLOps and how does it differ from DevOps?▾
MLOps is the discipline of operationalizing machine learning systems reliably and at scale. It extends DevOps with three additional concerns unique to ML: (1) Data versioning — you must version not just code but datasets (often terabytes), because the same code with different data produces a different model. (2) Model validation — beyond unit tests, you need statistical quality gates (accuracy, fairness, latency benchmarks) before deployment. (3) Continuous training — unlike software, ML models degrade over time as data distributions shift, requiring automated retraining pipelines triggered by drift. DevOps asks "did the deploy succeed?" MLOps adds "is the model still accurate on today's data distribution?"
QWhat is model drift? Describe the different types.▾
Model drift is when production performance degrades because something has changed since training. Types: (1) Data drift (covariate shift): input feature distributions change — P(X_prod) ≠ P(X_train). Example: a model trained before a recession sees different income distributions afterward. Detected with KS test, PSI. (2) Concept drift: the X→Y relationship changes — P(Y|X) changes. The hardest to detect without labels. Example: "work from home" sentiment shifting post-COVID. (3) Label drift: target variable distribution changes — P(Y) changes. Example: fraud rate spikes. (4) Prediction drift: model output distribution shifts — detectable without ground truth. Usually the first observable signal. Monitor all four continuously with tools like Evidently, Arize, or WhyLabs.
QWhat is training/serving skew? How do feature stores solve it?▾
Training/serving skew occurs when features are computed differently at training time vs. serving time. Classic example: training uses `avg_spend_last_30d` computed from a data warehouse; serving computes it from a different query or uses different time windows. The model was optimized for one distribution but receives another — causing silent degradation that's hard to diagnose. Feature stores solve this by: defining features once (as declarative feature definitions), computing them once (the same code path), and materializing to both an offline store (S3/Parquet for training) and an online store (Redis/DynamoDB for <10ms serving). Training and serving both read from the feature store API — guaranteed consistency. Uber's Michelangelo, LinkedIn's Frame, and Airbnb's Zipline were early internal implementations of this pattern.
QWhat are the MLOps maturity levels? Where is your current organization?▾
Google defines three levels: Level 0 — manual everything. Data scientists run notebooks, hand off pickle files to engineers, no reproducibility. Most teams start here. Level 1 — automated training pipelines. Models retrain continuously on new data, feature store in use, model validation gates before deployment, but CI/CD for the pipeline itself is still manual. Level 2 — full CI/CD automation. Any change to data, model code, or pipeline definition triggers automated testing and redeployment. Monitoring auto-triggers retraining on drift. Complete observability. When asked about your org, be honest: most production orgs are at L1 working toward L2. Describe specific gaps (e.g., "we have training pipelines but our monitoring doesn't auto-trigger retraining yet") and your plan to close them. L2 is aspirational for most orgs — even strong ML teams at large companies.
QWhat is a model registry and why is it important?▾
A model registry is a centralized catalog that stores trained model artifacts along with metadata (training data version, metrics, environment, lineage) and manages lifecycle stages (Staging → Production → Archived). Importance: (1) Single source of truth — serving code always loads from the registry, never from someone's local filesystem. (2) Instant rollback — promoting a previous version takes seconds. (3) Audit trail — regulators can see exactly which model version served which decisions on what date. (4) Governance gate — promotion requires approval (human or automated quality gate). MLflow Registry is the most common open-source option. In production: serving code uses `mlflow.pyfunc.load_model("models:/fraud_detector/Production")` — decoupled from the training process entirely. When a new model is promoted, serving picks it up on next restart or hot-reload.
QWhat is the difference between batch inference and real-time inference?▾
Batch inference: score a large dataset offline, store results, serve pre-computed predictions. Advantages: no latency constraint, optimized for throughput (vectorized operations, Spark), cheap compute (spot instances). Use when: predictions can be pre-computed ahead of need (e.g., nightly churn scores, product recommendations refreshed daily). Real-time inference: score individual requests on-demand, sub-100ms response required. Advantages: always uses the latest model and features, personalized per-request context. Use when: decision must react to real-time inputs (fraud detection, search ranking, dynamic pricing). Tradeoff: real-time is 10–100× more expensive due to always-on infrastructure. Some architectures use near-real-time (streaming via Kafka/Flink) — a middle ground where predictions update every few minutes rather than synchronously or nightly.
🟡 Architecture & Engineering
QHow would you design a CI/CD pipeline for an ML model?▾
A robust ML CI/CD has three quality gates: (1) Code Gate: unit tests, linting, type checking — same as software CI. Test feature engineering functions, custom model wrappers, preprocessing code. (2) Data Gate: schema validation (Great Expectations, Pandera), statistical tests (null rate, distribution checks), referential integrity. Block pipeline if data quality fails. (3) Model Gate: train on current data, evaluate on held-out test set. Check: accuracy vs. threshold, fairness across slices (gender, age groups), latency benchmark (<50ms p95), comparison against current production model (must beat champion by X%). If all gates pass: register to staging, deploy to staging environment, run integration tests, canary deploy to 5% production traffic, monitor for 24h, promote to 100%. Use GitHub Actions / Jenkins / CircleCI to orchestrate. Key insight: treat pipeline code changes with the same rigor as model changes — a broken pipeline is as bad as a broken model.
QWhat is a feature store? When would you build one vs. use a managed solution?▾
A feature store has three components: (1) Feature registry — definitions, owners, documentation, lineage, statistics for every feature. (2) Offline store — historical feature values in data warehouse (S3+Parquet, BigQuery) for training. (3) Online store — latest feature values in low-latency store (Redis, DynamoDB) for serving. When to build vs. buy: Build if you have very specific requirements (custom compute engines, unusual data sources) and a dedicated platform team. Takes 6–18 months to build properly. Managed (Feast + Redis) if you're a mid-size team — open source with reasonable operational burden, good community. Fully managed (Tecton, Databricks FS, Vertex FS) if you're an enterprise with compliance requirements and want zero infrastructure management. Key insight: the feature store is most valuable when multiple teams share features — a fraud team's transaction features can be reused by the credit team without rebuilding them.
QExplain the champion-challenger model deployment pattern.▾
Champion-challenger is a controlled A/B testing pattern for model deployment. The champion is the current production model. The challenger is a new candidate model. A traffic-splitting layer routes X% of production traffic to the challenger while keeping (100-X)% on the champion. Both models' predictions are logged. After sufficient traffic (statistical power), compare their business metrics (CTR, conversion, loss rate) and ML metrics (precision, recall). If the challenger wins statistically significantly, promote it to champion. Key implementation details: ensure cohort assignment is consistent (same user always hits the same model during the test), log all predictions from both models with experiment ID, use proper statistical tests (t-test for continuous metrics, chi-squared for conversions, Bayesian methods for faster decisions). This is safer than full canary because the champion always handles most traffic, and rollback is instant — just set traffic split to 0%/100%.
QHow do you handle data validation in ML pipelines?▾
Three layers: (1) Schema validation: check data types, required columns, expected value ranges. TensorFlow Data Validation (TFDV) or Great Expectations. Block the pipeline if schema mismatches. (2) Statistical validation: compare feature distributions against a reference baseline (usually training data). Alert if PSI > 0.2, null rate increases >5%, mean shifts >3σ. Run automatically before every training run. (3) Semantic validation: business logic checks — e.g., `age` must be >18, transaction amounts must be positive, event timestamps must be monotonically increasing. Great Expectations example: expect_column_values_to_be_between("age", 0, 120). Treat data tests like unit tests — check into version control, run in CI, fail loudly. The cost of training on corrupt data (wasted compute, bad model in production) is 100× the cost of writing the tests.
QWhat is shadow deployment and when would you use it?▾
In shadow deployment, 100% of production requests are sent to the current production model and also to the new shadow model. The shadow model's predictions are logged but never returned to users. This lets you: (1) Validate that the new model produces sensible predictions on real traffic before any user impact, (2) Measure latency in real production conditions, (3) Compare predictions of old vs. new model — flag cases where they disagree significantly for human review, (4) Test infrastructure stability (new container, new endpoint) under real load. Use shadow deployment when: the new model uses a fundamentally different architecture, the cost of a bad prediction is very high (medical, financial), or when you want production-scale traffic validation before canary. Downside: doubles serving costs during the shadow period. Duration: typically 24h–7 days depending on traffic volume and confidence requirements.
🟠 Monitoring & Operations
QWhat metrics would you monitor for a production ML model?▾
Four categories: (1) ML metrics (when labels are available): accuracy, precision, recall, AUC, RMSE. Monitor on a rolling window (7d, 30d). Alert on >2% drop. (2) Data/Drift metrics: feature distribution statistics (mean, std, quantiles, null rate) vs. training baseline. Prediction distribution. PSI per feature. No labels required. (3) System metrics: request latency (p50, p95, p99), error rate (4xx, 5xx), throughput (RPS), GPU/CPU utilization, memory usage, queue depth. Standard SRE metrics. (4) Business metrics: this is the gold standard — conversion rate, click-through, fraud loss rate, customer satisfaction. A model that passes all ML metrics but fails business metrics is still failing. Set SLOs on business metrics and alert the data science team when they degrade. Use a monitoring tool (Evidently, Arize, Fiddler) for ML metrics, Prometheus/Grafana for system metrics, and Amplitude/Looker for business metrics.
QHow would you handle a model that has degraded in production?▾
Incident response playbook: (1) Triage: is this a model problem or a data problem? Check feature distributions — if inputs look normal, it's the model. If features look abnormal, it's a data pipeline issue. (2) Immediate mitigation: if severity is high, rollback to the previous model version immediately (take seconds if you have a model registry). Better to serve a stale-but-working model than an accurate-on-paper-but-failing model. (3) Root cause analysis: use LangSmith-style traces or MLflow runs to compare current predictions vs. 30 days ago. Identify which features or segments changed most. (4) Fix strategy: if data drift, retrain on recent data — does retraining fix it? If concept drift, may need new feature engineering or architectural changes. (5) Prevention: update monitoring thresholds, add the specific failure mode to your monitoring suite, consider adding the triggering scenario to your evaluation test suite so it's caught in CI next time.
QWhat is PSI (Population Stability Index) and how do you use it for drift detection?▾
PSI measures how much the distribution of a variable has shifted between a reference period (usually training) and the current period. Formula: PSI = Σ ((%current - %expected) × ln(%current / %expected)) calculated over buckets/bins of the variable. Interpretation: PSI < 0.1 = no significant shift, no action needed. PSI 0.1–0.2 = moderate shift, monitor closely. PSI > 0.2 = major shift, investigate and likely retrain. Advantages over simpler tests (like just comparing means): it captures the full shape of the distribution, is sensitive to both tails, and is easy to explain to non-statisticians. Implemented easily with Evidently or custom code. Apply PSI to: each input feature, the model's prediction score distribution, and target variable distribution. In practice, set PSI > 0.2 on prediction distribution as an automatic alert that triggers retraining evaluation. PSI is widely used in financial services model validation frameworks (SR 11-7 guidance).
QHow do you debug a model that works well offline but fails in production?▾
Systematic debugging checklist: (1) Training/serving skew first: compare feature statistics logged at serving time vs. training data. Any feature with significantly different distribution is suspect. Even small differences (different normalization, different imputation for nulls) can cause large performance gaps. (2) Label leakage: did any training features use future information? A common bug: using `account_closed` as a feature when predicting churn — this leaks the label itself. (3) Data pipeline bugs: trace a specific production example end-to-end. What raw inputs come in? After preprocessing, what does the feature vector look like? Compare with a training example. (4) Model loading issues: verify the serialized model loads correctly — sometimes sklearn version differences cause silent discrepancies. Test with saved model artifacts, not the in-memory object. (5) Population mismatch: your test set may not represent production traffic. If model was evaluated on random 20% split but production gets a specific user segment, performance can differ dramatically. Use time-based splits for time-series data.
🔴 System Design
QDesign a complete MLOps platform for a fraud detection system.▾
Architecture:

Data Layer: Raw transactions → Kafka → Flink (real-time features) + Spark (batch features) → Feature Store (Tecton: Redis for online, S3/Delta for offline). Feature catalog tracks 500+ features with owners and SLAs.

Training Pipeline: Airflow DAG runs nightly. Steps: (1) Pull training data from offline feature store (point-in-time correct to avoid leakage), (2) Great Expectations validation, (3) Train XGBoost + LightGBM + Neural Net ensemble, (4) Evaluate: accuracy, AUC, FPR@1% threshold, fairness across merchant categories, latency benchmark, (5) Compare vs. champion model — must improve AUC by ≥0.3% to promote, (6) Register to MLflow Model Registry.

Serving: FastAPI + NVIDIA Triton on EKS. 2ms p95 latency SLO. Real-time feature lookup from Redis. Horizontal auto-scaling. Blue-green deployment.

Monitoring: Evidently for feature drift (hourly), Arize for prediction drift, Prometheus/Grafana for system metrics. Alert on PSI > 0.2, p99 latency > 10ms, error rate > 0.1%. Automatic retraining trigger on sustained drift. Human approval required before production promotion.

Governance: Every prediction logged with SHAP values for explainability. Quarterly fairness audits. Full lineage in MLflow — can reproduce any prediction from 3 years ago.
QHow do you scale ML training to handle 1TB+ datasets?▾
Multi-strategy approach: (1) Distributed data loading: partition data across workers. Use Parquet format (efficient columnar reads), multiple workers in PyTorch DataLoader, prefetch next batch while GPU trains on current. (2) Data parallelism: PyTorch DDP or Horovod — each GPU gets a different batch shard, gradients averaged via AllReduce. Linear speedup up to ~16 GPUs, then communication overhead. (3) Approximate training: for traditional ML (XGBoost), use Dask-ML or Spark MLlib for distributed training on commodity clusters. Can handle 10TB+ with enough nodes. (4) Streaming training: train incrementally on data chunks — online learning or mini-batch gradient descent. Never loads full dataset into memory. Vowpal Wabbit for large-scale linear models. (5) Smart sampling: often 10% of data gives 95% of the model quality. Use stratified sampling, then train on full data only for the final champion model. (6) Managed compute: SageMaker Training with P3/P4 instances, Spot instances (70% savings with checkpointing), automatic job retry on spot interruption.
QDesign a model monitoring system for 200 production models.▾
Core Components:

(1) Unified logging: all models emit predictions + features to Kafka. Standard schema: {model_id, version, timestamp, features, prediction, confidence, request_id}. Flink consumer materializes to data warehouse + real-time aggregations.

(2) Metric computation: nightly batch job computes PSI for each feature × each model (200 models × 50 features avg = 10K PSI scores/day). Streaming job computes rolling 1h/6h/24h prediction distribution shifts.

(3) Alerting tiers: Yellow (PSI 0.1–0.2) → email to model owner. Red (PSI >0.2 or accuracy drop >3%) → PagerDuty on-call + auto-create retraining run. Sev1 (latency SLO breach) → immediate rollback + incident.

(4) Dashboard: Grafana showing model health across the fleet. Red/yellow/green per model. Trend lines. Filter by team, business domain, deployment date.

(5) Self-service: model owners can configure their own drift thresholds and alert channels. Default thresholds applied if not configured.

(6) Label collection: delayed label join — when ground truth arrives (fraud confirmed, churn happened), automatically join to predictions for accuracy tracking. Different models have different label delay (fraud: hours, churn: 30 days).
QWhat is the difference between online learning and continual/continuous training?▾
Online learning: the model updates its weights incrementally with every incoming data point or mini-batch — no periodic retraining. Algorithms: SGD with streaming data, Vowpal Wabbit, river (Python library). Pros: always uses the most recent data, no training/serving gap. Cons: catastrophic forgetting (new data overwrites old patterns), hard to debug (model is always changing), no stable checkpoint to roll back to. Best for: high-volume, fast-changing streams where currency matters more than stability (ad CTR prediction, real-time bidding). Continuous/continual training: periodic full retraining triggered by schedule, data volume thresholds, or drift detection. Model is retrained from scratch (or from checkpoint) on a sliding or expanding window of data. The model is stationary between retrain cycles. Pros: reproducible, debuggable, easy rollback. Cons: latency between drift detection and model update (could be hours to days). Best for: most production ML systems. These are different architectures — choose based on how fast your data distribution changes and your tolerance for model instability vs. data currency.
QHow do you handle model versioning and rollback in production?▾
Complete versioning strategy: (1) Model registry: every trained model is registered with a semantic version (e.g., fraud_detector v7.2.1), along with: training data version hash (DVC), training code git SHA, evaluation metrics, environment (conda/pip lockfile), training date. (2) Lifecycle stages: Candidate → Staging → Production → Archived. Transitions require: automated quality gates + human approval for Production promotion. (3) Serving decoupling: serving code references `models:/fraud_detector/Production` — never a specific version. Promoting a new version is changing a registry pointer, not redeploying infrastructure. (4) Rollback procedure: if P99 latency spikes or accuracy degrades, on-call engineer runs: client.transition_model_version_stage("fraud_detector", version=6, stage="Production"). Serving picks up the previous version on next health check cycle (~60s). (5) Dark rollback: if even rollback is risky, use the feature flag system to re-route traffic to a simpler rule-based fallback while the ML team investigates.
💼 Conceptual & Behavioral
QExplain the ML model lifecycle from a compliance perspective.▾
For regulated industries (finance, healthcare), compliance requires a documented lifecycle: (1) Scope and purpose: documented intended use, business owner, risk tier (high-risk models need more oversight). (2) Data governance: data lineage (where did training data come from, how was it collected, was consent obtained), PII handling, data retention policies. (3) Model development: documented methodology, hyperparameter selection rationale, model card with limitations and assumptions. (4) Validation: independent validation (not by the team that built it), performance across segments, stress testing, sensitivity analysis. (5) Deployment approval: sign-off from risk management, legal, and compliance. Change management documentation. (6) Ongoing monitoring: regular performance reports (quarterly for high-risk models), drift monitoring with documented thresholds, annual independent validation. (7) Retirement: documented decommission, data retention per regulation. For EU AI Act high-risk systems: conformity assessment, CE marking, registration in EU database. SR 11-7 (US banking): effective challenge, independent review, ongoing performance monitoring.
QWhat is LLMOps and how does it differ from traditional MLOps?▾
LLMOps extends MLOps for foundation model-based systems. Key differences: (1) Development loop: traditional MLOps focuses on model training; LLMOps focuses on prompt engineering and RAG as primary optimization levers — fine-tuning is expensive and often unnecessary. (2) Versioning: version prompt templates and RAG index configurations, not just model weights. LangSmith tracks prompt versions like git for code. (3) Evaluation: without ground truth labels, use LLM-as-judge, human preference data, and task-specific metrics (RAGAS for RAG: faithfulness, relevance, context recall). (4) Cost management: token-level cost tracking is critical — $0.01–0.10 per request × millions of users = real money. Optimize with semantic caching, model routing (cheap model for simple tasks), response streaming. (5) Observability: every prompt + completion logged, hallucination detection, toxicity monitoring, latency at the token generation level. (6) Security: prompt injection detection, output filtering, PII detection in completions. Traditional MLOps security was mostly access control; LLMOps security is a runtime adversarial problem.
QHow would you prioritize what to monitor when you have limited resources?▾
Prioritization framework: (1) Monitor business impact first: the metric that matters most is the business metric the model drives. Conversion rate, fraud loss, churn. If you can only monitor one thing, monitor the downstream business KPI. (2) Prediction distribution second: requires no labels, detectable immediately, and is usually the first signal of drift. If prediction score distribution shifts significantly, something has changed. (3) Key features third: not all features are equal. Use SHAP values from training to identify the top 5–10 most impactful features — focus drift monitoring there. A drift in a low-importance feature rarely matters; a drift in the #1 feature always does. (4) System health always: latency, error rate. Non-negotiable regardless of resources. (5) Skip last: monitoring every feature equally, computing full accuracy without labels, extensive fairness reporting for low-risk models. Start minimal, add coverage as incidents teach you what matters. Each production incident should add a new monitor to prevent recurrence.
