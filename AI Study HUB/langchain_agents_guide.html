// 01 — FOUNDATIONS
What are AI Agents?
Systems that perceive, reason, act, and observe — in a loop
Definition

An AI Agent is a system where an LLM acts as a reasoning engine that dynamically decides which actions to take, in what order, and whether to continue or stop — based on the goal and what it observes. Unlike a fixed pipeline, agents adapt their execution path at runtime.

The key shift from traditional software: instead of you programming every step, the LLM decides the steps. You give it a goal and tools — it figures out how to use them.

🌍 Real-World Example

Task: "Research the top 3 EV manufacturers by market cap, get their latest quarterly revenue, and write a comparison report."

Traditional code: You'd write scraping → parsing → formatting logic explicitly.
Agent: It decides to → search for EV companies → use a finance API tool → summarize results → write a report. It may retry, handle errors, and adjust — all without you specifying each step.

The LCEL (LangChain Expression Language)

LCEL is the pipe-based composition syntax that chains components together declaratively. Every component is a Runnable with a standard interface.

from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# LCEL: pipe operator chains Runnables
prompt = ChatPromptTemplate.from_template("Explain {topic} in one sentence.")
llm = ChatOpenAI(model="gpt-4o")
parser = StrOutputParser()

chain = prompt | llm | parser   # left-to-right composition
result = chain.invoke({"topic": "LangChain agents"})
# Supports .stream(), .batch(), .ainvoke() out of the box

// 04 — ARCHITECTURE
Agent Core Components
The building blocks of every LangChain agent
1
Agent (LLM + Prompt)

The reasoning unit. Takes inputs (user message + tool results) and outputs either a tool call or a final answer. Uses the LLM's function-calling capability or structured output.

2
AgentExecutor

The runtime loop. Manages the observe-think-act cycle. Handles tool invocation, error recovery, max_iterations, and when to stop. Think of it as the "process manager" for the agent.

3
Tools

Functions the agent can call. Defined with name, description, and input schema. The description is critical — the LLM reads it to decide when to use the tool.

4
Memory / State

Stores conversation history and intermediate results. Injected into the prompt on each agent step so the LLM has full context of what's happened.

5
Prompt Template

Structures the agent's instructions. Includes system prompt, conversation history, available tools, and scratchpad (chain of thought). Format is critical for agent reliability.

6
Output Parser

Parses the LLM's output into structured action objects (tool name + input) or final answer. With modern tool-calling LLMs, this is handled natively by the model.

from langchain.agents import create_tool_calling_agent, AgentExecutor
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# 1. Define tools
@tool
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return call_weather_api(city)

# 2. Create prompt
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder("chat_history", optional=True),
    ("human", "{input}"),
    MessagesPlaceholder("agent_scratchpad"),
])

# 3. Create agent + executor
llm = ChatOpenAI(model="gpt-4o", temperature=0)
agent = create_tool_calling_agent(llm, [get_weather], prompt)
executor = AgentExecutor(agent=agent, tools=[get_weather], verbose=True)

result = executor.invoke({"input": "What's the weather in Tokyo?"})

Creating Custom Tools — 3 Ways

from langchain_core.tools import tool, BaseTool
from pydantic import BaseModel, Field

# Method 1: @tool decorator (simplest)
@tool
def search_products(query: str, max_price: float = 100.0) -> str:
    """Search product catalog. Use when user asks about products or prices."""
    return db.search(query, max_price)

# Method 2: StructuredTool with Pydantic schema
class OrderInput(BaseModel):
    order_id: str = Field(description="The order ID to look up")
    include_tracking: bool = Field(default=False)

@tool("get_order_status", args_schema=OrderInput)
def get_order(order_id: str, include_tracking: bool) -> str:
    """Retrieve status and details for a customer order."""
    return orders_api.get(order_id, tracking=include_tracking)

# Method 3: BaseTool subclass (full control)
class SentimentTool(BaseTool):
    name: str = "analyze_sentiment"
    description: str = "Analyzes sentiment of customer reviews. Returns: positive/negative/neutral + score."

    def _run(self, text: str) -> str:
        score = sentiment_model.predict(text)
        return f"Sentiment: {score.label}, confidence: {score.score:.2f}"

    async def _arun(self, text: str) -> str:  # async version
        return await async_sentiment.predict(text)
💡

Tool Description is Everything: The LLM decides which tool to call solely based on the description. "Search the web" is bad. "Use when you need current information, news, or facts from after your training cutoff. Input should be a specific search query." is good. Treat descriptions like API documentation.

Memory Systems
How agents remember — short-term, long-term, episodic
⚡

Buffer Memory

Store the full conversation history in the prompt. Simplest, most reliable. Limited by context window — eventually truncates older messages.

ConversationBufferMemory
🪟

Window Memory

Keep only the last K exchanges. Prevents context overflow. Loses older context but keeps recent dialogue coherent.

ConversationBufferWindowMemory
📝

Summary Memory

LLM periodically summarizes older conversation and stores the summary instead of raw messages. Compresses history while preserving key facts.

ConversationSummaryMemory
🔮

Vector Store Memory

Embed conversation turns and store in vector DB. Retrieve semantically relevant past exchanges, not just the most recent ones. True long-term memory.

VectorStoreRetrieverMemory
🕸️

LangGraph State

In LangGraph, memory is explicit graph state — persisted between runs via checkpointers (SQLite, Redis). The modern recommended approach.

StateGraph + Checkpointer
🏷️

Entity Memory

Track specific entities (people, places, things) mentioned in conversation. Maintains an entity-fact store that updates as the conversation progresses.

ConversationEntityMemory

// 09 — PATTERN
The ReAct Pattern
Reason + Act — the reasoning framework powering most agents

ReAct (Reason + Act) is the prompting framework where the model interleaves thoughts (reasoning steps) and actions (tool calls) in a structured scratchpad. Published by Google/Princeton in 2022, it dramatically improves agent reliability by forcing explicit reasoning before every action.

# ReAct Scratchpad Format
"""
Question: What is the population of the capital of France?

Thought: I need to find the capital of France, then get its population.
Action: search
Action Input: "capital of France"
Observation: Paris is the capital of France.

Thought: Now I need the population of Paris.
Action: search
Action Input: "population of Paris 2024"
Observation: Paris has a population of approximately 2.1 million (city proper) or 12 million (metro area).

Thought: I have the answer.
Final Answer: The population of Paris (capital of France) is about 2.1 million in the city, or 12 million in the greater metropolitan area.
"""

Why ReAct Works

🧠

Explicit Reasoning

The "Thought" step forces the model to plan before acting — reducing impulsive, incorrect tool calls. It's chain-of-thought applied to agentic settings.

🔍

Observable Intermediate Steps

Every thought and observation is logged. When things go wrong, you can see exactly where the reasoning broke down — invaluable for debugging.

🔄

Adaptive Replanning

Each observation can update the agent's plan. If search returns unexpected results, the "Thought" step allows the agent to adjust its approach dynamically.

🌍 Real-World Example — Research Agent

Task: "Compare the GitHub star growth of LangChain vs LlamaIndex in 2024."

Without ReAct: Model might hallucinate numbers or call the wrong tool.
With ReAct: Thought: "I need to get GitHub star counts at Jan 2024 and Dec 2024 for both repos. Let me search each." → calls search × 4 → Thought: "Now I'll calculate the growth rate." → Final answer with real data and math shown.

// 10 — LANGGRAPH
LangGraph Deep Dive
Stateful, controllable agent graphs — the future of LangChain agents
What is LangGraph?

LangGraph is a library for building stateful, multi-actor applications with LLMs using graph-based workflows. Each node in the graph is a function or agent. Edges define routing (conditional or fixed). State is explicitly typed and persisted between steps — making complex agent behavior controllable and debuggable.

Core Concepts

📊

State

A typed dictionary (TypedDict) shared across all nodes. Every node reads and writes to this state. Makes data flow explicit and inspectable.

TypedDict
⬡

Nodes

Functions that take state as input and return state updates. Can be LLM calls, tool executions, human checkpoints, or any Python function.

Callable
↗️

Edges

Connections between nodes. Can be fixed (always go to X after Y) or conditional (go to X or Y based on a function that reads state).

Router Logic
💾

Checkpointers

Persist graph state between steps and runs. Enable pause/resume, human-in-the-loop, and durable long-running agents.

SQLite / Redis / Postgres
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

# 1. Define state schema
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]  # reducer: append new messages
    next_step: str

# 2. Define node functions
def call_llm(state: AgentState):
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def run_tools(state: AgentState):
    tool_calls = state["messages"][-1].tool_calls
    results = [tools[tc["name"]].invoke(tc["args"]) for tc in tool_calls]
    return {"messages": results}

# 3. Router: decide what happens after LLM responds
def should_continue(state: AgentState):
    last_msg = state["messages"][-1]
    return "tools" if last_msg.tool_calls else "end"

# 4. Build graph
graph = StateGraph(AgentState)
graph.add_node("agent", call_llm)
graph.add_node("tools", run_tools)
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_continue, {"tools": "tools", "end": END})
graph.add_edge("tools", "agent")  # always go back to agent after tools
app = graph.compile(checkpointer=checkpointer)

// 11 — RAG AGENTS
Agents + RAG
Combining retrieval with dynamic reasoning

Plain RAG is a fixed pipeline: retrieve → generate. An agentic RAG system lets the LLM decide whether to retrieve, what to retrieve, and how many times — enabling more intelligent information gathering.

📚

Basic RAG Tool

Wrap your retriever as a tool. The agent can call it when it needs information from your knowledge base. Agent decides when to retrieve based on the question.

🔁

Iterative Retrieval

Agent retrieves, analyzes gaps in the retrieved context, then retrieves again with refined queries. Handles multi-hop questions that simple RAG can't.

🗂️

Multi-Source Routing

Agent decides which of multiple knowledge bases or APIs to query based on the question type. HR questions → HR DB, technical questions → tech docs.

✅

Self-Correcting RAG

Agent evaluates retrieved documents for relevance, falls back to web search if retrieval is poor, and verifies its answer against the sources.

from langchain.tools.retriever import create_retriever_tool
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

# Create vector store retriever
vectorstore = FAISS.from_documents(docs, OpenAIEmbeddings())
retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

# Wrap retriever as a tool the agent can call
retriever_tool = create_retriever_tool(
    retriever,
    name="search_company_docs",
    description="Search internal company documentation and policies. "
                "Use for questions about company procedures, HR policies, or product specs."
)

# Agent has both retrieval and web search
tools = [retriever_tool, tavily_search, python_repl]
agent = create_tool_calling_agent(llm, tools, prompt)
# Agent now decides: use internal docs, web, or code based on the question
🌍 Real-World Example

Enterprise Knowledge Agent: Employee asks "What's our parental leave policy, and how does it compare to the industry average?"

→ Agent thinks: "First question needs internal docs, second needs web."
→ Calls search_company_docs for policy → reads 16 weeks paid.
→ Calls web_search for "average parental leave tech companies 2024" → 12 weeks.
→ Synthesizes: "Your policy (16 weeks) exceeds the industry average (12 weeks)."

// 12 — MULTI-AGENT
Multi-Agent Systems
Orchestrating teams of specialized agents

Complex tasks often benefit from multiple specialized agents working together — just like a team of specialists outperforms a single generalist on hard problems. LangGraph enables rich multi-agent architectures.

👑

Supervisor Pattern

One supervisor agent orchestrates multiple specialized sub-agents. Supervisor decides which agent to call, routes results, and synthesizes the final answer.

Most Common
🔀

Hierarchical

Supervisor → sub-supervisors → worker agents. For very complex tasks. Each tier handles routing at its level of abstraction.

Enterprise Scale
🤝

Collaborative / Debate

Multiple agents reason independently, share findings, and critique each other. Improves accuracy on complex reasoning via ensemble effect.

High Accuracy
⚡

Parallel Execution

Send subtasks to multiple agents simultaneously (via LangGraph fan-out). Dramatically reduces latency for tasks with independent sub-components.

Performance
🌍 Real-World Example — Investment Research Team

Task: "Research Tesla as an investment opportunity."

Supervisor routes to 4 parallel agents:
• Financials Agent: Pulls and analyzes SEC filings, revenue, margins
• News Agent: Searches recent Tesla news, sentiment analysis
• Competitor Agent: Compares BYD, Rivian, legacy OEMs
• Technical Agent: Reads price charts, key levels

Supervisor synthesizes all 4 reports → comprehensive investment memo in 45 seconds vs. 4 hours manually.

Seeing inside the agent's mind

🔭

Tracing

Full trace of every agent step — input, output, tokens used, latency, cost. Nested view shows how sub-calls relate to the top-level request.

LangSmith
📊

Evaluation

Run test datasets against your agent, score outputs with LLM-as-judge or custom evaluators. A/B test prompt changes before deploying.

Evals
🔍

Debugging

Replay any failed trace, inspect the exact prompt sent to the LLM, see what the model "saw" at each step. Critical for fixing hallucinations and wrong tool calls.

Root Cause
💰

Cost Monitoring

Track token usage and cost per run, per user, per endpoint. Set alerts when cost/latency exceeds thresholds.

// 14 — PRODUCTION
Production Patterns & Best Practices
What separates toy agents from production systems
🛡️

Input/Output Guardrails

Validate user inputs, detect prompt injection. Run output through moderation API before returning. Use Guardrails AI or Nemo Guardrails for enterprise.

⏱️

Timeouts & Retries

Always set max_iterations (default 15). Wrap tool calls in try/except. Implement exponential backoff for API calls. Never let an agent run forever.

🔐

Minimal Permission Principle

Give agents only the tools they need for their task. A customer service agent shouldn't have database write access. Reduce blast radius of failures.

👤

Human-in-the-Loop

For high-stakes actions (send email, make payment, delete data), require human approval via LangGraph interrupt_before. Don't let agents take irreversible actions autonomously.

📦

Structured Outputs

Use Pydantic models for agent outputs. Validate with .with_structured_output(). Prevents downstream parsing failures from malformed LLM responses.

🔄

Fallback Strategies

Define what happens when the agent gets stuck or exceeds iterations. Graceful degradation: escalate to human, use simpler chain, or return a meaningful error.

⚠️

The Reliability Stack: Production agents need: (1) reliable tools with error handling, (2) LangSmith tracing for debugging, (3) max_iterations cap, (4) structured output validation, (5) human-in-the-loop for irreversible actions, and (6) comprehensive evals on a golden test set run before every deployment.

// ★ — INTERVIEW PREP
Interview Questions & Answers
50+ questions from fundamentals to system design
🟢 Fundamentals
QWhat is LangChain and why was it created?▾
LangChain is an open-source framework for building LLM-powered applications. It was created by Harrison Chase in Oct 2022 to solve the problem of composability — connecting LLMs to external tools, data sources, and each other was tedious and inconsistent. LangChain provides standard abstractions (Chains, Agents, Memory, Retrievers) so developers don't re-implement the same plumbing repeatedly. It's become the most popular LLM framework because it dramatically reduces boilerplate for common patterns like RAG, chatbots, and autonomous agents.
QWhat is the difference between a Chain and an Agent in LangChain?▾
A Chain has a fixed, predetermined sequence of steps. You define the order, and it always runs the same way regardless of input. A Chain is deterministic and predictable. An Agent uses an LLM to dynamically decide what steps to take — which tools to call, in what order, and when to stop. The LLM is the "controller" of the execution flow. Chains are better when you know exactly what needs to happen (e.g., RAG pipeline). Agents are better when the task requires flexible reasoning and tool use (e.g., research assistant). In practice: use chains by default, reach for agents when you need dynamic tool selection.
QWhat is LCEL (LangChain Expression Language)? Why is it useful?▾
LCEL is a declarative way to compose LangChain components using the pipe operator (|). Every component is a Runnable with a standard interface (.invoke(), .stream(), .batch(), .ainvoke()). Benefits: (1) Automatic streaming — the full chain streams without extra code, (2) Async support — all runnables support async out of the box, (3) Parallel execution — use RunnableParallel to run steps in parallel, (4) Tracing — all LCEL chains are automatically traced by LangSmith, (5) Type safety — input/output types are enforced at each step. Example: chain = prompt | llm | parser is more readable than manually calling each step.
QWhat is the ReAct pattern? Why does it improve agent reliability?▾
ReAct (Reason + Act) is a prompting framework where the agent alternates between Thought (explicit reasoning about what to do next) and Action (calling a tool), followed by an Observation (tool result). This loop continues until a final answer is reached. It improves reliability because: (1) The explicit "Thought" step forces the model to plan before acting — reduces impulsive wrong tool calls, (2) Multi-step tasks are broken into traceable sub-steps — easier to debug, (3) Each observation can trigger replanning — the agent adapts to unexpected results, (4) The scratchpad creates a "working memory" for complex reasoning. It was shown to significantly outperform pure reasoning (CoT) or pure acting approaches on complex multi-step tasks.
QHow do you define a tool in LangChain?▾
Three main ways: (1) @tool decorator — simplest, puts the function docstring as description, infers input schema from type hints. (2) @tool with args_schema — attach a Pydantic model for complex, multi-field inputs with descriptions per field. (3) BaseTool subclass — full control over name, description, sync and async _run methods, and custom error handling. The most critical design decision is the description — the LLM decides when to call a tool solely based on its description. A well-crafted description that specifies "when to use" and "what to input" is more important than the implementation.
QWhat memory options does LangChain provide? When would you use each?▾
ConversationBufferMemory: stores all messages — use for short conversations where you need full history. ConversationBufferWindowMemory: keeps last K turns — use when you need to prevent context overflow but older history isn't needed. ConversationSummaryMemory: LLM summarizes older turns — use for long conversations where you need compressed but complete history. VectorStoreRetrieverMemory: semantic retrieval of past turns — use for long-term user memory across sessions. LangGraph Checkpointer: explicit state persistence — the modern recommended approach for production agents needing durable memory across sessions, restarts, and failures.
🟡 Architecture & Internals
QExplain AgentExecutor. What does it do internally?▾
AgentExecutor is the runtime that manages the agent loop. Internally: (1) It calls the agent (LLM) with the current input + scratchpad, (2) Parses the LLM output to extract tool name and input, (3) Executes the tool, (4) Appends the tool result to the scratchpad as an "Observation", (5) Loops back to step 1 — until the LLM produces a "Final Answer" or max_iterations is reached. It also handles: error recovery (if tool throws, feed error back to LLM), handle_parsing_errors (if output parsing fails), early_stopping_method (generate final answer or force stop). Key parameters: max_iterations (default 15), max_execution_time (wall clock timeout), return_intermediate_steps (expose scratchpad to caller).
QWhat is LangGraph and how does it differ from AgentExecutor?▾
LangGraph is a graph-based framework for stateful agent workflows. Key differences: State is explicit (TypedDict) and inspectable vs. hidden inside AgentExecutor. Cycles are arbitrary graph cycles vs. a simple agent loop. Persistence is built-in via checkpointers (SQLite, Redis, Postgres) vs. in-memory only. Human-in-the-loop is first-class via interrupt_before nodes vs. very difficult to implement. Parallelism — LangGraph supports fan-out (parallel node execution) for speed. Multi-agent — subgraphs can act as agents in a larger graph. LangGraph is the recommended approach for new production agents. AgentExecutor is simpler for basic single-agent use cases but hitting a ceiling for complex requirements.
QWhat is a Runnable in LangChain? Why is it central to the framework?▾
A Runnable is the core interface in langchain-core that every component implements. It provides a standard set of methods: invoke(input), stream(input), batch(inputs), ainvoke(input), astream(input), abatch(inputs). This uniformity means: (1) Any component can be composed with any other via the pipe | operator, (2) Streaming propagates automatically through the entire chain, (3) Async versions are available for every chain without extra code, (4) All components are automatically traced by LangSmith. Standard implementations include RunnableLambda (wrap any function), RunnableParallel (run multiple in parallel), RunnablePassthrough (pass input unchanged), and RunnableBranch (conditional routing).
QHow does tool calling differ from the original ReAct text-parsing approach?▾
Original ReAct relied on text parsing: the LLM was prompted to output a specific text format ("Action: tool_name\nAction Input: ...") which was then parsed with regex/string matching. This was fragile — any formatting variation would break parsing. Tool calling (function calling) uses the model's native structured output capability: tools are defined as JSON schemas sent to the model alongside the conversation. The model outputs a structured JSON object ({name: "tool_name", arguments: {...}}) as part of its response, not as plain text. Tool calling is: more reliable (no parsing failures), supports complex multi-field inputs, allows parallel tool calls (model can call multiple tools simultaneously in one step), and is type-safe. All modern LLMs (GPT-4, Claude, Gemini) support tool calling natively. ReAct text parsing is now primarily used for open-source models that don't yet support tool calling APIs.
QWhat is the agent_scratchpad and why is it needed in the prompt?▾
The agent_scratchpad is the placeholder in the prompt template where all intermediate steps (tool calls + their results) are injected. It serves as the agent's working memory for the current task. Without it, the LLM has no context of what tools it already called or what results it received — it would repeat the same actions forever. For tool-calling agents, the scratchpad contains the assistant's tool_call messages and the ToolMessage results. For ReAct agents, it contains the formatted "Action/Observation" text. It's populated by MessagesPlaceholder("agent_scratchpad") in the prompt template and filled by the AgentExecutor as it runs the loop. The scratchpad grows with each step and is included in full in every subsequent LLM call, which is why agent runs can consume many tokens.
🟠 Advanced Topics
QHow do you implement human-in-the-loop with LangGraph?▾
LangGraph supports human-in-the-loop via interrupt_before and interrupt_after node flags. When a graph is compiled with interrupt_before=["sensitive_action_node"], execution pauses before that node. The state is saved to the checkpointer. A human reviews the pending action and either approves (resumes the graph with None input) or rejects (updates state with a rejection message). Implementation: app = graph.compile(checkpointer=checkpointer, interrupt_before=["send_email"]). Then call app.invoke(input, config) — it pauses, returns the current state. Human inspects and calls app.invoke(None, config) to resume. This is essential for agents that take irreversible actions like sending emails, making payments, or deleting data.
QWhat is a Supervisor pattern in multi-agent systems? How would you implement it?▾
The Supervisor pattern has one controller agent that routes tasks to specialized sub-agents. The supervisor reads the current state and conversation, then decides which agent should act next (or if the task is complete). Implementation in LangGraph: (1) Create specialized sub-graphs or node functions for each agent (researcher, coder, writer etc.), (2) Create a supervisor node that calls an LLM with a structured output prompt asking "who should act next: {list_of_agents} | FINISH", (3) Add conditional edges from the supervisor to each agent and to END, (4) Each agent node does its work and routes back to the supervisor. Key design decision: the supervisor's prompt must clearly describe each agent's specialty so it routes correctly. For parallel tasks, use a fan-out pattern: supervisor creates subtasks, multiple agents execute in parallel, a merge node combines results.
QHow do you handle agent failures and errors in production?▾
Multi-layer strategy: (1) Tool-level: wrap tool implementations in try/except, return error messages (not exceptions) — let the agent adapt. Use handle_tool_error=True in AgentExecutor. (2) Agent-level: set max_iterations and max_execution_time to prevent infinite loops. Implement handle_parsing_errors to recover from malformed LLM outputs. (3) System-level: implement fallback chains that trigger when the agent fails — degrade to a simpler RAG chain or return a canned "I need help from a human" response. (4) LLM-level: use RunnableWithFallbacks to try GPT-4, then Claude if OpenAI is down. (5) Monitoring: LangSmith alerts on error rate or latency spikes. (6) Retries: use tenacity library for exponential backoff on transient API errors.
QExplain how streaming works in LangChain agents.▾
Streaming in LangChain happens at multiple levels. (1) Token streaming: the LLM streams individual tokens as they're generated. Access via chain.stream(input) — you get tokens as they arrive. (2) Intermediate step streaming: for agents, you often want to stream tool call events and observations before the final answer. Use astream_events() which emits structured events: on_llm_start, on_tool_start, on_tool_end, on_llm_stream. (3) LangGraph streaming: app.stream(input) yields the full state after each node completes — you can stream intermediate states to the UI. For chatbots, use astream_events to show tool calls happening in real-time (e.g., "🔍 Searching the web...") and stream tokens as they arrive for the final answer. This dramatically improves perceived latency in user-facing applications.
QWhat is the difference between vectorstore retrieval and semantic caching in LangChain?▾
Vectorstore retrieval: embedding your user's query → finding semantically similar documents → returning those documents as context for the LLM to answer from. The LLM still generates a new answer each time. Used in RAG pipelines. Semantic caching (RedisSemanticCache, GPTCache): cache LLM responses by semantic similarity of the input. If a new query is semantically similar to a cached query (cosine similarity > threshold), return the cached response without calling the LLM at all. Used for cost/latency reduction. Example: "What's the weather in NYC?" and "What is New York City's current weather?" would hit the same cache entry. Key tradeoff: freshness vs. cost. Semantic caches need TTLs for time-sensitive information. Combine both: use semantic cache as a fast-path, fall back to RAG + LLM for cache misses.
QHow would you evaluate a LangChain agent in production?▾
Three-tier evaluation: (1) Unit tests on tools: test each tool independently with mocked inputs — validate input/output schemas, error handling. (2) Agent trace evaluation: use LangSmith to run a golden test set of (input, expected_output) pairs. Score with: LLM-as-judge (GPT-4 rates helpfulness/accuracy), exact match (for factual tasks), trajectory evaluation (did agent take efficient steps, avoid unnecessary tool calls?). Key metrics: task success rate, average steps per task, average tokens/cost per task, tool call accuracy. (3) Regression testing: run the full eval suite before every deployment. Track scores over time in LangSmith's dataset view. Alert if task success rate drops by >5%. Also: shadow mode testing — run new agent version in parallel with production, log results without exposing to users, compare before switching.
🔴 System Design
QDesign a customer support agent for an e-commerce platform using LangChain.▾
Architecture:

Tools: get_order_status(order_id), get_product_info(sku), search_knowledge_base(query) [RAG over FAQ/policy docs], initiate_return(order_id, reason), escalate_to_human(issue_summary).

Agent: Tool-calling agent with GPT-4o. System prompt includes: company persona, policies (return window, etc.), escalation rules (always escalate billing disputes over $100).

Memory: LangGraph checkpointer with Redis. Thread ID = session ID. Retrieves past issues for returning customers.

Guardrails: Input moderation, interrupt_before=["initiate_return"] for orders >$500 (require human approval). Rate limit per user.

Observability: LangSmith tracing with user_id metadata. Alert on: high step count (agent getting stuck), tool error rate, escalation rate.

Evaluation: Weekly golden test set run (200 scenarios). Success metric: resolved without escalation + customer satisfaction score.
QHow would you build a code review agent using LangGraph?▾
Graph design:

State: {pr_diff, review_comments, security_issues, test_coverage, final_review}

Nodes: (1) fetch_pr — GitHub tool fetches diff, (2) security_scan — specialized agent checks for SQL injection, hardcoded secrets, XSS. (3) style_check — linting agent checks naming, complexity, (4) test_coverage_check — checks if new code has tests, (5) synthesize_review — final agent combines all findings into a structured PR review.

Edges: fetch_pr → [security_scan, style_check, test_coverage_check] (parallel) → synthesize_review → END.

Parallel execution: use RunnableParallel or LangGraph fan-out for steps 2-4 — runs in parallel, reducing total latency from ~45s to ~15s.

Output: Structured Pydantic model with severity-tagged issues, posted back to GitHub via tool. interrupt_before posting to GitHub for human preview.
QWhat are the main security risks of agentic systems and how do you mitigate them?▾
Key risks:

(1) Prompt injection: malicious content in tool outputs tries to hijack the agent. Mitigation: treat tool outputs as untrusted, use separate system prompt that explicitly instructs the model to be skeptical of tool output instructions. (2) Over-privileged tools: agent has database write access but only needs read. Mitigation: principle of least privilege — scope each agent's tools to what it strictly needs. (3) Unintended irreversible actions: agent deletes production data. Mitigation: human-in-the-loop interrupts before all write/delete operations. (4) Data exfiltration: prompt injection tricks agent into sending data to attacker's URL. Mitigation: allowlist domains for any web request tool. (5) Runaway agents: agent loops indefinitely, burning API budget. Mitigation: max_iterations, max_execution_time, spending limits in LLM API settings. (6) Sensitive data in traces: LangSmith traces contain user data. Mitigation: PII stripping before logging, or self-host LangSmith.
QHow do you optimize LangChain agent latency and cost in production?▾
Latency optimizations: (1) Model selection: use GPT-4o-mini or Claude Haiku for simple reasoning steps, reserve GPT-4o for complex decisions. (2) Parallel tool calls: configure LLMs to call multiple tools simultaneously when steps are independent. (3) Streaming: stream tokens to reduce perceived latency even if total latency is the same. (4) Semantic caching: cache frequent identical queries. (5) Reduce tool call chains: design tools that return richer data so fewer calls are needed.

Cost optimizations: (1) Prompt compression: trim verbose system prompts, use LLMLingua to compress context. (2) Selective tracing: only trace 10% of production traffic (sample), 100% of errors. (3) Context window management: don't send the entire conversation history — use summary memory for old turns. (4) Tool result truncation: don't return 10KB tool results if 500 bytes would answer the question. (5) Batch processing: for non-real-time workflows, batch requests instead of streaming. Track cost per task in LangSmith to identify the most expensive operations.
💼 Conceptual & Behavioral
QWhen would you NOT use an agent? What are the pitfalls of over-agentification?▾
Don't use an agent when: (1) The task is a fixed, well-understood pipeline — a chain is cheaper, faster, and more predictable. (2) Reliability is paramount — agents introduce non-determinism that's unacceptable in some domains. (3) Latency is critical — each agent step is at minimum one LLM call; a 5-step agent takes 5× longer than a direct call. (4) You need an audit trail of exact steps — agents can take unexpected paths. Pitfalls of over-agentification: Runaway costs from unnecessary tool calls, debugging nightmares when the agent takes unexpected paths, false sense of capability (the agent "works" in dev but fails on edge cases in prod), security attack surface grows with every tool added. Rule: start with the simplest thing that could work. Usually that's a direct LLM call, then a chain, then a RAG chain, and only then an agent if truly needed.
QExplain LangSmith and why it's essential for production agents.▾
LangSmith is Anthropic's (LangChain's) LLMOps platform for tracing, evaluation, and monitoring. It's essential because agents are black boxes by default — without tracing, when an agent gives a wrong answer you have no idea which step failed. LangSmith captures: every LLM call's exact input prompt and output, every tool call's input and result, token usage and cost per step, end-to-end latency per step. This enables: debugging (replay any trace, find the exact moment reasoning broke), evaluation (run test suites, score outputs automatically), monitoring (alert on quality degradation), optimization (find which steps are slowest/most expensive). Enabling it is two env vars: LANGCHAIN_TRACING_V2=true and LANGCHAIN_API_KEY. Without LangSmith (or a similar tool), operating a production agent is essentially flying blind. It's the first thing to set up before deploying any agent.
QWhat is the difference between zero-shot, few-shot, and fine-tuned tool use?▾
Zero-shot tool use: give the model tools and a task, it figures out which to call based solely on descriptions and its pre-training. Works well with capable models (GPT-4, Claude) for common tool types. Few-shot tool use: include example tool call sequences in the system prompt — (user request → thought → tool call → observation → answer). Dramatically improves reliability for complex tool schemas or unusual tool interaction patterns the model hasn't seen before. Fine-tuned tool use: fine-tune the model on examples of correct tool call sequences for your specific domain. Use when: few-shot isn't reliable enough, you have proprietary tools with unusual schemas, or you need very consistent formatting. The tradeoff: fine-tuning is expensive and requires maintenance when tools change. In practice: start zero-shot → add few-shot examples for problem cases → consider fine-tuning only if you have >1000 high-quality examples and consistency is still insufficient.
