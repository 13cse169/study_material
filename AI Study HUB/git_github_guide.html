What is Git?

// Distributed version control fundamentals

BEGINNER

📁 Version Control System

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005. It tracks changes in source code during software development, enabling multiple developers to collaborate without overwriting each other's work.

  • Every developer has the full project history locally
  • Works offline — no network needed for most operations
  • Fast branching and merging compared to older VCS tools
  • GitHub, GitLab, Bitbucket are hosting platforms for Git repos
Repository (repo)

A directory tracked by Git. Contains all files plus hidden .git/ folder with full history and config.

Working Directory

Your local files. Changes here are "untracked" until you add them to staging.

Staging Area (Index)

A holding zone for changes you plan to commit. Lets you craft precise commits.

Commit

A snapshot of staged changes with a SHA-1 hash, author, timestamp, and message.

Git Internals

// How Git stores data under the hood

INTERMEDIATE
Blob

Stores raw file content as a compressed object. Two identical files share one blob (content-addressable).

Tree

Maps filenames/dirs to blobs and other trees. Represents a directory snapshot.

Commit Object

Points to a root tree + parent commits + author/committer/message metadata.

Tag Object

Annotated tag: points to a commit with extra metadata (tagger, message, GPG sig).

Rebasing

// Rewrite history for a cleaner timeline

ADVANCED
⚠️
Golden Rule of Rebasing Never rebase commits that have been pushed to a shared public branch. Rebase only local commits or on feature branches others aren't using. Rebasing rewrites commit SHA hashes.
rebase.sh GIT 
# ── Standard rebase ─────────────────────────
$ git switch feature/api
$ git rebase main         # replay feature commits on top of main

# Result: linear history instead of merge commit
Before: A──B──C──M   (M is merge commit)
               \ /
                D──E

After:  A──B──C──D'──E'  (D' and E' are replayed)

# ── Interactive rebase (rewrite history) ─────
$ git rebase -i HEAD~4    # edit last 4 commits
pick a1b2c3d feat: add login
pick e4f5g6h fix: typo in login form   ← squash into above
pick h7i8j9k feat: add logout
reword k1l2m3n chore: cleanup           ← rename this

# Commands in interactive rebase:
# pick   = keep as is
# reword = keep, edit message
# edit   = pause to amend
# squash = merge into previous commit
# fixup  = squash, discard message
# drop   = remove commit entirely

# ── Rebase --onto ───────────────────────────
$ git rebase --onto main server client

Core Git Concepts

// Fundamentals every developer must know

BEGINNER–MID
01
What is the difference between Git and GitHub?
▼

Git is a free, open-source distributed version control system — a command-line tool installed locally that tracks file changes, manages branches, and maintains history. It works entirely offline.

GitHub is a cloud-based hosting platform for Git repositories. It adds collaboration features: pull requests, code review, issues, project boards, GitHub Actions (CI/CD), and a web interface. Alternatives include GitLab and Bitbucket.

Analogy: Git is like the engine; GitHub is the car built around it that makes driving easier for a team.

02
Explain the three states of files in Git.
▼

Files in Git exist in one of three states:

Modified — File has been changed in the working directory but not yet staged. Git sees the difference but hasn't recorded it.

Staged — File changes are marked (via git add) to go into the next commit. The staging area (index) holds this snapshot.

Committed — Changes are safely stored in the local Git database (repository) as a commit object with a unique SHA-1 hash.

There's also the Untracked state — new files Git doesn't know about yet (before any git add).

03
What is HEAD in Git?
▼

HEAD is a special pointer (reference) that points to the currently checked-out commit or branch. It represents "where you are now."

Normally HEAD points to a branch name (e.g., main), and that branch points to the latest commit — this is an "attached HEAD."

A detached HEAD occurs when you checkout a specific commit hash directly (git checkout abc123). HEAD points to the commit itself rather than a branch. New commits won't belong to any branch and can be lost. Fix by creating a branch: git switch -c new-branch

Notation: HEAD~1 = one parent back, HEAD~3 = three parents back, HEAD^2 = second parent of a merge commit.

04
What is the difference between git fetch and git pull?
▼

git fetch downloads changes from the remote into your remote-tracking branches (e.g., origin/main) but does not change your local working branch. It's safe — you can inspect what changed before integrating.

git pull = git fetch + git merge (or rebase). It downloads and immediately integrates changes into your current branch.

Best practice: prefer git fetch + git log origin/main to see what's new, then git merge or git rebase when ready. Or use git pull --rebase for a cleaner linear history.

git fetch is especially useful in CI pipelines and when you want to compare branches without modifying local state.

05
How does Git store data? What is SHA-1?
▼

Git uses a content-addressable filesystem. Every piece of data is stored as an object, identified by its SHA-1 hash — a 40-character hexadecimal string computed from the object's content.

Four object types: blob (file content), tree (directory), commit (snapshot + metadata), tag (annotated tag).

If two files have identical content, they share one blob. Renaming a file doesn't create a new blob — only content changes do. This makes Git extremely space-efficient.

SHA-1 ensures integrity — any corruption changes the hash, and Git detects it. Note: Git is transitioning to SHA-256 for new repositories (for security), though SHA-1 is still default in most installations.

06
What is a .gitignore file and how does it work?
▼

A .gitignore file specifies intentionally untracked files that Git should ignore. Patterns are matched against file paths relative to the .gitignore location.

Key rules: * matches anything except slash, ** matches any path, ! negates a pattern, / prefix anchors to directory, trailing / means directory only.

Important: .gitignore only works for untracked files. If a file was already committed, adding it to .gitignore doesn't stop tracking. You must: git rm --cached filename then commit the removal.

Levels: repo .gitignore, global ~/.gitignore_global (via git config --global core.excludesfile), and .git/info/exclude for local-only rules.

07
What is the difference between git stash pop and git stash apply?
▼

Both apply the stashed changes to the working directory, but:

git stash pop — applies the most recent stash (stash@{0}) and then removes it from the stash list. One-step operation.

git stash apply — applies the stash but keeps it in the stash list. You can apply the same stash to multiple branches. You must manually drop it later with git stash drop.

Use apply when you want to re-apply the same stash to different branches or keep it as a safety net. Use pop when you're done and want cleanup.

08
What is git reflog and when is it useful?
▼

git reflog (reference log) records every time HEAD moves — checkouts, commits, resets, merges, rebases. It's a local safety net that Git keeps for ~90 days by default.

It's invaluable for recovering "lost" commits after a git reset --hard or accidental branch deletion:

git reflog # see HEAD history
git reset --hard HEAD@{3} # jump back to earlier state
git checkout -b rescue HEAD@{5} # recover from specific point

Reflog is local only — not shared with remotes — so it only works if you haven't wiped your local repo.

09
What is the difference between merge and rebase?
▼

git merge creates a new "merge commit" that joins two branch histories. It preserves the exact history of what happened and when. Non-destructive — doesn't rewrite commits. Best for public/shared branches.

git rebase moves (replays) your commits onto the tip of another branch, rewriting their SHA hashes. Creates a perfectly linear history as if you'd started your work from the latest commit. Best for private feature branches.

When to use what:
• Merging main into a long-lived feature branch → merge (preserves context)
• Updating a personal feature branch with latest main → rebase (stays linear)
• Never rebase commits that are pushed to a shared remote branch.

10
What is a fast-forward merge?
▼

A fast-forward merge occurs when the target branch (e.g., main) has no new commits since the feature branch was created — the branch histories haven't diverged.

Instead of creating a merge commit, Git simply moves the branch pointer forward to the feature branch's tip. No new commit is created; history appears perfectly linear.

To force a merge commit even when FF is possible: git merge --no-ff feature. This is useful when you want to preserve the record that a feature branch existed, especially in team workflows where you want to see feature groupings in history.

GitHub's "Squash and merge" and "Rebase and merge" buttons both produce linear histories similar to fast-forward.

11
How do you resolve a merge conflict?
▼

A merge conflict occurs when two branches modify the same lines in a file differently. Git pauses the merge and marks conflicts in the file with:

<<<<<<< HEAD — your changes
======= — divider
>>>>>>> branch — incoming changes

Step-by-step resolution:
1. git status — identify conflicted files
2. Open each file, manually edit to resolve (remove markers, keep correct code)
3. git add <resolved-file> — mark as resolved
4. git commit — complete the merge

Tools: VS Code has built-in merge editor (Accept Current / Accept Incoming / Accept Both). git mergetool opens configured visual tool. git checkout --ours/--theirs accepts one version wholesale.

12
What is interactive rebase and what can you do with it?
▼

Interactive rebase (git rebase -i HEAD~N) opens an editor listing the last N commits with action keywords. You can reorder, modify, or remove commits before they're applied.

Available actions:
• pick — keep commit as-is
• reword — keep commit, edit the message
• edit — pause to amend the commit (add files, split it)
• squash — combine with previous commit, merge messages
• fixup — combine with previous commit, discard message
• drop — delete the commit entirely

Common use cases: clean up messy WIP commits before a PR, squash related commits, fix a commit message, reorder commits for logical grouping.

13
What branching strategies have you used? Explain GitFlow.
▼

Common branching strategies:

GitFlow — Uses long-lived branches: main (production), develop (integration), plus short-lived feature/, release/, and hotfix/ branches. Merges are heavily structured. Good for software with explicit releases. Complex overhead for small teams.

GitHub Flow — Simple: main is always deployable. Feature branches off main, PR → review → merge → deploy. Best for continuous deployment teams. No develop branch.

GitLab Flow — Adds environment branches (staging, production) to GitHub Flow. Deploy by merging main into environment branches.

Trunk-Based Development — All developers commit to main (trunk) frequently (daily). Feature flags hide incomplete features. Requires strong CI/CD. Used at Google and Facebook.

14
What is git cherry-pick and when would you use it?
▼

git cherry-pick applies a specific commit (or range of commits) from one branch onto your current branch. It creates a new commit with the same changes but a different SHA-1.

git cherry-pick abc123 # apply one commit
git cherry-pick abc..def # apply a range
git cherry-pick -n abc123 # apply changes without committing

When to use:
• You merged a feature but need just one bug fix from it on main now
• A hotfix was made on a release branch and needs to go to develop too
• You accidentally committed to the wrong branch
• Backporting a specific fix to an older release branch

Avoid overusing it — if you need many cherry-picks, a full merge may be more appropriate.

15
What is the difference between git reset and git revert?
▼

git reset moves the branch pointer backwards, effectively removing commits from history. Has three modes: --soft (keep staged), --mixed (unstage), --hard (discard all). Rewrites history — dangerous on shared branches.

git revert creates a new commit that undoes the changes of a specific previous commit. History is preserved — the original commit still exists, and a new "inverse" commit is added. Safe for public branches because it doesn't rewrite history.

Rule of thumb: reset for local/private history cleanup; revert for undoing pushed commits on shared branches. If colleagues have already pulled your commits, resetting forces them to deal with diverged histories.

16
What is git bisect and how does it work?
▼

git bisect performs a binary search through commit history to find the exact commit that introduced a bug. Instead of manually checking commits one-by-one, it narrows down the culprit in O(log n) steps.

git bisect start
git bisect bad # current is broken
git bisect good v1.0 # this version was fine

Git checks out a midpoint commit. You test it, then mark good or bad. Git narrows down and repeats. After ~7 steps for 100 commits, it identifies the exact first-bad commit.

It can even be automated: git bisect run npm test — Git runs your test suite automatically at each step, marking good/bad based on exit code (0 = good, non-zero = bad). Extremely powerful for debugging regressions.

17
What is --force-with-lease and why is it safer than --force?
▼

git push --force overwrites the remote branch unconditionally. If a teammate pushed commits while you were rebasing, their work is silently overwritten. Very dangerous on shared branches.

git push --force-with-lease checks that your local knowledge of the remote matches what's actually on the remote before pushing. If someone else pushed in between, Git refuses and you must first fetch and integrate their changes.

It's like an optimistic lock: "force push, but only if the remote is still at the commit I last saw." Always prefer --force-with-lease over plain --force when you must force push (e.g., after interactive rebase).

Optionally add --force-if-includes (Git 2.30+) for an even stricter check.

18
How does GitHub pull request review work? What are protected branches?
▼

Pull Request review: Reviewers can leave line comments, suggest changes (which the author can apply with one click), approve, or request changes. PRs can require N approvals before merging. Conversations can be resolved to track progress.

Protected branches are configured in GitHub Settings → Branches. Rules include:
• Require pull request before merging (no direct push)
• Require N approving reviews
• Require status checks to pass (CI/CD must be green)
• Require linear history (no merge commits)
• Require signed commits (GPG verification)
• Restrict who can push or force-push

This enforces code quality and prevents accidental direct commits to main/production.

19
What are Git submodules? When would you use them?
▼

Submodules allow you to include one Git repository inside another as a subdirectory, pinned to a specific commit. The parent repo stores just a reference (the SHA of the submodule commit) rather than the full submodule contents.

git submodule add https://github.com/lib/utils.git libs/utils
git submodule update --init --recursive # clone submodule contents
git submodule update --remote # update to latest

Use cases: Shared libraries used across multiple projects, third-party dependencies you want at a specific version, monorepo-like setups.

Downsides: Complex workflows — team must remember to run --recurse-submodules on clone/pull. Most teams prefer package managers (npm, pip) over submodules for dependencies.

20
How do you undo a git commit that has already been pushed?
▼

The safest way is git revert:
git revert HEAD # undo last commit
git push origin main # push the revert commit

This creates a new commit that is the inverse of the bad commit. History is preserved; colleagues are not affected.

If you need to completely erase history (rare, e.g., accidentally committed secrets):
git reset --hard HEAD~1
git push --force-with-lease

⚠️ This rewrites history. Every collaborator must then reset their local branch: git fetch && git reset --hard origin/main. Coordinate with the whole team first. For exposed secrets, also rotate them immediately — even after force-push, secrets may be cached in GitHub's history or other clones.

21
What is git worktree?
▼

git worktree lets you check out multiple branches simultaneously in different directories, all sharing the same single .git repository. No need to clone the repo multiple times.

git worktree add ../hotfix-branch hotfix/critical-bug
git worktree list
git worktree remove ../hotfix-branch

Use case: You're deep in a feature branch and need to quickly switch to fix a hotfix without stashing. Open the hotfix in a separate directory, fix it, while your feature branch stays open untouched in the original directory.

Each worktree has its own working directory and index, but they share the object database and config. You cannot check out the same branch in two worktrees simultaneously.

22
Explain Conventional Commits and Semantic Versioning.
▼

Conventional Commits is a specification for commit messages: type(scope): description

Types: feat (new feature), fix (bug fix), docs, style, refactor, perf, test, chore, ci, build
Breaking change: add ! after type: feat!: remove old API or add BREAKING CHANGE: in footer.

Semantic Versioning (SemVer): MAJOR.MINOR.PATCH
• fix commits → bump PATCH (1.0.1)
• feat commits → bump MINOR (1.1.0)
• Breaking change → bump MAJOR (2.0.0)

Tools like semantic-release or standard-version parse commit messages and automatically bump versions, generate changelogs, and publish releases in CI/CD pipelines.

23
What is git clean and when is it useful?
▼

git clean removes untracked files from the working directory. Unlike git restore which handles tracked files, git clean targets files Git doesn't know about.

git clean -n # dry run — show what would be deleted
git clean -f # force delete untracked files
git clean -fd # also delete untracked directories
git clean -fX # delete only ignored files (.gitignore'd)
git clean -fdx # delete ALL untracked (including ignored)

Always run with -n first! Once deleted, these files are gone — unlike tracked changes, they don't go to trash. Useful for cleaning build artifacts, resetting to a pristine state, or removing leftover generated files after switching branches.

24
How do Git hooks work and how do you share them with a team?
▼

Git hooks are scripts in .git/hooks/ that run automatically at lifecycle events. They're executable shell scripts named after the event (pre-commit, commit-msg, pre-push, etc.).

Problem: .git/ is not tracked by Git, so hooks aren't shared automatically.

Solutions for sharing hooks:
1. Husky (JavaScript): npx husky init — stores hooks in .husky/ which IS tracked. npm lifecycle script runs husky install on npm install.
2. pre-commit (Python): multi-language hook manager with .pre-commit-config.yaml
3. lefthook: fast Go-based hook manager for any language
4. Set core.hooksPath: git config core.hooksPath .githooks — stores hooks in tracked .githooks/ folder

25
What is the difference between origin and upstream?
▼

These are just remote names (aliases for URLs) — Git itself has no special meaning for them, but they follow a strong convention:

origin — the primary remote, typically your own fork or the repo you cloned. When you run git clone, Git automatically names the source remote "origin".

upstream — the original repository that you forked from. Used in open-source workflows to sync your fork with the original project's changes.

Typical fork workflow:
origin → git@github.com:YOU/repo.git (your fork)
upstream → https://github.com/ORIG/repo.git (original)

Syncing: git fetch upstream && git rebase upstream/main — then push to origin to keep your fork up-to-date.

26
What is git blame and how do you use it effectively?
▼

git blame annotates each line of a file with the commit SHA, author, and date that last modified that line. Useful for understanding why code was written a certain way and who to ask for context.

git blame app.js
git blame -L 20,40 app.js # lines 20-40 only
git blame -w app.js # ignore whitespace changes
git blame --follow -C app.js # track through renames/copies

Limitations: git blame only shows who last modified a line, not necessarily who originally wrote the logic (a code formatter might show as "last modified"). Use -C to track moved code.

VS Code, GitHub UI, and GitLens all have visual blame that shows hover info on each line. Remember: blame is for understanding, not assigning fault — hence some prefer the term "git annotate."

27
How do you handle large binary files in Git?
▼

Git is designed for text files — binary files (images, videos, datasets, design files) don't diff or compress well, and every version of a binary is stored in full, bloating the repository.

Git LFS (Large File Storage) is the standard solution. It replaces large files in your repo with small text pointers, while storing actual file content on a separate LFS server (GitHub, Bitbucket, or self-hosted).

git lfs install
git lfs track "*.psd" "*.mp4" "*.zip"
git add .gitattributes
git add design.psd
git commit -m "add design file"
git push # LFS uploads automatically

Alternatives: store assets on S3/CDN and reference them by URL; use separate repos for assets; use .gitignore for generated binaries.

28
What is a GitHub Action and how does CI/CD work with Git?
▼

GitHub Actions is a CI/CD platform built into GitHub. Workflows are defined in .github/workflows/*.yml files and trigger on Git events (push, PR, tag, schedule, manual).

CI (Continuous Integration): Every push/PR triggers automated tests, linting, build checks. This ensures code quality before merging. Required status checks on protected branches enforce that CI must pass.

CD (Continuous Delivery/Deployment): Merges to main automatically deploy to staging/production. Tagging a release can trigger a production deployment.

Key concepts: workflow (YAML file), trigger (on event), job (runs on a runner VM), step (individual command or action), action (reusable step from marketplace), secret (encrypted env vars for credentials).

Alternatives: GitLab CI, Jenkins, CircleCI, Travis CI, Buildkite.

29
What is the difference between git merge --squash and regular merge?
▼

Regular merge preserves all individual commits from the feature branch. You can see every step that was taken. Merge commit (if non-FF) records the junction point.

Squash merge (git merge --squash) combines all commits from the feature branch into a single staged change, which you then commit as one clean commit. The individual commits are not preserved in the target branch history.

Pros of squash: Clean, linear main branch history; each feature is one tidy commit; easy to revert an entire feature.

Cons of squash: Lose granular commit history; harder to understand what changed step-by-step; the original feature branch isn't "merged" per se (it still shows as unmerged in git log).

GitHub's "Squash and merge" button is essentially --squash. Many teams use squash for small PRs and regular merge for large/complex features.

30
How would you recover a deleted branch in Git?
▼

Since branch labels are just pointers to commits, deleting a branch doesn't delete the commits — they're still in the object database until garbage collected (GC runs after ~30 days by default).

Recovery using reflog:
git reflog # find the SHA of the last commit on deleted branch
git checkout -b recovered-branch HEAD@{4} # recreate branch at that SHA

Recovery using ORIG_HEAD: After an accidental merge/reset, ORIG_HEAD still points to where you were: git branch recovered ORIG_HEAD

From remote: If the branch was pushed: git fetch origin deleted-branch:deleted-branch

GitHub UI: GitHub keeps deleted branches for 30 days. Go to the PR → "Restore branch" button. Fast and foolproof.
