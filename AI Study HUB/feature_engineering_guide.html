What is Encoding?

// Converting categorical data to numeric form for ML models

BEGINNER

🗂️ Why Encode Categorical Features?

Most ML algorithms operate on numerical data. Categorical features (like color = "Red" or city = "Mumbai") must be converted to numbers before feeding into models.

  • Nominal categories have no order — e.g., colors, cities, countries
  • Ordinal categories have a meaningful order — e.g., Low / Medium / High
  • Choosing the wrong encoding can introduce false ordering or inflate dimensionality
02

Label Encoding

// Assign each category an integer

BEGINNER
ℹ️
When to Use Best for ordinal categories where order matters (Small=0, Medium=1, Large=2) or for tree-based models (Decision Tree, Random Forest, XGBoost) that don't assume numeric order.
label_encoding.py Python 
from sklearn.preprocessing import LabelEncoder
import pandas as pd

df = pd.DataFrame({
    'size': ['Small', 'Large', 'Medium', 'Small', 'Large'],
    'color': ['Red', 'Blue', 'Green', 'Blue', 'Red']
})

# ── sklearn LabelEncoder ─────────────────────────
le = LabelEncoder()
df['size_encoded'] = le.fit_transform(df['size'])
print(le.classes_)           # ['Large' 'Medium' 'Small']
print(df['size_encoded'])   # [2, 0, 1, 2, 0]

# ── Manual mapping (more control) ────────────────
size_order = {'Small': 0, 'Medium': 1, 'Large': 2}
df['size_ordinal'] = df['size'].map(size_order)

# ── Pandas Categorical ───────────────────────────
df['size_cat'] = pd.Categorical(
    df['size'],
    categories=['Small', 'Medium', 'Large'],
    ordered=True
)
df['size_codes'] = df['size_cat'].cat.codes

# ── Inverse transform ────────────────────────────
original = le.inverse_transform([0, 2])   # ['Large', 'Small']

One-Hot Encoding

// Create binary indicator columns for each category

BEGINNER
⚠️
Dummy Variable Trap Drop one column when using OHE with linear models (use drop_first=True) to avoid perfect multicollinearity. Tree-based models are immune to this.
one_hot_encoding.py Python 
from sklearn.preprocessing import OneHotEncoder
import pandas as pd

df = pd.DataFrame({'city': ['Mumbai', 'Delhi', 'Pune', 'Mumbai']})

# ── pandas get_dummies ───────────────────────────
dummies = pd.get_dummies(df['city'], drop_first=True)
# city_Mumbai  city_Pune  (Delhi dropped as reference)

# ── sklearn OneHotEncoder ────────────────────────
ohe = OneHotEncoder(
    sparse_output=False,   # return dense array
    handle_unknown='ignore', # unknown categories → all 0s
    drop='first'            # avoid multicollinearity
)
encoded = ohe.fit_transform(df[['city']])
print(ohe.get_feature_names_out())  # ['city_Mumbai' 'city_Pune']

# ── ColumnTransformer for pipelines ──────────────
from sklearn.compose import ColumnTransformer
from sklearn.pipeline import Pipeline
from sklearn.linear_model import LogisticRegression

preprocessor = ColumnTransformer([
    ('ohe', OneHotEncoder(drop='first'), ['city']),
])

pipe = Pipeline([
    ('prep', preprocessor),
    ('model', LogisticRegression())
])

Target Encoding

// Replace categories with target mean — but beware leakage

INTERMEDIATE
⚠️
Data Leakage Risk Always compute target encoding statistics on training data only. Use cross-fold target encoding or smoothing to prevent overfitting.
target_encoding.py Python 
import pandas as pd
import numpy as np
from category_encoders import TargetEncoder

df = pd.DataFrame({
    'city':   ['A', 'B', 'A', 'C', 'B', 'A'],
    'target': [1, 0, 1, 0, 1, 0]
})

# ── Manual Target Encoding ───────────────────────
means = df.groupby('city')['target'].mean()
df['city_target'] = df['city'].map(means)

# ── Smoothed Target Encoding (reduces overfitting)
def smooth_target_encode(df, col, target, alpha=10):
    global_mean = df[target].mean()
    stats = df.groupby(col)[target].agg(['mean', 'count'])
    # λ controls how much to trust category mean vs global
    smoothed = ((stats['count'] * stats['mean'] +
                 alpha * global_mean) /
                (stats['count'] + alpha))
    return df[col].map(smoothed)

df['city_smooth'] = smooth_target_encode(df, 'city', 'target')

# ── Leave-One-Out Encoding (zero leakage) ────────
from category_encoders import LeaveOneOutEncoder
loo = LeaveOneOutEncoder()
df['city_loo'] = loo.fit_transform(df[['city']], df['target'])

Ordinal & Frequency Encoding

// Ordered mapping and frequency-based encoding

INTERMEDIATE
ordinal_freq_encoding.py Python 
import pandas as pd
from category_encoders import OrdinalEncoder, HashingEncoder

# ── Ordinal Encoding with custom order ───────────
from sklearn.preprocessing import OrdinalEncoder
oe = OrdinalEncoder(
    categories=[['cold', 'warm', 'hot']]
)
df['temp_ord'] = oe.fit_transform(df[['temperature']])

# ── Frequency / Count Encoding ───────────────────
# Replace category with how often it appears
freq_map = df['city'].value_counts() / len(df)
df['city_freq'] = df['city'].map(freq_map)

# ── Rank Encoding ────────────────────────────────
rank_map = df['city'].value_counts().rank(ascending=False)
df['city_rank'] = df['city'].map(rank_map)

# ── Hashing Encoding (for very high cardinality) ─
he = HashingEncoder(cols=['city'], n_components=8)
df_hashed = he.fit_transform(df[['city']])
# Maps any category to 8 columns via hash function
# No new categories at inference time — great for IDs

Why Scaling Matters

// Algorithms sensitive to feature magnitude

BEGINNER

🧪 Impact of Unscaled Features

  • KNN & SVM: Distance-based — large-scale features dominate distances
  • Linear/Logistic Regression: Coefficient magnitudes become incomparable
  • Neural Networks: Gradient descent converges much slower without scaling
  • PCA: Components biased toward high-variance features
  • Tree-based models: Not affected — splits are threshold-based, scale-invariant

Min-Max Scaling

// Scale features to [0,1] range

BEGINNER
Min-Max Normalization Formula
X_scaled = (X − X_min) / (X_max − X_min)
minmax_scaling.py Python 
from sklearn.preprocessing import MinMaxScaler
import numpy as np

X = np.array([[100], [200], [150], [300], [50]])

# ── Standard Min-Max [0,1] ───────────────────────
scaler = MinMaxScaler()
X_scaled = scaler.fit_transform(X)
print(X_scaled.ravel())  # [0.2, 0.6, 0.4, 1.0, 0.0]

# ── Custom range [-1, 1] ─────────────────────────
scaler_custom = MinMaxScaler(feature_range=(-1, 1))
X_custom = scaler_custom.fit_transform(X)

# ── IMPORTANT: fit on train, transform on test ───
from sklearn.model_selection import train_test_split
X_train, X_test = train_test_split(X, test_size=0.2)

scaler.fit(X_train)                 # learn from train only
X_train_s = scaler.transform(X_train)
X_test_s  = scaler.transform(X_test)  # use same scaler!

# Sensitive to outliers — one extreme value ruins scaling

Standardization (Z-score)

// Zero mean, unit variance

BEGINNER
Z-Score Standardization
X_std = (X − μ) / σ    where μ = mean, σ = std dev
standardization.py Python 
from sklearn.preprocessing import StandardScaler
import numpy as np

X = np.array([[100], [200], [150], [300], [50]])

scaler = StandardScaler()
X_std = scaler.fit_transform(X)

print(f"Mean:  {scaler.mean_[0]:.1f}")      # 160.0
print(f"StdDev:{scaler.scale_[0]:.2f}")    # 87.75
print(X_std.ravel().round(2))
# [-0.68, 0.46, -0.11, 1.59, -1.25]

# Check: mean ≈ 0, std ≈ 1 after scaling
print(X_std.mean().round(10))  # 0.0
print(X_std.std().round(10))   # 1.0

Robust Scaler & L1/L2 Normalization

// Outlier-resistant scaling and row-wise normalization

INTERMEDIATE
robust_norm.py Python 
from sklearn.preprocessing import RobustScaler, Normalizer
import numpy as np

X = np.array([[1], [2], [3], [100], [4]])  # 100 is outlier

# ── Robust Scaler: uses median and IQR ───────────
# Formula: X_rob = (X − median) / IQR
rs = RobustScaler(quantile_range=(25.0, 75.0))
X_rob = rs.fit_transform(X)
# Outlier 100 is compressed, not ignored

# ── L2 Normalization (row-wise unit norm) ─────────
# Each sample (row) scaled so its Euclidean norm = 1
X_rows = np.array([[3, 4], [1, 0], [0, 5]])
norm_l2 = Normalizer(norm='l2').fit_transform(X_rows)
# [3,4] → [0.6, 0.8]  (3²+4²=25, √25=5; each/5)

# ── L1 Normalization (each row sums to 1) ─────────
norm_l1 = Normalizer(norm='l1').fit_transform(X_rows)
# [3,4] → [3/7, 4/7]  (|3|+|4|=7; each/7)

# ── Power Transforms: Yeo-Johnson / Box-Cox ───────
from sklearn.preprocessing import PowerTransformer

pt = PowerTransformer(method='yeo-johnson')  # handles negatives
X_normal = pt.fit_transform(X)
# Transforms skewed data toward Gaussian distribution
# box-cox requires all positive values

pca_complete.py Python 
from sklearn.decomposition import PCA
from sklearn.preprocessing import StandardScaler
import numpy as np
import matplotlib.pyplot as plt

# ALWAYS scale before PCA!
X = np.random.randn(200, 10)          # 200 samples, 10 features
X_scaled = StandardScaler().fit_transform(X)

# ── Fit PCA ──────────────────────────────────────
pca = PCA()                             # keep all components first
pca.fit(X_scaled)

# ── Explained variance ───────────────────────────
evr = pca.explained_variance_ratio_
cumulative = np.cumsum(evr)
print("Explained variance per component:")
print(evr.round(3))
print(f"First 3 PCs explain: {cumulative[2]:.1%}")

# ── Choose n_components to keep 95% variance ─────
pca_95 = PCA(n_components=0.95)        # auto-select k
X_reduced = pca_95.fit_transform(X_scaled)
print(f"Reduced from 10 → {X_reduced.shape[1]} features")

# ── 2D visualization ─────────────────────────────
pca_2d = PCA(n_components=2)
X_2d = pca_2d.fit_transform(X_scaled)
plt.scatter(X_2d[:, 0], X_2d[:, 1], alpha=0.6)
plt.xlabel('PC1'); plt.ylabel('PC2')

# ── Feature importance from loadings ─────────────
loadings = pca.components_             # shape: (n_components, n_features)
feature_importance = np.abs(loadings[0])  # first PC loadings

t-SNE & UMAP

// Non-linear methods for visualization and clustering

ADVANCED
t-SNE

Preserves local structure. Great for visualization. Non-deterministic. Slow on large data. Different perplexity values change output drastically. Not for preprocessing — only visualization.

UMAP

Faster than t-SNE. Preserves both local AND global structure. Deterministic. Can be used as preprocessing step (unlike t-SNE). Hyperparams: n_neighbors, min_dist.

PCA vs t-SNE

PCA is linear, global, fast, and reproducible. t-SNE/UMAP are non-linear, local-structure-preserving, slower. Use PCA first to reduce to ~50 dims, then t-SNE.

LDA (Supervised)

Linear Discriminant Analysis uses class labels. Maximizes between-class separation. At most C-1 components (C = number of classes). Better for classification than PCA.

tsne_umap_lda.py Python 
from sklearn.manifold import TSNE
from sklearn.discriminant_analysis import LinearDiscriminantAnalysis
import umap  # pip install umap-learn

# ── t-SNE ────────────────────────────────────────
tsne = TSNE(
    n_components=2,
    perplexity=30,       # typical range: 5–50
    random_state=42,
    n_iter=1000,
    learning_rate='auto'
)
X_tsne = tsne.fit_transform(X_scaled)

# ── UMAP ─────────────────────────────────────────
reducer = umap.UMAP(
    n_components=2,
    n_neighbors=15,   # local neighborhood size
    min_dist=0.1,     # how tightly points cluster
    random_state=42
)
X_umap = reducer.fit_transform(X_scaled)

# ── LDA (supervised) ─────────────────────────────
lda = LinearDiscriminantAnalysis(n_components=2)
X_lda = lda.fit_transform(X_scaled, y)    # uses class labels!
print(f"Variance explained: {lda.explained_variance_ratio_}")

# ── Best practice pipeline ────────────────────────
# 1. StandardScaler → 2. PCA (50 dims) → 3. UMAP/t-SNE
pca50 = PCA(n_components=50, random_state=42)
X_pca = pca50.fit_transform(X_scaled)
X_vis = TSNE(n_components=2).fit_transform(X_pca)

Filter Methods

// Statistical tests — fast, model-independent

BEGINNER
filter_methods.py Python 
from sklearn.feature_selection import (
    VarianceThreshold, SelectKBest,
    f_classif, mutual_info_classif,
    chi2, f_regression
)
import pandas as pd

# ── 1. Variance Threshold ────────────────────────
# Remove features with near-zero variance (constant/near-constant)
vt = VarianceThreshold(threshold=0.01)
X_vt = vt.fit_transform(X)
print("Removed features:", X.shape[1] - X_vt.shape[1])

# ── 2. Correlation Filter ────────────────────────
corr_matrix = pd.DataFrame(X).corr().abs()
upper = corr_matrix.where(
    np.triu(np.ones(corr_matrix.shape), k=1).astype(bool))
to_drop = [col for col in upper.columns if upper[col].max() > 0.95]
X_uncorr = pd.DataFrame(X).drop(columns=to_drop)

# ── 3. ANOVA F-test (classification) ─────────────
selector_f = SelectKBest(f_classif, k=5)
X_f = selector_f.fit_transform(X, y)
print("F-scores:", selector_f.scores_.round(2))

# ── 4. Mutual Information ────────────────────────
# Captures non-linear relationships too
selector_mi = SelectKBest(mutual_info_classif, k=5)
X_mi = selector_mi.fit_transform(X, y)

# ── 5. Chi-squared (non-negative features only) ──
selector_chi = SelectKBest(chi2, k=5)
X_chi = selector_chi.fit_transform(X_counts, y)   # count data

Wrapper & Embedded Methods

// Model-driven feature selection

INTERMEDIATE
wrapper_embedded.py Python 
from sklearn.feature_selection import (
    RFE, RFECV, SelectFromModel
)
from sklearn.ensemble import RandomForestClassifier
from sklearn.linear_model import LassoCV, ElasticNet

# ── Recursive Feature Elimination (RFE) ──────────
rf = RandomForestClassifier(n_estimators=100)
rfe = RFE(estimator=rf, n_features_to_select=5)
X_rfe = rfe.fit_transform(X, y)
print("Selected features:", rfe.support_)
print("Feature ranking:", rfe.ranking_)

# ── Embedded: Lasso (L1 shrinks to 0) ────────────
lasso = LassoCV(cv=5).fit(X, y)

# ── Tree Feature Importances ─────────────────────
rf.fit(X, y)
importances = pd.Series(
    rf.feature_importances_,
    index=[f'f{i}' for i in range(X.shape[1])]

print(importances.head(5))

SHAP & Permutation Importance

// Model-agnostic, unbiased importance

ADVANCED
✅
Why SHAP over Feature Importances? Tree feature importances are biased toward high-cardinality features. SHAP gives each feature a fair Shapley value based on its actual contribution to individual predictions. Works for any model.
shap_permutation.py Python 
import shap  # pip install shap
from sklearn.inspection import permutation_importance

# ── SHAP for tree models (fast TreeExplainer) ─────
explainer = shap.TreeExplainer(rf)
shap_values = explainer.shap_values(X_test)

# Summary plot (global importance)
shap.summary_plot(shap_values[1], X_test, plot_type="bar")

# Dependence plot (1 feature interaction)
shap.dependence_plot("feature_0", shap_values[1], X_test)

# ── SHAP for any model ────────────────────────────
explainer_k = shap.KernelExplainer(model.predict, X_background)
shap_vals_k = explainer_k.shap_values(X_test[:50])

# ── Permutation Importance ────────────────────────
# Shuffles each feature, measures score drop
perm = permutation_importance(
    rf, X_test, y_test,
    n_repeats=10, random_state=42
)
sorted_idx = perm.importances_mean.argsort()[::-1]
for i in sorted_idx[:5]:
    print(f"f{i}: {perm.importances_mean[i]:.4f} ± {perm.importances_std[i]:.4f}")

Time Series Features

// Rolling statistics, lag features, frequency domain

INTERMEDIATE
time_series_features.py Python 
import pandas as pd
import numpy as np

df = pd.DataFrame({'value': np.random.randn(200)})

# ── Lag features ─────────────────────────────────
for lag in [1, 7, 14, 30]:
    df[f'lag_{lag}'] = df['value'].shift(lag)

# ── Rolling statistics ───────────────────────────
for window in [7, 30]:
    roll = df['value'].rolling(window)
    df[f'roll_mean_{window}'] = roll.mean()
    df[f'roll_std_{window}']  = roll.std()
    df[f'roll_min_{window}']  = roll.min()
    df[f'roll_max_{window}']  = roll.max()

# ── Date/time features ───────────────────────────
df['hour']       = df.index.hour
df['dayofweek']  = df.index.dayofweek
df['month']      = df.index.month
df['quarter']    = df.index.quarter
df['is_weekend'] = df['dayofweek'] >= 5

# ── Cyclical encoding (sin/cos) ───────────────────
# Preserves circular nature: hour 23 is close to hour 0
df['hour_sin'] = np.sin(2 * np.pi * df['hour'] / 24)
df['hour_cos'] = np.cos(2 * np.pi * df['hour'] / 24)

# ── tsfresh (automated feature extraction) ────────
from tsfresh import extract_features
features = extract_features(df, column_id='id', column_sort='time')
# Extracts 700+ statistical features automatically

ADVANCED
interaction_features.py Python 
from sklearn.preprocessing import PolynomialFeatures
import numpy as np

X = np.array([[2, 3], [4, 5]])   # [a, b]

# ── Polynomial features ──────────────────────────
poly = PolynomialFeatures(degree=2, include_bias=False)
X_poly = poly.fit_transform(X)
# Output: [a, b, a², a·b, b²]
print(poly.get_feature_names_out())

# ── Interaction only (no powers) ─────────────────
inter = PolynomialFeatures(
    degree=2, interaction_only=True, include_bias=False
)
X_inter = inter.fit_transform(X)
# Output: [a, b, a·b] — only the cross term

# ── Manual domain-specific features ──────────────
df['bmi']            = df['weight'] / (df['height'] ** 2)
df['price_per_sqft'] = df['price'] / df['area']
df['age_income']     = df['age'] * df['income']   # interaction

Q1

Encoding Questions

// Categorical feature handling

BEGINNER–MID
01
What is the difference between Label Encoding and One-Hot Encoding? When would you use each?
▼

Label Encoding assigns each category an integer (e.g., Red=0, Green=1, Blue=2). It implies a numerical ordering that doesn't actually exist for nominal variables. Use it for: (1) ordinal features where order is meaningful (Small<Medium<Large), and (2) tree-based models (Decision Tree, Random Forest, XGBoost) which don't interpret numeric order linearly.

One-Hot Encoding creates a binary column per category. No ordinal relationship is implied. Use for nominal features with any linear model, SVM, KNN, or neural networks. Drawback: high-cardinality features explode dimensionality.

Rule of thumb: <10–15 categories → OHE; >15 categories → Target/Frequency/Hashing encoding; ordinal data → manual ordinal mapping; tree models → label encoding is fine.

02
What is the Dummy Variable Trap? How do you avoid it?
▼

The Dummy Variable Trap (perfect multicollinearity) occurs when one dummy variable can be perfectly predicted from the others. For example, OHE of [Red, Green, Blue] creates 3 columns where column3 = 1 − col1 − col2. This makes the design matrix singular (non-invertible), causing numerical issues in linear models and making coefficients non-unique.

Solution: Drop one reference category using drop='first' in sklearn's OneHotEncoder or drop_first=True in pandas get_dummies(). This is also called using k-1 dummies for k categories.

Tree models are immune because they use splits rather than matrix inversion. Neural networks have built-in redundancy handling. The trap only affects methods that solve normal equations (Linear/Logistic Regression, PCA).

03
What is Target Encoding and how do you prevent data leakage?
▼

Target Encoding replaces each category with the mean of the target variable for that category (e.g., city Mumbai → average price in Mumbai). It handles high-cardinality well and captures target-feature relationships.

Leakage risk: If you compute the mean on the same data you're training on, the model sees the target during training through the encoding — overfitting severely.

Prevention strategies:
(1) Cross-fold encoding: compute mean for fold k using all other folds (like cross-validation)
(2) Smoothing: blend category mean with global mean: (count × cat_mean + α × global_mean) / (count + α)
(3) Leave-one-out encoding: compute mean excluding current row
(4) Add Gaussian noise: small random perturbation to encoded values
Always compute encoding statistics on train set only, then apply to test set.

04
How do you handle high-cardinality categorical features with thousands of unique values?
▼

High-cardinality (e.g., user IDs, product SKUs, zip codes) makes OHE impractical (thousands of sparse columns).

Options by use case:
Hashing Encoding: map categories to a fixed number of bins using a hash function. Fast, constant-memory, handles unseen categories. Some collision between categories.

Binary Encoding: convert to integer → binary representation. 1000 categories → just 10 binary columns (log₂(1000)).

Target / Frequency Encoding: works well for high-cardinality with sufficient data per category.

Entity Embeddings (neural nets): learn a dense representation jointly with the task — often the best approach for very high cardinality.

Grouping rare categories: bin infrequent values into an "Other" bucket before encoding.

Q2

Scaling & Normalization Questions

// When, why, and which method

INTERMEDIATE
05
What is the difference between Normalization and Standardization?
▼

Normalization (Min-Max Scaling) rescales values to [0,1] (or a custom range): X' = (X - min) / (max - min). Bounded output, sensitive to outliers. Best when: you need values in a specific range (neural networks with sigmoid/tanh), algorithm is not distance-based, data has no significant outliers.

Standardization (Z-score) transforms to zero mean, unit variance: X' = (X - μ) / σ. Unbounded output, robust to extreme values on the scaling computation (though outliers remain). Best when: data is approximately Gaussian, using distance-based or gradient-based algorithms, you need to compare features.

In practice: Standardization generally outperforms Normalization for SVM, Logistic Regression, Linear models, and PCA. Normalization preferred for neural networks and KNN on bounded data.

Critical rule: Fit scaler on training data only. Apply the same fitted scaler to test/production data. Never refit on test data — that's leakage.

06
When should you use Robust Scaler instead of StandardScaler?
▼

RobustScaler uses the median and IQR instead of mean and std: X' = (X - median) / IQR. Since median and IQR are not affected by extreme outliers (they use the 25th and 75th percentiles), the scaler is robust to outliers.

Use RobustScaler when:
• Your data contains outliers that you cannot or don't want to remove
• A few extreme values would heavily skew the mean/std in StandardScaler
• Features follow skewed distributions

StandardScaler is preferable when data is roughly Gaussian without extreme outliers. If outliers are genuine signal (not errors), you might want them to retain their relative distance — in which case StandardScaler is better since outliers will simply have large Z-scores.

07
What is the difference between L1 and L2 normalization?
▼

Both normalize individual samples (rows), not features — unlike StandardScaler which normalizes columns.

L1 (Manhattan): divides each feature value by the sum of absolute values. Result: each row sums to 1. Robust to outliers within a sample. Used in text analysis (term frequency normalization).

L2 (Euclidean): divides by the Euclidean norm. Result: each row vector has unit length (norm=1). Corresponds to projecting onto a unit hypersphere. Used most commonly in ML: cosine similarity, SVM, neural networks.

When to use: TF-IDF already incorporates L2 normalization. Explicitly use Normalizer for comparing document similarity (cosine = dot product of L2-normalized vectors). Not appropriate when the scale/magnitude itself carries information.

08
Why should you NEVER fit the scaler on test data?
▼

This is a fundamental form of data leakage. Fitting a scaler on test data means the scaler learns the test set's distribution (min, max, mean, std). This "leaks" information about the test set into preprocessing, producing overly optimistic evaluation metrics that won't replicate in production.

Correct approach:
scaler.fit(X_train) # learn from train only
X_train_s = scaler.transform(X_train)
X_test_s = scaler.transform(X_test) # same scaler

In real production: you fit the scaler once on training data, serialize it (joblib/pickle), and use it on all incoming data at inference time. Test set simulates production — it must be treated as unseen data.

Pipeline pattern: sklearn Pipeline automatically ensures the scaler is fit-transformed on train and transform-only on test during cross-validation — use it to prevent accidental leakage.

Q3

PCA & Dimensionality Reduction

// Understanding components and variance

INTERMEDIATE
09
Explain PCA. How do you choose the number of components?
▼

PCA (Principal Component Analysis) finds a new coordinate system aligned with the directions of maximum variance in the data. Each principal component (PC) is a linear combination of the original features, and PCs are orthogonal to each other.

Mathematically: compute the covariance matrix Σ, decompose into eigenvectors/eigenvalues. Eigenvectors = principal components (directions). Eigenvalues = variance explained per component. Project data onto top k eigenvectors.

Choosing number of components:
(1) Explained variance threshold: pick k such that cumulative explained variance ≥ 95% (or 90%): PCA(n_components=0.95)
(2) Elbow plot (scree plot): plot eigenvalues, find the "elbow" where additional components add little variance
(3) Task performance: cross-validate downstream model with different k values
(4) Visualization: always use 2 or 3 for plotting

Always scale before PCA! PCA is variance-based — unscaled features with large ranges dominate all components.

10
What are the limitations of PCA? When would you NOT use it?
▼

Limitations of PCA:
(1) Linear only: PCA only captures linear relationships. Non-linear structure is missed. Use kernel PCA, t-SNE, or UMAP for non-linear reduction.
(2) Interpretability loss: Principal components are linear combinations of all original features — hard to explain to stakeholders. You lose the ability to say "feature X matters."
(3) Unsupervised: PCA ignores class labels. Maximum variance direction may not be discriminative. Use LDA for supervised dimensionality reduction.
(4) Assumes Gaussian: PCA is optimal for Gaussian-distributed features. Performs poorly on binary or count data.
(5) Sensitive to scaling: Must standardize first.

Don't use PCA when: interpretability is critical; data is non-linear; you have sparse data (use TruncatedSVD instead); features are categorical; you need feature selection (PCA creates new features, doesn't select original ones).

11
What is the difference between PCA and LDA?
▼

PCA (unsupervised): maximizes variance in the projected space. Ignores class labels. Goal: preserve as much data variability as possible. Used for compression, noise reduction, visualization.

LDA (Linear Discriminant Analysis, supervised): maximizes the ratio of between-class variance to within-class variance. Uses class labels to find directions that best separate classes. Maximum components = min(n_classes − 1, n_features).

Example: If data has two overlapping clusters that don't differ in total variance (PCA gives useless components), but they're separated along a specific axis — LDA finds that axis while PCA misses it.

Choose LDA when you're doing classification and need to reduce dimensions while preserving class separability. Choose PCA when you have no labels or want general-purpose compression. In practice, try both and let downstream classifier performance guide the choice.

Q4

Feature Selection Questions

// Filter, Wrapper, Embedded methods

INTERMEDIATE–ADV
12
What is the difference between Filter, Wrapper, and Embedded feature selection methods?
▼

Filter Methods: Use statistical measures to score features independently of any model. Fast, scalable, no overfitting risk. Examples: Variance Threshold, Correlation, ANOVA F-test, Mutual Information, Chi-squared. Drawback: don't consider feature interactions; may select redundant features.

Wrapper Methods: Use a model to evaluate subsets of features. Capture feature interactions. Examples: RFE (Recursive Feature Elimination), Forward/Backward Selection. Expensive — O(2^n) combinations. Risk of overfitting to training data.

Embedded Methods: Feature selection happens during model training. Most efficient — one training run. Examples: Lasso (L1 regularization shrinks weights to 0), Tree feature importances (Gini/entropy gain), Elastic Net. Balance between filter speed and wrapper accuracy.

Best practice: Start with filter methods to remove obviously irrelevant/constant features. Then use embedded methods (Lasso or tree importances) for a refined selection. Use wrapper only for small datasets where computational cost is acceptable.

13
What is Mutual Information and how does it differ from correlation?
▼

Pearson Correlation measures only linear relationships between two variables. If X and Y have a non-linear relationship (e.g., Y = X²), correlation ≈ 0 even though they're perfectly related.

Mutual Information (MI) measures the reduction in uncertainty about one variable given knowledge of the other. MI = 0 means complete independence; MI > 0 means knowing X tells you something about Y. Crucially, MI captures any relationship — linear or non-linear.

Mathematically: MI(X,Y) = H(Y) − H(Y|X), where H is Shannon entropy.

When to prefer MI: non-linear feature-target relationships, discrete/categorical features (use Chi-squared for purely discrete), when you suspect complex patterns. When to prefer correlation: interpretability, linear models, quickly checking for simple linear associations.

Note: MI does not have an upper bound (unlike correlation which is [-1, 1]), making cross-feature comparisons tricky. Normalized MI (NMI) addresses this.

14
Why are Random Forest feature importances sometimes misleading?
▼

Tree-based feature importances (Gini/entropy) are biased toward:

(1) High-cardinality features: More unique values → more split opportunities → artificially higher importance. A random ID column can appear important.
(2) Correlated features: Importance is split between correlated features arbitrarily. Remove one and the other gets all the importance.
(3) Continuous vs discrete: Continuous features can split at many points — systematically preferred over binary features even if the binary feature is more informative.

Better alternatives:
• Permutation Importance: shuffle feature values, measure score drop. Unbiased, model-agnostic, but expensive
• SHAP values: Shapley game-theoretic approach. Fair allocation of feature contributions per prediction. Handles correlations better
• Drop-column importance: retrain without each feature, measure performance change — most accurate but extremely slow

15
What is SHAP and how does it improve feature importance understanding?
▼

SHAP (SHapley Additive exPlanations) applies cooperative game theory to ML. A SHAP value for a feature in a prediction represents the average marginal contribution of that feature across all possible orderings of features — ensuring a fair attribution that satisfies key axioms (efficiency, symmetry, dummy, additivity).

Key advantage: SHAP provides local explanations (why did the model predict X for this specific sample?) and global importance (mean |SHAP values| across all samples).

Types: TreeExplainer (fast, exact for trees), LinearExplainer (for linear models), KernelExplainer (model-agnostic, slow), DeepExplainer (neural nets).

Compared to Gini importance: SHAP handles correlated features better, is consistent (a more impactful feature always gets higher SHAP), and provides directional information (positive/negative contribution). SHAP is now the gold standard for feature importance in production ML.

Q5

Advanced Feature Engineering

// Extraction, interaction, and practical questions

ADVANCED
16
What is TF-IDF and how does it improve over Bag of Words?
▼

Bag of Words (BoW) counts word occurrences in each document. Problem: common words like "the," "is," "and" appear frequently in every document and dominate the feature space without carrying useful discriminative information.

TF-IDF (Term Frequency – Inverse Document Frequency) addresses this:
TF-IDF(t, d) = TF(t, d) × IDF(t)
TF(t,d) = count(t in d) / total_terms(d)
IDF(t) = log(N / df(t)), where N=total docs, df=docs containing term t

Words appearing in many documents get low IDF (penalized). Rare but discriminative words get high IDF (rewarded). Common words like "the" appear in all docs → IDF ≈ 0 → TF-IDF ≈ 0.

Improvements over BoW: document length normalization (TF divides by doc length), penalizes common words, rewards rare-but-informative terms. Sublinear TF (log(1+tf)) further prevents very frequent terms from dominating. TF-IDF + L2 normalization is the standard baseline for text classification.

17
How do you handle missing values as part of feature engineering?
▼

Missing value handling is itself a feature engineering step:

Simple imputation: Mean/median (numerical), mode (categorical). Fast but ignores relationships. Use SimpleImputer.

KNN imputation: Replace missing value with weighted average of k nearest neighbors. Respects local structure but O(n²) — slow on large datasets.

Iterative imputation (MICE): Model each feature as a function of others, iterating until convergence. Most accurate for MAR (Missing At Random) data.

Indicator features: Create binary column feature_X_missing to signal that the value was imputed. This tells the model that missingness itself may be informative (e.g., income not reported might correlate with behavior).

Critical: Fit imputer on training data only. For categorical: impute with mode or create "Unknown" category. For tree models: some support native NaN handling (XGBoost, LightGBM) — use it. Always document the missingness pattern and mechanism (MCAR/MAR/MNAR) before choosing strategy.

18
What is the Curse of Dimensionality and how does feature engineering address it?
▼

The Curse of Dimensionality describes how data becomes increasingly sparse as the number of features grows. In high-dimensional spaces: (1) all points become approximately equidistant, making distance-based algorithms (KNN, SVM-RBF) fail; (2) the volume of space grows exponentially — you need exponentially more data to maintain the same density; (3) models overfit easily as there are more feature combinations to memorize than actual patterns.

Feature engineering solutions:
Dimensionality reduction: PCA, UMAP — reduce to intrinsic dimensionality
Feature selection: Remove irrelevant/redundant features
Regularization: L1 (Lasso) implicitly does feature selection by zeroing weights
Domain knowledge: Hand-craft meaningful features from raw data — fewer but better features
Embeddings: Learn compact dense representations (vs sparse high-dimensional representations)

Rule: more features ≠ better model. Quality over quantity is the core principle of feature engineering.

19
What are interaction features and when should you create them?
▼

Interaction features are products (or other combinations) of two or more existing features, capturing effects that are only meaningful in combination. Example: age × income might predict loan default better than either feature alone (a young person with high income is different from an old person with high income).

When to create interaction features:
• You're using a linear model that can't learn interactions natively
• Domain knowledge suggests a multiplicative relationship (BMI = weight/height²)
• Exploratory analysis shows correlation with target only when combined
• Tree models already learn interactions internally — rarely needed there

Types: polynomial features (x², x³, xy), ratios (price/area), differences (score_after - score_before), boolean combinations (is_senior AND is_premium_member).

Warning: Polynomial features grow combinatorially — degree-2 on 100 features → 5050 new features. Always combine with feature selection after generating interactions.

20
How do you encode cyclical features like hour of day, month, or day of week?
▼

Cyclical features have a circular nature: hour 23 is close to hour 0, December is close to January. If you encode hour as a plain integer (0–23), the model sees 23 and 0 as far apart (difference of 23) when they're actually adjacent (difference of 1).

Solution: sin/cos encoding
hour_sin = sin(2π × hour / 24)
hour_cos = cos(2π × hour / 24)

This maps each hour to a point on a unit circle. Hour 0 and hour 24 map to the same point. The distance between any two times is properly represented in (sin, cos) space.

You need both sin AND cos — just sin alone has ambiguity (hours 3 and 9 have the same sin value). Together they uniquely identify each position in the cycle.

Apply to: hour (period=24), day of week (period=7), month (period=12), day of year (period=365), wind direction (period=360°). This is always better than treating these as plain integers or OHE-encoding them.

21
What is feature leakage and how do you detect it?
▼

Feature leakage occurs when a feature contains information from the future (at prediction time) or directly encodes the target, making model performance deceptively perfect in evaluation but useless in production.

Common forms:
(1) Target leakage: feature computed after the target event (e.g., "days in hospital" to predict "will be readmitted")
(2) Train-test contamination: scaling, imputation, encoding fitted on combined train+test data
(3) Time leakage: using future data in rolling features (using tomorrow's price to predict today's)
(4) Surrogate features: a proxy for the target (e.g., "discharge date" when predicting "hospitalized")

Detection signals:
• Suspiciously high accuracy (AUC > 0.99 on real-world data)
• One feature overwhelmingly dominates importance
• Performance collapses dramatically in production
• Temporal: features have timestamps after prediction time

Prevention: Strict train/validation/test split before any preprocessing. Use sklearn Pipelines. For time series: use time-based splits only.

22
How do you handle skewed features? What transformations help?
▼

Skewed features (e.g., income, house prices, population) violate the normality assumption of linear models and can harm gradient descent convergence in neural networks.

Common transforms for right-skewed (long right tail):
• Log transform: log(x+1) — most common, handles zeros
• Square root: √x — milder compression
• Box-Cox: family of transforms, finds optimal λ via MLE. Requires positive values.
• Yeo-Johnson: generalization of Box-Cox, handles negative values

For bimodal / heavy-tailed distributions:
• Quantile Transformer: maps to uniform or Gaussian distribution. Robust to outliers, handles any shape.
• Rank transform: replace with rank order

When not needed: Tree-based models (Random Forest, XGBoost, LightGBM) are completely invariant to monotonic transforms — skewness doesn't affect them. Transform primarily for linear models, neural networks, distance-based algorithms, and when features will be used in PCA.

23
What is a Feature Store? Why is it important in production ML?
▼

A Feature Store is a centralized repository for storing, discovering, and serving ML features. It solves the "feature engineering at scale" problem in production systems.

Problems it solves:
(1) Training-serving skew: features computed differently in training (batch on historical data) vs serving (real-time on new data) → model behaves differently in production
(2) Duplication: multiple teams recomputing the same features independently → wasted compute, inconsistencies
(3) Discovery: no way to find what features already exist across the organization
(4) Point-in-time correctness: ensures features used for training reflect values that were actually available at prediction time (no future leakage)

Key capabilities: offline store (historical features for training), online store (low-latency features for inference), feature definitions as code, versioning, monitoring for feature drift.

Popular implementations: Feast, Tecton, Hopsworks, Databricks Feature Store, Vertex AI Feature Store. Essential for teams running >5 models in production.

24
What is feature drift and how do you monitor for it?
▼

Feature drift (also called data drift or covariate shift) occurs when the statistical distribution of input features changes over time in production, causing model predictions to degrade even though the model parameters haven't changed.

Example: A model trained on pre-pandemic user behavior starts drifting when behavior changes drastically post-pandemic.

Detection methods:
• Population Stability Index (PSI): measure distribution shift for a single feature. PSI < 0.1 = stable, 0.1–0.25 = minor shift, > 0.25 = major drift
• KL Divergence / Jensen-Shannon Distance: information-theoretic drift measures
• Kolmogorov-Smirnov test: statistical test for distribution equality (numerical features)
• Chi-squared test: for categorical features
• Wasserstein distance: "earth mover's distance" between distributions

Tools: Evidently AI, WhyLabs, Arize, Great Expectations, MLflow.

When drift is detected: retrain on recent data, investigate root cause, add the new patterns to the feature engineering pipeline.

25
Explain the complete feature engineering pipeline you would build for a real ML project.
▼

A production-grade feature engineering pipeline:

1. Data Profiling: EDA — distributions, missing rates, cardinality, correlation matrix, outlier detection

2. Data Cleaning: Remove constant/duplicate features, fix data types, handle obvious errors

3. Missing Value Handling: Impute + create missingness indicators. Fit on train only.

4. Encoding: OHE for low-cardinality nominals, ordinal encoding for ordered features, target encoding (with CV) for high-cardinality. Handle unseen categories.

5. Feature Creation: Domain-specific features, time features (sin/cos cyclical, lags, rolling stats), interaction terms, ratio features

6. Skew Correction: Log/Yeo-Johnson transforms for linear models / neural nets

7. Scaling: StandardScaler or RobustScaler depending on outliers. Fit on train only.

8. Feature Selection: Remove near-zero variance, correlated features (>0.95), then SHAP or Lasso for final selection

9. Dimensionality Reduction (optional): PCA if still too many features and model allows

Wrap everything in sklearn Pipeline to prevent leakage during cross-validation. Version the pipeline. Monitor for drift in production.
